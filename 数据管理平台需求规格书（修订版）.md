# 数据管理平台需求规格书（修订版）

> 修订说明：本文档基于初版需求规格书，结合逻辑审查和行业最佳实践（Collibra、DataHub、Dataphin 等）进行了系统性修订。修订内容以 `[修订]` 标记标注，便于对照原文档查看变更点。

---

# 一、项目概述

本数据管理平台以「主题（库）、口径（表）、要素（表字段）」为核心数据资产，采用「项目驱动」的增量迭代模式，解决业务需求提报、口径澄清、数据开发、审核验收、上线归档全流程的协作痛点。平台核心目标是实现业务口径与技术口径的双向对齐、口径变更可追溯、项目迭代可管控，让业务人员无需接触技术底层，数据开发人员高效落地需求，确保数据资产的规范性、一致性和可复用性。

核心定位：业务驱动、资产可控、流程闭环、追溯可查的企业级数据协作与管理平台。

`[修订]` **口径定位声明**：本系统中的「口径」严格对应一张目标宽表/统计口径表/指标表（即逻辑口径与物理表一一映射）。若未来业务出现一个逻辑口径需跨多张物理表的场景，需通过拆分为多个口径 + 标签关联的方式处理，作为后续扩展点。

# 二、核心数据模型（核心骨架）

平台核心数据资产分为三层，层层关联，同时新增口径版本、业务口径、技术口径关联逻辑，确保变更可追溯、业务与技术对齐。

## 2.1 主题（库）

定义：数据的最高层级分类，对应企业业务域/业务线，本质是数据的"容器"，用于对口径（表）进行归类、归置和权限隔离。

核心属性：

- 主题ID（唯一标识）
- 主题名称（如：用户主题、订单主题、商品主题、营销主题）
- 所属部门（如：运营部、销售部、技术部）
- 主题描述（业务含义、覆盖范围）
- 创建人、创建时间、更新人、更新时间
- 关联口径列表（该主题下包含的所有口径）

核心操作：新增、编辑、删除（删除需校验无关联口径）、查询、检索。

## 2.2 口径（表）

定义：对应业务宽表/统计口径表/指标表，是业务需求的核心载体，承接业务规则和统计逻辑，也是要素（字段）的归属主体。每个口径具备版本概念，记录全量变更历史，支持在同一个项目中多次修改。

核心属性：

- 口径ID（唯一标识）
- 口径名称（如：用户留存口径表、GMV汇总口径表、订单明细口径表）
- 所属主题（关联唯一主题ID）
- 当前版本号（如：V1.0、V1.1、V2.0，规则见下方版本号规则）
- `[修订]` 口径生命周期状态（草稿、生效、废弃）—— 仅表示口径本身的生命阶段
- `[修订]` 锁定状态（空闲、已锁定）—— 独立于生命周期状态，表示并发控制状态。两者正交，互不干扰。例如：一个「生效」状态的口径可以同时处于「已锁定」状态
- 锁定时记录：锁定项目ID、锁定时间
- 创建人、创建时间、最新更新时间、责任人
- `[修订]` 口径标签（支持多个标签，如：核心指标、派生指标、同比环比类、高频变更等，用于分类检索和资产发现）
- 版本历史（关联所有历史版本的业务口径、技术口径、要素）
- 关联项目列表（该口径参与的所有项目）

核心规则：

- **版本号规则**：采用"主版本号.次版本号"格式。项目内的多次修改递增次版本号（如 V1.0 -> V1.1 -> V1.2）；项目上线后，该口径的版本定格为本次项目的最终版本；下一个项目再次修改该口径时，主版本号递增、次版本号归零（如上次项目定格为 V1.3，下次项目首次修改为 V2.0，后续修改为 V2.1、V2.2）。

- **版本演进示例**：

  | 项目  | 操作                       | 口径版本变化         | 说明                     |
  | ----- | -------------------------- | -------------------- | ------------------------ |
  | 项目A | 首次创建口径并提交业务口径 | V1.0                 | 口径初始版本             |
  | 项目A | 澄清阶段修改2次            | V1.0 -> V1.1 -> V1.2 | 次版本号递增             |
  | 项目A | 上线定格                   | V1.2（生效）         | 版本定格为生效版本       |
  | 项目B | 首次修改该口径             | V2.0                 | 主版本号递增，次版本归零 |
  | 项目B | 开发过程中修改1次          | V2.0 -> V2.1         | 次版本号递增             |
  | 项目B | 验收驳回后修改1次          | V2.1 -> V2.2         | 次版本号继续递增         |
  | 项目B | 上线定格                   | V2.2（生效）         | 版本定格为生效版本       |

- `[修订]` **版本号边界场景规则**：

  | 场景                                     | 规则                                                                                                                                                                                                               |
  | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
  | 项目取消/废弃                            | 该项目产生的所有口径版本标记为「已废弃」，不计入正式版本序列。下一个项目修改该口径时，基于最近一个「生效」版本递增主版本号。例如：V1.2（生效）-> 项目B创建V2.0后项目B取消（V2.0废弃）-> 项目C关联时起始版本为 V3.0 |
  | 项目移除口径关联                         | 该口径在该项目中已产生的版本标记为「已废弃」，释放口径锁定。下一个项目关联时按照上述取消规则处理                                                                                                                   |
  | 口径从未有生效版本（首次创建的项目取消） | 口径本身标记为「草稿」状态（因为从未上线生效过），下一个项目关联时，主版本号重新从 V1.0 开始                                                                                                                       |
  | 口径在项目中创建但未提交业务口径就被移除 | 不产生任何版本记录，口径保持原状态                                                                                                                                                                                 |

- **版本快照规则**：每个版本为深拷贝（Snapshot），记录该版本时刻的完整要素列表及其所有属性。版本一旦生成不可修改。

- `[修订]` **要素描述修改规则**：仅修改要素的描述信息（如业务含义的文字润色、补充说明），不触发版本递增，视为对当前版本的补充说明。但修改内容记录在变更日志中（修改人、修改时间、修改前后内容），确保可追溯。版本对比时，展示的是版本快照时刻的描述信息，不受后续补充修改影响。

- 口径的结构性变更（新增要素、删除要素、修改要素类型、调整口径规则等），均生成新的版本，历史版本保留，不可修改，仅可查看。

- 同一个口径可在同一个项目中多次修改，每次修改对应一个新的次版本号。

- **口径锁定机制**：当口径被某个进行中的项目关联修改时，自动进入「已锁定」状态，其他项目无法同时修改同一口径。锁定信息包括锁定项目ID、锁定时间。口径锁定释放条件：项目上线归档后自动释放；项目取消/废弃后自动释放；或项目主动移除该口径的关联后释放。锁定期间，其他项目在创建时可查看该口径，但无法将其加入关联口径列表进行修改。

- 口径废弃后，关联的要素、业务口径、技术口径、版本历史均保留，仅不可再参与新的项目迭代。

## 2.3 要素（表字段）

定义：口径（表）的最小组成单元，对应表中的字段/指标/属性，是业务人员最关心的核心需求点，也是数据开发的最小落地单元。

核心属性：

- 要素ID（唯一标识）
- 要素名称（如：用户ID、支付金额、活跃状态、下单时间）
- 所属口径（关联唯一口径ID，随口径版本同步变更）
- 要素类型（如：字符串、数字、日期、布尔值）
- 业务含义（业务人员对该要素的解释）
- 是否必填、默认值（可选）
- 创建版本（该要素首次新增时的口径版本）
- 变更记录（修改人、修改时间、修改内容）

核心规则：要素归属口径，口径版本更新时，若要素有修改，自动关联至新的口径版本，历史版本的要素状态保留。

`[修订]` **后续扩展点 - 数据元**：当前版本中要素独立归属于各口径。未来若出现跨口径的字段标准化需求（如统一「用户ID」的定义），可引入「数据元」概念作为要素的全局标准定义，口径中的要素变成对数据元的引用 + 本口径的特定配置。此扩展不影响当前版本的数据模型设计。

## 2.4 业务口径

定义：由业务人员提交，用于描述口径的业务规则、统计逻辑、要素要求，是业务需求的正式载体。每个业务口径对应一个口径版本，业务人员每次修改业务口径，口径版本同步递增。业务口径的修改主要发生在需求澄清阶段，但只要项目未上线（即项目状态不为「已上线」或「已归档」），业务人员均可修改业务口径。

核心属性：

- 业务口径ID（唯一标识）
- 关联口径ID（对应哪个口径）
- 关联口径版本（对应该口径的哪个版本）
- 关联项目ID（属于哪个项目的需求）
- 业务规则（详细描述口径的统计逻辑、计算范围、业务场景）
- 要素要求（新增/修改/删除的要素列表，及要素的业务含义、校验规则）
- 提交人（业务人员）、提交时间、修改记录
- `[修订]` 状态（草稿、已提交、已澄清、已废弃）—— 完整状态机见附录A

`[修订]` **业务口径状态机**：

| 当前状态 | 触发动作               | 目标状态 | 操作人     | 说明                                             |
| -------- | ---------------------- | -------- | ---------- | ------------------------------------------------ |
| 草稿     | 提交业务口径           | 已提交   | 业务人员   | 首次提交或项目创建时自动提交                     |
| 已提交   | 开发发起澄清评论       | 已提交   | 数据开发   | 状态不变，触发通知业务人员                       |
| 已提交   | 业务修改业务口径       | 已提交   | 业务人员   | 版本递增，生成新业务口径记录                     |
| 已提交   | 项目负责人确认澄清完成 | 已澄清   | 项目负责人 | 需所有关联口径的业务口径均操作                   |
| 已澄清   | 业务再次修改业务口径   | 已提交   | 业务人员   | 版本递增，状态回退为「已提交」，触发下游影响提示 |
| 任意状态 | 项目取消               | 已废弃   | 系统自动   | 随项目取消自动废弃                               |

## 2.5 技术口径

定义：由数据开发人员提交，对应业务口径的技术实现方案，记录开发逻辑、代码、数据来源等技术细节。技术口径基于项目关联的指定版本业务口径进行编写（项目可选择关联某一特定版本的业务口径），需经过组内审核、组长审批，审批通过后方可生效，作为项目推进的前提条件。

技术口径版本关联规则：技术口径跟随业务口径版本，每个业务口径版本都可以补充对应的技术口径。开发人员在口径详情页面点击"技术口径"时，可以选择任意版本的业务口径进行技术口径补充（例如：业务口径更新到V2.1后，开发人员既可以选择V2.1版本进行补充，也可以选择V2.0版本进行补充）。

核心属性：

- 技术口径ID（唯一标识）
- 关联业务口径ID（对应的业务口径，一对一关系；项目可选择关联指定版本的业务口径）
- 关联口径ID、关联口径版本（与业务口径对应）
- 关联项目ID
- 技术实现细节：数据来源（关联的原始表、字段）、ETL逻辑、SQL代码片段、加工规则
- `[修订]` 数据来源口径（结构化字段，记录该技术口径引用了哪些上游口径/源表，用于构建血缘关系）
- 开发人（数据开发）、开发时间、修改记录
- `[修订]` 状态（未开发、开发中、审核中、待审批、审批通过、审批驳回、待重新评估、已废弃）—— 完整状态机见附录A
- 预览数据状态（未就绪、已就绪），就绪时记录：标记人、标记时间、预览数据说明
- 审核记录（审核人、审核意见、审核时间）
- 问题列表（组内提出的问题、提出人、提出时间、修复状态、修复人、修复时间）
- 审批记录（审批人、审批意见、审批时间）

`[修订]` **技术口径状态机**：

| 当前状态   | 触发动作                   | 目标状态   | 操作人   | 说明                               |
| ---------- | -------------------------- | ---------- | -------- | ---------------------------------- |
| 未开发     | 开发人员开始编写           | 开发中     | 数据开发 | -                                  |
| 开发中     | 提交审核                   | 审核中     | 数据开发 | 自动通知组内成员                   |
| 审核中     | 所有问题处理完成，提交审批 | 待审批     | 数据开发 | 需所有问题为「已修复」或「已驳回」 |
| 待审批     | 组长审批通过               | 审批通过   | 开发组长 | 开发人员需去机房生成预览数据       |
| 待审批     | 组长审批驳回               | 审批驳回   | 开发组长 | 附驳回原因                         |
| 审批驳回   | 开发人员修改后重新提交审核 | 审核中     | 数据开发 | 重新走完整审核流程                 |
| 审批通过   | 关联的业务口径被修改       | 待重新评估 | 系统自动 | 提示开发人员评估影响               |
| 待重新评估 | 开发人员确认无需修改       | 审批通过   | 数据开发 | 记录确认意见                       |
| 待重新评估 | 开发人员修改后重新提交审核 | 审核中     | 数据开发 | 重新走完整审核审批流程             |
| 审批通过   | 验收驳回（技术实现问题）   | 开发中     | 系统自动 | 退回开发阶段                       |
| 任意状态   | 项目取消                   | 已废弃     | 系统自动 | 随项目取消自动废弃                 |

## 2.6 核心关联关系（关键）

- 主题 <-> 口径：一对多（一个主题包含多个口径，一个口径仅归属一个主题）。

- 口径 <-> 要素：一对多（一个口径包含多个要素，一个要素仅归属一个口径）。

- 口径 <-> 版本：一对多（一个口径对应多个版本，每个版本对应一套完整的业务口径 + 技术口径 + 要素快照）。`[修订]` 版本状态包括：草稿、生效、已废弃。

- 业务口径 <-> 技术口径：一对零或一（每个业务口径最多对应一个技术口径，也可能暂无对应的技术口径；技术口径基于项目关联的指定版本业务口径编写），核心关联条件：项目ID + 口径ID + 口径版本。

- 项目 <-> 口径/业务口径/技术口径：一对多（一个项目可包含多个口径、多个业务口径、多个技术口径，对应本次迭代的所有需求变更）。

- 口径 <-> 锁定：一个口径同一时间最多被一个进行中的项目锁定，确保不会出现并发修改冲突。

- `[修订]` 技术口径 -> 数据来源口径/源表：多对多（一个技术口径可引用多个上游数据源，一个源表/口径可被多个技术口径引用），用于构建数据血缘关系图谱。

# 三、用户角色与职责（明确分工）

## 3.1 系统角色

| 角色     | 核心职责                                                                                 | 关键操作                                                                                                                                                    |
| -------- | ---------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 业务人员 | 发起项目、提交业务口径、澄清需求、参与验收、查看口径版本和要素                           | 创建项目、新增/修改业务口径、评论澄清需求、验收数据、查询口径/要素/版本、查看口径版本对比结果                                                               |
| 数据开发 | 澄清需求、编写技术口径、数据开发、配合组内审核、修复技术口径问题、提交组长审批、配合验收 | 查看业务口径、编写/修改技术口径、提交技术口径审核、查看审核问题、修复问题、提交组长审批、在数据开发平台生成预览数据、配合业务验收、查看技术口径版本对比结果 |
| 开发组长 | 组织组内审核、审核技术口径的合理性、把控开发质量、审批技术口径                           | 审核技术口径、提出技术口径问题、审核技术口径修复结果、审批技术口径（通过/驳回）、给出审核/审批意见、委托审批权限                                            |
| 管理员   | 维护平台配置、管理角色权限、监控项目流程、处理异常、归档数据                             | 配置角色权限、处理流程异常、审核口径/主题的新增/删除、查看全量日志、归档项目、查看技术口径审核/审批记录、处理审核/审批异常、管理评论（标记隐藏）            |

上述角色为系统级别的角色定义，用于控制用户在平台上的基本操作权限。

`[修订]` **开发组定义**：数据开发人员按组织架构归属于「开发组」，每个开发组有一名或多名「开发组长」。技术口径的审批规则为：**技术口径的审批人 = 该技术口径开发人员所在组的组长**。管理员在「组织管理」模块中维护开发组与人员的归属关系。

## 3.2 项目角色

每个项目在创建时需指定以下项目角色（均支持多人）：

- **项目负责人**（多个）：负责项目的整体把控，可为业务人员或管理人员，拥有项目全流程的查看权限和关键节点的操作权限。`[修订]` 新增职责：确认需求澄清完成（触发项目状态从「待澄清」转为「开发中」），确认验收通过（触发项目状态从「待验收」转为「待上线」）。
- **项目开发负责人**（多个）：负责项目的技术开发管理，拥有将项目状态流转为「待验收」的操作权限，协调开发团队完成所有口径的技术实现。

> 项目角色与系统角色（业务人员、数据开发、开发组长、管理员）独立，是对项目维度的补充。同一用户可以在不同项目中担任不同的项目角色。

## 3.3 口径责任人 `[新增]`

每个口径有一名「口径责任人」，职责包括：

- 负责口径信息准确性的日常维护（如修正描述错误、更新联系人等非结构性变更）
- 口径废弃操作的发起人（废弃需经管理员审批）
- 当口径被多个项目关注（如锁定排队）时，协调优先级
- 口径责任人默认为口径创建人，可由管理员变更

# 四、核心业务流程（项目驱动增量迭代，闭环管理）

整体流程：业务创建项目 -> 提交业务口径（含要素变更） -> 需求澄清 -> 数据开发（编写技术口径） -> 组内审核 -> 组长审批 -> 业务验收 -> 项目上线 -> 口径版本更新、资产归档。

`[修订]` 项目状态完整流转：**待澄清 -> 开发中 -> 待验收 -> 待上线 -> 已上线 -> 已归档**。另有两个终止/挂起状态：**已暂停**（可恢复）、**已取消**（不可恢复）。其中"组内审核"和"组长审批"发生在"开发中"阶段内部，以技术口径状态驱动。

以下为详细流程步骤：

## 4.1 步骤1：业务创建项目（增量迭代入口）

业务人员基于新增/变更需求，创建项目，明确本次迭代的核心内容：

- 项目基础信息：项目名称、所属部门、项目描述（本次迭代的核心目标）、期望上线时间、项目负责人、项目开发负责人等。

- 关联口径选择：勾选本次需要变更的口径（可跨多个主题、多个口径）。系统自动校验所选口径的锁定状态，若口径已被其他项目锁定，则无法选择，页面提示锁定信息（锁定项目名称、锁定时间）。

- `[修订]` **口径开发类型标记**：关联口径时，须为每个口径标记本次迭代的开发类型：
  - **需技术开发**：口径存在结构性变更（新增/修改/删除要素、调整业务规则），需要数据开发编写技术口径并完成审核审批。
  - **仅业务口径调整**：口径仅做业务口径层面的文字修订（如修正业务规则描述、补充统计场景说明），不涉及要素结构变更，不需要编写技术口径。

  > 该标记影响项目进入「待验收」的前置条件判断，详见步骤5。

- 口径锁定：项目创建成功后，所有关联口径自动进入「已锁定」状态，绑定当前项目ID。

- 初始业务口径提交：针对每个关联口径，提交初始版本的业务口径（含要素新增/修改/删除需求、业务规则），生成对应口径的新主版本（如口径上一次生效版本为 V1.3，则本次初始版本为 V2.0）。

- 提交后，项目状态为「待澄清」，自动通知对应的数据开发人员。

## 4.2 步骤2：需求澄清（业务 <-> 开发，多轮沟通）

核心目标：对齐业务口径的细节，解决歧义，确保开发理解准确，支持多轮澄清、多轮修改业务口径。需求澄清是业务口径修改最集中的阶段，但业务人员在项目上线前的任何阶段（待澄清、开发中、待验收）均可修改业务口径。

- 数据开发查看项目中的业务口径和要素需求，针对不明确的点（如要素含义、统计逻辑），在平台内发起评论，@业务人员澄清。

- 业务人员看到评论后，补充说明，可修改业务口径内容（如调整要素要求、完善业务规则），修改后自动递增口径次版本号（如 V2.0 -> V2.1），同时生成新的业务口径记录（与原业务口径关联同一项目、同一口径）。

- `[修订]` **澄清完成确认**：当所有关联口径的业务口径均完成澄清后，由**项目负责人**在项目详情页点击「确认澄清完成」，系统校验所有业务口径状态均为「已澄清」后，项目状态自动变更为「开发中」。若部分口径尚未澄清，系统提示未澄清口径列表，项目负责人可选择等待或标记为已澄清。

- `[修订]` **业务口径修改影响提示**：业务人员在「开发中」「待验收」阶段修改业务口径时，系统自动检测并提示：
  - 该口径是否已有对应的技术口径？当前状态是什么？
  - 若关联的技术口径已审核/审批通过，系统提示：「修改业务口径将触发关联的技术口径进入'待重新评估'状态，开发人员需重新确认技术口径是否需要调整」
  - 业务人员确认后方可提交修改

- 示例：业务发起项目，对A口径提交 V2.0 业务口径（要素变更），澄清后发现问题，修改为 V2.1 业务口径（新的业务口径记录），达成一致后，项目负责人确认澄清完成，进入开发阶段。

## 4.3 步骤3：数据开发（编写技术口径，落地需求）

- 数据开发基于项目关联的指定版本业务口径（项目可选择关联某一特定版本），进行数据加工开发：多表关联、ETL处理、字段计算等。

- 开发完成后，在平台提交技术口径，关联项目指定版本的业务口径，记录技术实现细节（SQL代码、数据来源、加工逻辑），提交后技术口径状态改为「审核中」。

- `[修订]` **数据来源声明**：提交技术口径时，开发人员需在「数据来源口径」字段中结构化填写该技术口径引用的上游口径/源表（如：引用了「订单明细口径表 V1.2」「用户基础表」），用于构建数据血缘关系。此字段为非必填，但建议尽可能完整填写。

- 若业务在开发过程中再次修改业务口径（如发现遗漏需求），业务口径自动递增版本（如 V2.1 -> V2.2），`[修订]` 若该口径已有审批通过的技术口径，技术口径状态自动变为「待重新评估」；开发人员评估后，若需调整则修改技术口径并重新提交审核，若无需调整则确认无影响。

- 数据开发的实际加工操作（ETL处理、SQL执行等）在外部数据开发平台完成，本平台仅记录技术口径（技术实现方案）。两个平台之间网络隔离（数据开发平台仅可在机房访问，数据管理平台支持办公网和机房访问），不存在接口互通，所有跨平台操作均由人工完成。

## 4.4 步骤4：组内审核

核心目标：确保技术口径的合理性、准确性、规范性，避免开发偏差，所有问题修复完成后，方可进入组长审批环节。

- 审核发起：技术口径提交审核后，自动通知开发组内成员（除开发本人外），技术口径状态为「审核中」。

- 组内审核：组内成员查看技术口径的技术实现细节、关联的业务口径，针对逻辑漏洞、代码问题、与业务口径不符等情况，提出具体问题，记录在技术口径的「问题列表」中，明确问题描述、提出人、提出时间。

- 问题修复：开发人员查看组内提出的问题，逐一修复，完成修复后标记问题状态为「已修复」，填写修复说明、修复时间。若认为问题不合理，可与提出人在线沟通，沟通一致后可驳回问题（需注明驳回原因）。

- 重复审核：若组内成员对修复结果有异议，可再次提出问题，开发人员重新修复，直至所有问题均为「已修复」或「已驳回」状态。

- 提交审批：所有问题处理完成后，开发人员提交技术口径至开发组长，申请审批，技术口径状态改为「待审批」。`[修订]` 审批人自动指定为该开发人员所在组的组长。

## 4.5 步骤5：组长审批

- 组长查看：开发组长查看技术口径的完整信息（技术实现、审核记录、问题列表及修复情况、关联的业务口径）。

- 审批操作：
  - 审批通过：确认技术口径无问题，提交审批意见，技术口径状态改为「审批通过」。开发人员在机房的数据开发平台生成预览数据（小批量样本），完成后回到数据管理平台手动标记"预览数据已就绪"（可填写预览数据说明，如数据行数、数据存放位置等）。`[修订]` 当项目满足以下条件时，项目开发负责人可将项目状态流转为「待验收」：
    - 所有标记为「需技术开发」的口径，其技术口径均已「审批通过」且「预览数据已就绪」
    - 所有标记为「仅业务口径调整」的口径，无需技术口径（自动视为就绪）
    - 不存在状态为「待重新评估」的技术口径

    流转为「待验收」后，自动通知项目负责人和业务人员前往机房的数据开发平台查看验收。

  - 审批驳回：若发现技术口径仍有问题（如修复不彻底、逻辑不符），填写驳回原因，技术口径状态改为「审批驳回」，退回至开发人员，开发人员重新修改技术口径并提交审核，重复步骤4.4-4.5，直至审批通过。

- 委托机制：组长可将审批权限临时委托给指定人员（需为同组成员），委托期间由被委托人执行审批操作，委托记录留痕（委托人、被委托人、委托时间、委托结束时间）。超过设定时限（如48小时）未审批的技术口径，系统自动提醒组长及其上级。

- 规则约束：技术口径未审批通过前，项目无法进入验收环节，确保技术实现的规范性。

## 4.6 步骤6：业务验收（确认需求落地准确性）

- 业务人员前往外部数据开发平台查看预览数据，核对要素的准确性、业务规则的符合性（与项目关联版本的业务口径一致），可在本平台通过口径版本对比功能，查看当前版本与历史版本的要素变更差异，辅助验收。

- `[修订]` **验收粒度**：验收按口径维度进行，每个口径独立验收。项目中所有口径均验收通过后，项目整体验收完成。

- 验收结果分为两种：
  - 验收通过：业务人员逐个确认口径验收通过，提交验收意见。当所有口径均验收通过后，由**项目负责人**确认项目整体验收完成，项目状态改为「待上线」。

  - 验收不通过：分为以下两种场景，业务人员需在验收意见中注明驳回类型和原因。
    - 场景一：业务口径存在问题（需求定义有误或遗漏）。业务人员填写驳回原因，`[修订]` **仅该口径**退回至开发阶段（该口径的技术口径状态变为「开发中」），业务修改业务口径（自动递增次版本号），开发基于新版本业务口径重新进行数据开发，重复步骤4.3-4.5，直至验收通过。项目整体状态退回「开发中」。

    - 场景二：技术开发实现有问题（业务口径正确，但数据结果不符合预期）。业务人员填写驳回原因，`[修订]` 该口径的技术口径状态自动变为「开发中」，开发人员修改技术口径后重新提交审核，重复步骤4.4-4.5，直至验收通过。本场景下业务口径无需修改，不产生新的口径版本。项目整体状态退回「开发中」。

  - 重要说明：审批已通过的技术口径不存在「审批回退」概念。验收驳回后，技术口径状态变为「开发中」，开发人员修改后需重新走完整的审核 -> 审批流程。

## 4.7 步骤7：项目上线与归档（闭环收尾）

- 项目验收通过后，由项目开发负责人或管理员执行上线操作，更新项目状态为「已上线」。

- 上线后，自动更新相关口径的版本：将本次项目中最终确认的业务口径、技术口径、要素，关联为该口径的最新生效版本（如 V2.3），历史版本保留，可随时查看、对比。

- 口径锁定释放：上线完成后，项目关联的所有口径自动解除锁定，恢复为「空闲」状态，允许后续项目关联修改。

- 项目归档：上线完成后，项目自动进入「已归档」状态，所有需求记录、业务口径、技术口径、审核记录、审批记录、验收记录均保留，不可修改，仅可查询。

- 数据资产更新：主题、口径、要素的关联关系、状态，同步更新为最新状态，供后续项目迭代复用。

- 上线后问题处理：若上线后发现数据问题，不支持在已归档项目上直接回退或热修复，需由业务人员重新创建新项目，走完整的项目流程（提交业务口径 -> 澄清 -> 开发 -> 审核 -> 审批 -> 验收 -> 上线），确保每一次变更均有完整的审核追溯记录。

- 紧急代码问题处理：若上线后数据开发平台出现紧急代码问题（如影响生产数据的严重bug），此类问题的应急处理全程在本平台之外完成——相关人员在自己电脑上通过邮件向领导说明情况、在企业审批平台发起紧急上线申请、审批通过后在数据开发平台进行代码修复上线。本数据管理平台仅涉及事后补录环节：应急处理完成后，需在本平台创建新项目，补录变更记录，确保口径版本的完整追溯。

## 4.8 项目暂停与取消 `[新增]`

### 4.8.1 项目暂停

- **触发条件**：项目在「待澄清」「开发中」「待验收」状态下，项目负责人或管理员可执行暂停操作。
- **暂停影响**：
  - 项目状态变为「已暂停」，记录暂停原因、暂停时间、操作人。
  - 项目关联的口径**保持锁定**（暂停不等于放弃，口径仍被该项目占用）。
  - 所有进行中的技术口径审核/审批流程冻结，不可继续操作。
  - 暂停期间，业务人员仍可查看项目信息，但不可修改业务口径。
- **恢复条件**：项目负责人或管理员执行「恢复」操作，项目状态恢复为暂停前的状态，所有冻结流程继续。
- **超时处理**：暂停超过30天（可配置），系统自动提醒项目负责人，超过90天仍未恢复，系统提醒管理员考虑是否取消。

### 4.8.2 项目取消

- **触发条件**：项目在「待澄清」「开发中」「待验收」「已暂停」状态下，项目负责人发起取消申请，需经管理员审批。
- **取消影响**：
  - 项目状态变为「已取消」，记录取消原因、取消时间、审批人。
  - 项目关联的所有口径**自动解除锁定**，恢复为「空闲」状态。
  - 该项目产生的所有口径版本标记为「已废弃」（不计入正式版本序列，但数据保留可查）。
  - 该项目的所有业务口径、技术口径状态标记为「已废弃」。
  - 项目进入「已取消」状态后不可恢复，所有记录保留，仅可查询。
- **重要说明**：已取消项目产生的口径版本号不影响后续版本号计算。例如：口径最近生效版本为 V1.2，项目B创建了 V2.0 后取消（V2.0标记为废弃），项目C关联该口径时起始版本为 V3.0（主版本号继续递增，确保版本号全局唯一不回退）。

# 五、核心功能模块详情

按"数据资产管理 + 项目流程管理 + 协作验收 + 数据看板 + 权限审计 + 通知提醒"分类，明确每个模块的核心功能。

## 模块1：数据资产管理（核心模块）

### 1.1 主题管理

- 新增主题：填写主题名称、所属部门、描述，提交后生成唯一主题ID。

- 编辑/删除主题：编辑主题基本信息；删除主题需校验无关联口径，否则无法删除。

- 主题查询：支持按名称、所属部门检索，查看主题详情及关联的所有口径。

### 1.2 口径管理

- 新增口径：选择所属主题，填写口径名称、描述、标签，指定口径责任人，生成初始版本（V1.0），状态为「草稿」。

- 口径版本管理：查看口径的所有历史版本（含生效版本和废弃版本，废弃版本以灰色标记），每个版本关联对应的业务口径、技术口径、要素；支持切换版本查看详情。

- 口径版本对比：支持选择同一口径的任意两个版本，进行要素变更对比，清晰展示变更细节：
  - 新增要素：标注新增的要素名称、类型、业务含义等信息。
  - 修改要素：标注修改的要素，对比修改前后的属性（如业务含义、类型、必填项等）差异。
  - 删除要素：标注删除的要素名称及相关属性。
  - 对比结果支持导出，便于留存和沟通。

- 口径锁定状态查看：在口径列表和详情页展示锁定状态（空闲/已锁定）、锁定项目名称、锁定时间，便于用户了解口径当前是否可编辑。

- 口径查询：支持按名称、所属主题、状态、版本、锁定状态、`[修订]` 标签检索，查看口径的完整信息（版本历史、关联要素、关联项目）。`[修订]` 支持全文搜索（在口径名称、描述、业务规则中搜索关键词）。

- 口径废弃：由口径责任人发起，经管理员审批后，将无需使用的口径标记为「废弃」，保留所有历史数据，不可再关联新项目。

- `[修订]` **口径影响分析**：在口径详情页展示该口径的影响范围，包括：
  - 当前关联的项目（进行中的项目）
  - 历史关联的项目（已归档的项目）
  - 引用该口径作为数据来源的其他技术口径（基于血缘关系）

### 1.3 要素管理

- 要素新增/修改/删除：业务人员在提交业务口径时，新增/修改/删除要素；开发人员不可直接修改要素，仅可通过技术口径关联要素。

- 要素批量操作：支持通过 Excel/CSV 文件批量导入要素（含要素名称、类型、业务含义等），系统校验格式和重复项后批量创建；支持批量导出要素列表。

- 要素查询：支持按要素名称、所属口径、要素类型检索，查看要素的详细信息、变更记录、关联的口径版本。

- 要素关联查看：查看要素所属的口径、主题，及该要素在不同口径版本中的变更情况，可联动口径版本对比功能查看详情。

### 1.4 口径台账（核心查询页面）

汇总所有口径的核心信息，支持业务和开发快速查询，包含：口径名称、所属主题、当前版本、锁定状态、`[修订]` 口径标签、口径责任人、最近更新时间、关联项目、业务口径摘要、技术口径摘要，支持检索和导出。

### 1.5 数据导入（初始化与迁移）

- 批量导入口径：支持通过 Excel/CSV 模板批量导入口径及其要素，适用于平台初始化阶段的存量数据迁移。

- 导入模板下载：提供标准导入模板（含主题、口径、要素的字段说明和示例数据）。

- 导入校验：系统对导入数据进行格式校验、必填项校验、重复项校验，校验不通过的条目标注错误原因，支持修正后重新导入。

- 导入记录：记录每次导入的操作人、导入时间、导入数量、成功/失败条目数，支持查看导入详情。

### 1.6 数据血缘（基础版） `[新增]`

- **血缘来源**：基于技术口径中的「数据来源口径」字段，自动构建口径之间的上下游引用关系。

- **血缘查看**：在口径详情页展示该口径的上游来源（数据从哪里来）和下游引用（数据被哪些口径使用），以可视化图谱形式展示。

- **影响分析联动**：结合血缘关系，在修改口径时展示下游影响范围。

- **数据质量**：血缘关系的完整性依赖于开发人员在技术口径中主动填写数据来源，平台以「数据来源填写率」指标鼓励完善。

> 注：本版本的血缘为「手动声明式血缘」，未来可扩展为自动解析SQL的血缘分析。

## 模块2：项目管理（迭代流程核心）

### 2.1 项目创建与编辑

- 项目创建：业务人员填写项目基础信息（含项目负责人、项目开发负责人），勾选关联口径并标记开发类型（系统校验口径锁定状态），提交初始业务口径，生成项目ID。

- 项目编辑：仅可在「待澄清」「开发中」状态下，编辑项目基础信息、新增/删除关联口径（新增时校验锁定状态，删除时释放口径锁定并废弃相关版本）、修改业务口径。

- 项目查询：支持所有角色按项目名称、状态、所属部门、时间检索，查看项目详情，包含技术口径审核、审批进度。

- `[修订]` 项目状态流转：完整状态机见附录A。关键流转规则：

  | 当前状态                    | 触发动作     | 目标状态   | 操作人                     | 前置条件                                                                                           |
  | --------------------------- | ------------ | ---------- | -------------------------- | -------------------------------------------------------------------------------------------------- |
  | 待澄清                      | 确认澄清完成 | 开发中     | 项目负责人                 | 所有业务口径状态为「已澄清」                                                                       |
  | 开发中                      | 提交验收     | 待验收     | 项目开发负责人             | 所有「需技术开发」口径的技术口径均「审批通过」且「预览数据已就绪」；无「待重新评估」状态的技术口径 |
  | 待验收                      | 确认验收通过 | 待上线     | 项目负责人                 | 所有口径均验收通过                                                                                 |
  | 待验收                      | 验收驳回     | 开发中     | 业务人员                   | 至少一个口径验收不通过                                                                             |
  | 待上线                      | 执行上线     | 已上线     | 项目开发负责人/管理员      | -                                                                                                  |
  | 已上线                      | 自动归档     | 已归档     | 系统自动                   | 上线操作完成                                                                                       |
  | 待澄清/开发中/待验收        | 暂停         | 已暂停     | 项目负责人/管理员          | -                                                                                                  |
  | 已暂停                      | 恢复         | 暂停前状态 | 项目负责人/管理员          | -                                                                                                  |
  | 待澄清/开发中/待验收/已暂停 | 取消         | 已取消     | 项目负责人（需管理员审批） | -                                                                                                  |

### 2.2 业务口径管理（业务主导）

- 业务口径提交：针对项目中的每个口径，填写业务规则、要素需求，提交后生成业务口径记录，口径版本自动递增。

- 业务口径修改：项目上线前的任何阶段（待澄清、开发中、待验收），业务人员均可修改业务口径，修改主要集中在需求澄清阶段。修改后生成新的业务口径记录，关联同一项目和口径，口径版本自动递增次版本号。`[修订]` 修改时系统展示影响提示（详见4.2步骤2）。

- 业务口径查看：查看所有业务口径，关联的项目、口径、版本、技术口径（如有），及修改记录。

### 2.3 技术口径管理（开发主导）

- 技术口径提交：开发人员基于项目关联的指定版本业务口径，填写技术实现细节（SQL代码、数据来源、加工逻辑），关联对应版本的业务口径，提交后状态改为「审核中」。

- 技术口径修改：在开发中、审核中、审批驳回、`[修订]` 待重新评估阶段，可修改技术口径，可更新关联至项目指定版本的业务口径，修改后需重新提交审核。

- 技术口径查看：查看所有技术口径，关联的业务口径、项目、口径、版本，及开发记录、审核记录、问题列表、审批记录。

- 技术口径版本对比：支持选择同一技术口径的任意两个版本，对比技术实现细节的变更差异，清晰展示：
  - 技术实现细节变更：如SQL代码、数据来源、ETL逻辑、加工规则的修改内容，标注修改前后的差异。
  - 关联业务口径变更：若关联的业务口径版本有调整，同步展示关联关系的变更。
  - 审核/问题记录变更：对比不同版本的审核意见、问题列表及修复情况差异。
  - 对比结果支持导出，便于技术复盘和问题追溯。

- 技术口径审核：
  - 审核查看：组内成员可查看待审核的技术口径，提出问题、查看他人提出的问题。
  - 问题管理：开发人员可修复、驳回问题，标记问题状态，填写修复说明。
  - 审核记录：自动记录所有审核操作、问题及修复情况，不可修改，仅可查看。

- 技术口径审批：
  - 审批提交：开发人员在所有问题处理完成后，提交技术口径至开发组长审批。
  - 审批操作：开发组长查看技术口径及相关记录，提交审批意见（通过/驳回），记录审批时间和原因。
  - 审批委托：组长可临时委托审批权限给指定同组成员，委托期间、委托人信息留痕记录。
  - 审批追溯：所有审批记录永久保留，可随时查看，支持关联查看对应的审核记录和问题修复记录。

## 模块3：协作与验收

### 3.1 需求澄清协作

- 评论功能：项目详情页、业务口径详情页内置评论区，支持业务和开发互相评论、@成员、上传图片/文档，所有评论留痕，支持管理员标记隐藏不当评论（隐藏操作留痕记录，原始内容保留于系统中）。

- 通知提醒：业务提交/修改业务口径、开发发起澄清、评论@成员时，相关人员收到平台通知（站内消息 + `[修订]` 邮件通知）。

### 3.2 业务验收

- 预览数据查看：开发人员在外部数据开发平台生成预览数据（小批量样本）后，在本平台标记"预览数据已就绪"；业务人员收到通知后，前往数据开发平台查看预览数据，核对数据准确性。

- `[修订]` 验收操作：业务人员在本平台按**口径维度**逐个提交验收结果（通过/不通过），填写验收意见，不通过需注明驳回类型（业务口径问题 / 技术实现问题）和原因。所有口径验收通过后，项目负责人确认项目整体验收完成。

- 验收记录：保留所有验收记录，关联对应的业务口径、技术口径，可追溯，支持联动口径版本对比功能辅助验收。

## 模块4：数据看板（全局概览）

### 4.1 项目进度看板

- 项目状态分布：按状态（待澄清、开发中、待验收、待上线、已上线、已归档、`[修订]` 已暂停、已取消）统计项目数量，支持图表展示。

- 逾期项目预警：展示超过期望上线时间仍未上线的项目列表，标注逾期天数。

- 项目趋势：按时间维度展示项目创建、上线数量趋势。

### 4.2 口径资产概览

- 资产统计：展示主题总数、口径总数、要素总数、各状态口径数量（草稿/生效/废弃）、`[修订]` 锁定中的口径数量（独立统计）。

- 版本变更趋势：按时间维度展示口径版本变更频率，识别高频变更口径。

- 主题分布：按主题维度展示口径和要素的分布情况。

- `[修订]` **口径质量指标**：
  - 口径完整度：统计各口径的信息填写完整率（如：描述、标签、责任人是否齐全）
  - 技术口径覆盖率：统计已有生效版本的口径中，技术口径的覆盖比例
  - 数据来源填写率：统计技术口径中「数据来源口径」字段的填写比例
  - 口径活跃度：统计口径被项目引用的频率，识别高复用口径和长期未使用的口径

### 4.3 个人工作台

- 待我处理：汇总当前用户待处理的事项，包括待澄清的业务口径、待审核的技术口径、待审批的技术口径、待验收的项目、`[修订]` 待重新评估的技术口径等。

- 我参与的项目：展示当前用户参与的所有项目及进度状态。

- 最近动态：展示与当前用户相关的最近操作记录（评论、审核、审批、验收等）。

## 模块5：权限与审计

### 5.1 角色权限控制（RBAC）

- 角色配置：管理员可新增、编辑角色，分配操作权限（如：业务人员仅可提需求、验收，开发人员仅可编写技术口径、提交审核，开发组长拥有技术口径审批权限）。

- 用户分配：管理员将用户分配至对应角色，实现权限隔离。

- `[修订]` **开发组管理**：管理员维护开发组与人员的归属关系（一个开发人员归属一个开发组，一个开发组有一名或多名组长），用于技术口径审批人的自动指定。

- 数据权限：按主题、部门分配数据查看权限（如：运营部业务人员仅可查看营销主题、用户主题的口径和要素）。

- 审批委托管理：管理员可查看所有审批委托记录，处理委托异常。

### 5.2 操作日志审计

- 日志记录：记录所有关键操作，包括项目创建/编辑/上线/`[修订]`暂停/取消、业务口径/技术口径新增/修改、审核/审批/验收操作、权限变更、口径锁定/解锁、评论隐藏、`[修订]` 口径废弃、开发组变更等。

- 日志查询：支持按操作人、操作类型、时间、项目ID、口径ID检索日志，查看操作详情。

## 模块6：通知与提醒

`[修订]` **通知通道**：

| 通道     | 说明                       | 适用场景                           |
| -------- | -------------------------- | ---------------------------------- |
| 站内消息 | 平台内实时消息，登录后可见 | 所有通知                           |
| 邮件通知 | 发送至用户注册邮箱         | 关键节点通知（审批、验收、逾期等） |

> 用户可在「个人设置」中配置各通知类型的接收通道偏好（站内消息默认开启不可关闭，邮件通知可按类型开启/关闭）。

- 关键节点提醒：项目状态变更、需求澄清评论、审核通知、审批通知、验收通知、上线提醒、技术口径问题提醒、口径锁定通知、`[修订]` 业务口径变更影响通知、技术口径待重新评估通知等。

- 逾期提醒：项目未按期望时间完成澄清、开发、审核、审批、验收时，提醒项目负责人和相关人员。

- 审批超时提醒：技术口径提交审批后超过设定时限（如48小时）未处理，自动提醒审批人及其上级。

- `[修订]` **通知聚合**：同一事项在短时间内（如1小时内）产生的多条通知，合并为一条聚合通知，避免信息轰炸。例如：一个项目在1小时内收到3条澄清评论，合并为「项目XX有3条新的澄清评论」。

# 六、非功能需求（保障平台可用、稳定、安全）

## 6.1 易用性

- 业务端：界面简洁，操作流程清晰，表单引导明确，关键操作路径（如创建项目、提交业务口径、验收）提供新手引导；版本对比功能操作简单，直观展示变更差异。

- 开发端：技术口径编辑支持代码高亮、语法校验；技术口径审核、问题修复、审批流程操作便捷。预览数据由数据开发平台生成和管理（仅可在机房访问），本平台仅记录就绪状态和验收结果，不与数据开发平台进行接口交互。

- 个人工作台：提供统一的待办入口，用户登录后可快速了解待处理事项，减少信息查找成本。

- `[修订]` **搜索与发现**：平台提供全局搜索功能，支持在口径名称、口径描述、业务规则、要素名称中进行全文关键词搜索，帮助用户快速发现和定位相关数据资产。

## 6.2 安全性

- 认证与登录：支持与企业现有身份认证系统对接（如 LDAP/SSO），支持账号密码登录；登录失败连续5次后锁定账号30分钟；session 超时自动登出（默认30分钟无操作）。

- 传输加密：所有数据传输使用 HTTPS 加密。

- 敏感数据保护：SQL代码、技术实现细节等敏感内容仅对有权限的角色可见；导出文件需记录操作日志。

- 接口安全：API 接口需进行身份验证和权限校验，防止未授权访问。

## 6.3 稳定性

- 核心流程（项目创建、口径提交、审核、审批、验收、上线）7x24小时稳定运行，支持高并发（同时500+用户操作）。

- 性能指标：列表查询页面响应时间不超过2秒；版本对比页面响应时间不超过3秒；批量导入（1000条以内）处理时间不超过30秒；`[修订]` 全文搜索响应时间不超过3秒。

- 数据备份：所有数据（项目、口径、要素、日志、技术口径审核/审批记录）每日自动备份，备份数据保留不少于90天，支持异常恢复。

## 6.4 可追溯性

- 所有变更均可追溯：口径版本、业务口径、技术口径、要素的每一次修改，均记录修改人、修改时间、修改内容；版本对比功能可快速追溯变更细节。

- 流程可追溯：项目的每一个节点（澄清、开发、审核、审批、验收），均记录操作人、操作时间、操作结果；技术口径的审核记录、问题列表、修复情况、审批记录均永久保留。

- `[修订]` 项目暂停与取消可追溯：暂停/恢复/取消操作均记录操作人、操作时间、原因，取消审批记录永久保留。

## 6.5 数据归档与存储策略

- 热数据：近1年内活跃的项目及相关口径、版本数据，存储于主库，保障查询性能。

- 冷数据：归档超过1年且无关联活跃项目的数据，自动迁移至冷存储（归档库），仍可查询但响应时间放宽至5秒以内。

- 数据保留：所有数据永久保留，不可物理删除（仅口径可标记为废弃），冷热分离策略确保存储成本可控。

## 6.6 可扩展性

- 预留接口：支持后续与BI工具、数据仓库、业务系统集成，实现数据同步和复用；支持版本对比功能、审核审批功能的扩展（如批量对比、批量审批）。

- 功能扩展：支持新增角色、新增需求类型、新增口径属性，适配未来业务发展。

- `[修订]` **后续扩展规划**：
  - 数据元管理：引入全局数据元标准，实现跨口径的字段标准化
  - 自动化血缘：基于SQL解析自动构建数据血缘，替代手动声明
  - 企业IM集成：对接企业微信/钉钉/飞书等企业IM的通知通道
  - 批量审批：支持组长对多个技术口径进行批量审批操作
  - 口径推荐：基于项目历史，推荐「使用了口径A的项目通常也使用口径B」

# 七、关键规则与约束（重点强调）

- **口径版本规则**：口径的每一次业务/技术变更，均生成新的版本，历史版本不可修改，仅可查看。项目内修改递增次版本号（如 V2.0 -> V2.1），项目上线后定格为最终版本；下一个项目修改该口径时，主版本号递增（如 V2.3 -> V3.0）。`[修订]` 项目取消/口径移除产生的版本标记为「已废弃」，不计入正式版本序列，但版本号全局唯一不回退。支持任意两个版本的要素对比、技术口径对比。

- **口径锁定规则**：口径被进行中的项目关联修改时自动锁定，其他项目无法同时修改同一口径，避免并发冲突。锁定随项目上线归档自动释放，`[修订]` 随项目取消自动释放，或在项目移除口径关联时释放。项目暂停期间口径保持锁定。

- **业务/技术口径关联规则**：一对零或一关系，每个业务口径最多对应一个技术口径，也可能暂无对应的技术口径。技术口径基于项目关联的指定版本业务口径编写，项目可选择关联某一特定版本的业务口径。核心关联条件为项目ID + 口径ID + 口径版本。

- **业务口径修改规则**：业务口径的修改主要集中在需求澄清阶段，但只要项目未上线（项目状态不为「已上线」或「已归档」），业务人员均可修改业务口径，修改后口径版本自动递增次版本号。`[修订]` 修改时系统检测并提示下游影响，若关联的技术口径已审批通过，则触发技术口径进入「待重新评估」状态。

- **技术口径审核审批规则**：技术口径需先经过组内审核、问题修复，所有问题处理完成后，方可提交开发组长审批；审批通过后，技术口径方可生效，项目方可进入验收环节；未审批通过的技术口径，无法推进项目流程。`[修订]` 审批人自动指定为开发人员所在组的组长。

- **审批委托规则**：组长可临时委托审批权限给同组指定成员，委托记录留痕；超时未审批系统自动提醒。

- **上线后问题处理规则**：不支持在已归档项目上回退或热修复，需重新创建新项目走完整流程，确保变更可追溯。

- **验收驳回处理规则**：验收不通过分为两类：（1）业务口径问题：业务修改业务口径，版本递增，开发重新开发；（2）技术实现问题：开发修改技术口径后重新提交审核审批。`[修订]` 验收驳回后，技术口径状态自动变为「开发中」，不存在「审批回退」概念，修改后的技术口径需重新走完整的审核 -> 审批流程。

- `[修订]` **项目暂停/取消规则**：进行中的项目可暂停（口径保持锁定、流程冻结）或取消（口径解锁、版本废弃）。暂停可恢复，取消不可恢复。取消需经管理员审批。详见4.8节。

- **平台网络隔离规则**：数据开发平台与数据管理平台网络隔离，数据开发平台仅可在机房访问，数据管理平台支持办公网和机房访问，两个平台之间不存在接口互通。所有涉及数据开发平台的操作（如预览数据生成、数据验收查看）均由相关人员在机房线下完成，在数据管理平台中仅记录操作状态和结果。

- **权限约束**：业务人员不可修改技术口径和开发相关内容；开发人员不可修改业务口径和要素的业务含义；开发组长仅拥有技术口径审批权限，不可直接修改技术口径内容；管理员拥有全权限，但操作需留痕。

- **数据保留规则**：所有项目、口径、业务口径、技术口径、要素、日志、审核/审批/验收记录、问题修复记录，均永久保留，不可物理删除（仅口径可标记为废弃），采用冷热分离策略管理存储。

- `[修订]` **口径状态分离规则**：口径的「生命周期状态」（草稿/生效/废弃）与「锁定状态」（空闲/已锁定）是两个正交维度，互不干扰。一个生效的口径可以同时被锁定。

- `[修订]` **口径开发类型规则**：项目关联口径时须标记开发类型（需技术开发 / 仅业务口径调整），该标记影响项目进入「待验收」的前置条件判断。

## A.1 项目状态机

```
                          +-----------+
                          | 已取消    |
                          +-----------+
                               ^
                               | 取消（需审批）
                               |
+---------+  确认澄清完成  +---------+  提交验收  +---------+  确认验收通过  +--------+  执行上线  +--------+  自动归档  +--------+
| 待澄清  | ------------> | 开发中  | --------> | 待验收  | ------------> | 待上线 | --------> | 已上线 | --------> | 已归档 |
+---------+               +---------+           +---------+               +--------+           +--------+           +--------+
     |                         ^  |                  |
     |                         |  |  验收驳回        |
     |                         +--+                  |
     |                         |                     |
     +------- 暂停 -------> +---------+  <--- 暂停 --+
                             | 已暂停  |
                             +---------+
                                  |
                             恢复（回到暂停前状态）
```

## A.2 业务口径状态机

```
+------+  提交  +--------+  确认澄清完成  +--------+
| 草稿 | -----> | 已提交 | ------------> | 已澄清 |
+------+        +--------+               +--------+
                     ^                        |
                     |   业务再次修改          |
                     +------------------------+

任意状态 ---> 已废弃（项目取消时系统自动触发）
```

## A.3 技术口径状态机

```
+--------+  开始编写  +--------+  提交审核  +--------+  提交审批  +--------+  审批通过  +----------+
| 未开发 | --------> | 开发中 | --------> | 审核中 | --------> | 待审批 | --------> | 审批通过 |
+--------+           +--------+           +--------+           +--------+           +----------+
                          ^                                        |                     |    |
                          |                           审批驳回     |                     |    | 业务口径变更
                          |                    +----------+        |                     |    v
                          +-------- 修改 ----- | 审批驳回 | <------+                     |  +------------+
                          |                    +----------+                              |  | 待重新评估 |
                          |                                                              |  +------------+
                          |                                                              |    |      |
                          |                                验收驳回（技术问题）          |    |      | 确认无需修改
                          +--------------------------------------------------------------+    |      v
                          |                                                                   | (回到审批通过)
                          +--- 修改后重新提交审核 <-------------------------------------------+

任意状态 ---> 已废弃（项目取消时系统自动触发）
```
