# AGENTS.md 工作流生命周期改造方案 v2

## 一、问题回顾与改造动机

上一版改造方案经过评审，发现以下核心痛点：

1. **阶段线性假设过强**：5 阶段被设计为严格线性流转（①→②→③→④→⑤），但现实中用户可能①→④、②→⑤、甚至①②③循环多次
2. **跨会话状态丢失**：无状态持久化机制，新会话无法恢复上下文，导致重复沟通
3. **api-merge 强制捆绑**：生成 API 和合并规格书被强制绑定，但用户可能只有 Swagger 无 PRD
4. **Task 拆解绑定阶段**：Task 拆解被绑定到特定阶段，但任何阶段都可能面对多文件场景
5. **需求变更处理粗糙**：仅 4 个回溯分支，未覆盖删除/裁剪场景和混合变更
6. **约束无分级**：所有约束平铺，token 紧张时无法优先保证核心约束
7. **产出物格式模糊**：「规范化 PRD 理解结果」等中间产物无精确定义
8. **无法弹性退出**：未明确用户可在任意阶段声明"到此为止"

---

## 二、设计原则

### 原则 1：弹性流转，不假设线性

5 个阶段之间是**图关系**，不是链关系。每个阶段独立定义前置条件和产出物，AI 根据当前项目状态 + 用户请求自动判断进入点。

### 原则 2：状态持久化，跨会话可恢复

设计轻量的 `lifecycle.md` 状态文件，记录当前阶段、已完成阶段、各阶段产出物路径、待处理事项。新会话先读取状态文件恢复上下文。

### 原则 3：条件分支，而非强制捆绑

接口合并阶段自动检测分支条件：有 PRD 走完整合并流程，无 PRD 仅生成 API 层跳过规格书合并。

### 原则 4：Task 拆解是规模控制机制，不绑定阶段

Task 拆解是通用能力，启用条件：「当前阶段涉及 ≥3 个独立文件/页面时自动启用」。任何阶段都可以使用。

### 原则 5：需求变更处理更细粒度

补充「删除/裁剪」场景和「混合变更」处理。变更影响必须生成清单并等用户确认后才执行回溯。在 feature-spec 中标注「实体→页面」依赖关系，辅助自动计算影响范围。

### 原则 6：约束分级

引入 P0/P1/P2 三级约束。P0 致命级（违反则代码不可运行），P1 规范级（违反则不一致但可运行），P2 建议级（最佳实践）。AI 在 token 紧张时优先保证 P0。

### 原则 7：每个阶段的产出物精确定义

为每个阶段定义：产出物名称、存储位置（精确到文件路径模式）、格式要求、是否为后续阶段的前置输入。

### 原则 8：容错 —— 允许用户说"够了"

明确用户可以在任意阶段声明"当前模块到此为止"。简单页面可能①+⑤就够了，内部工具页面可能不需要③改造适配。

### 3.2 阶段定义表

| 阶段 | 名称     | 触发信号                           | 前置条件           | 做什么                                      | 产出物                                                                                                  | 可跳转目标                                                                       |
| ---- | -------- | ---------------------------------- | ------------------ | ------------------------------------------- | ------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| ①    | 画 Demo  | PRD/需求 + "画页面/demo/骨架"      | 无                 | 规范化PRD → 生成demo页面+占位API            | `src/pages/{m}/*.tsx`<br>`src/api/{m}/types.ts`<br>`src/api/{m}/index.ts`<br>`[可选] specs/{f}/spec.md` | ②(接口到了)<br>⑤(小修改)<br>④(接口直接到位)                                      |
| ②    | 接口合并 | Swagger/接口文档                   | 无（可独立触发）   | 生成/更新API层<br>有PRD则合并为feature-spec | `src/api/{m}/types.ts`<br>`src/api/{m}/index.ts`<br>`[条件] specs/{f}/feature-spec.md`                  | ③(有feature-spec)<br>④(无feature-spec但接口齐)<br>②(分批接口，循环)<br>⑤(小调整) |
| ③    | 改造适配 | feature-spec就绪 + "改造/对齐规格" | ①已完成<br>②已完成 | 根据feature-spec改造已有demo页面            | 更新后的`src/pages/{m}/*.tsx`<br>更新后的`src/api/{m}/types.ts`                                         | ④(接口就绪)<br>⑤(小调整)<br>②(接口变更)                                          |
| ④    | 接口对接 | 真实接口就绪 + "对接/联调"         | API层已存在        | 占位URL→真实URL<br>删除TODO注释             | 更新后的`src/api/{m}/index.ts`<br>按需更新页面联动                                                      | ⑤(日常迭代)<br>②(接口变更)                                                       |
| ⑤    | 迭代修复 | "改一下/加字段/修复/调整"          | 无                 | 最小范围修改                                | 变更的文件                                                                                              | 任意阶段(视变更性质)                                                             |

> `{m}` = 模块标识, `{f}` = 功能标识

### 3.3 阶段判断决策树

```
用户提交请求
    │
    ├─ 是否声明"当前模块到此为止"?
    │   └─ 是 → 生成完成摘要，结束生命周期
    │
    ├─ 读取 lifecycle.md 存在?
    │   └─ 是 → 恢复上下文，从记录的阶段继续
    │
    ├─ 提供 PRD/需求 + 要求画页面?
    │   └─ 是 → ① 画 Demo
    │
    ├─ 提供 Swagger/接口文档?
    │   └─ 是 → ② 接口合并
    │
    ├─ 已有 feature-spec + 要求根据规格改造?
    │   └─ 是 → ③ 改造适配
    │
    ├─ 要求对接真实接口/联调?
    │   └─ 是 → ④ 接口对接
    │
    ├─ PRD 有更新/需求变更?
    │   └─ 是 → 需求变更回溯（见3.5）
    │
    ├─ 小修改/加字段/改bug/调整?
    │   └─ 是 → ⑤ 迭代修复
    │
    └─ 无法判断? → 向用户确认当前阶段
```

### 3.4 接口合并的多轮机制与条件分支

**条件分支检测（自动，无需用户选择）**:

```
收到 Swagger
    │
    ├─ 检查是否存在规范化 PRD（①的产出或用户提供）
    │   ├─ 存在 → 分支 A: 完整合并流程
    │   └─ 不存在 → 分支 B: 仅生成 API 层
    │
    ▼
分支 A: 完整合并
    1. 生成/更新 API 层代码
    2. 与 PRD 交叉对比 → 生成 feature-spec.md
    3. 标注差异和完成度
    4. 等待用户确认差异决策
    5. 进入 ③ 改造适配

分支 B: 仅生成 API
    1. 生成/更新 API 层代码
    2. 以 Swagger 推断生成简化 feature-spec（仅接口维度）
    3. 标注 `[无PRD-待补充]`
    4. 询问用户：直接进入 ④ 接口对接，或等待 PRD 后再走 ③
```

**多轮机制**:

```
第 N 批 Swagger 到达
    │
    ├─ 读取已有 API 文件（如有）
    ├─ 解析新 Swagger，提取新增接口
    ├─ 追加到 types.ts / index.ts
    │
    ├─ 有 feature-spec?
    │   ├─ 是 → 更新 feature-spec，补充新接口
    │   │         更新完成度统计
    │   └─ 否 → （分支B场景，不生成）
    │
    └─ 询问用户：是否继续等待更多接口？
            ├─ 是 → 保持当前阶段，更新 lifecycle.md
            └─ 否 → 进入下一阶段（③ 或 ④）
```

### 3.5 需求变更回溯机制

**细粒度决策树**:

```
PRD 变更到达
    │
    ├─ 生成变更影响清单（对比新旧 PRD）
    │   ├─ 新增页面/模块级功能？
    │   ├─ 修改数据模型/字段定义？
    │   ├─ 修改页面交互/表单字段/表格列？
    │   ├─ 删除/裁剪功能或字段？
    │   └─ 文案调整/小改动？
    │
    ├─ 向用户展示影响清单，等待确认
    │   「以下文件将受影响：... 是否继续？」
    │
    └─ 用户确认后，确定回溯目标阶段：

变更类型 → 回溯目标
─────────────────────────────────────────
新增页面/模块功能          → ① 画 Demo
修改数据模型/字段定义      → ② 接口合并（更新feature-spec）→ ③ 改造适配
修改页面交互/字段          → ③ 改造适配
删除/裁剪功能或字段        → ③ 改造适配（删除逻辑）+ ⑤ 清理代码
文案/小改动                → ⑤ 迭代修复
混合变更（多种同时）       → 取最早阶段（如①+③同时 → 走①）
```

> 确定回溯阶段后，按第五章「变更驱动的增量迭代机制」执行增量 Task。

**混合变更处理规则**:

| 变更组合     | 处理策略                             | 原因                  |
| ------------ | ------------------------------------ | --------------------- |
| ①+③ 同时出现 | 走 ①，③ 作为 ① 的子任务              | 新增页面需要完整 Demo |
| ②+③ 同时出现 | 走 ②，更新 feature-spec 后自动进入 ③ | 数据模型变更是源头    |
| ③+⑤ 同时出现 | 走 ③，⑤ 的内容作为改造的一部分       | 改造适配包含调整      |
| ①+②+③        | 走 ① → ② → ③ 顺序执行                | 全链路变更            |

**影响清单模板**:

```markdown
## 变更影响清单

| 变更项           | 类型     | 影响文件               | 影响阶段 |
| ---------------- | -------- | ---------------------- | -------- |
| 新增用户状态字段 | 数据模型 | types.ts, ListPage.tsx | ② → ③    |
| 删除批量删除功能 | 功能裁剪 | ListPage.tsx, index.ts | ③        |
| 修改搜索交互     | 页面交互 | ListPage.tsx           | ③        |

**建议回溯阶段**: ③ 改造适配
**预计涉及文件**: 3 个
**是否继续?** [Y/n]
```

### 3.6 Task 拆解通用能力

**启用条件**:

```
当前阶段涉及文件数 ≥ 3
或 用户明确要求 Task 拆解
或 涉及多个独立页面（列表+表单+详情）
─────────────────────────────────────────
是 → 启用 Task 拆解
否 → 单步执行
```

**各阶段 Task 拆解策略**:

| 阶段       | Task 粒度               | 拆解依据                     |
| ---------- | ----------------------- | ---------------------------- |
| ① 画 Demo  | 1 Task = 1 页面         | 按页面类型（列表/表单/详情） |
| ② 接口合并 | 1 Task = 1 批接口       | 按 Swagger 分批              |
| ③ 改造适配 | 1 Task = 1 页面改造     | 按 feature-spec 中的页面清单 |
| ④ 接口对接 | 通常不拆解              | 单批变更，直接执行           |
| ⑤ 迭代修复 | 1 Task = 1 个独立修改点 | 按用户描述的修改项拆分       |

**Task 拆解文件**:

- `specs/{feature}/spec.md` — Task 定义清单
- `specs/{feature}/progress.md` — 进度追踪
- `specs/{feature}/session-{n}.md` — 会话交接（跨会话时）

### 3.7 模块生命周期状态持久化

**lifecycle.md 位置**: `specs/{feature}/lifecycle.md`

**文件模板**:

```markdown
# {feature} 生命周期状态

## 基本信息

| 字段     | 值                   |
| -------- | -------------------- |
| 模块标识 | {module}             |
| 功能名称 | {name}               |
| 当前阶段 | {①/②/③/④/⑤}          |
| 状态     | {进行中/暂停/已完成} |
| 最后更新 | {YYYY-MM-DD HH:mm}   |

## 阶段完成记录

| 阶段       | 状态     | 完成时间 | 产出物路径               |
| ---------- | -------- | -------- | ------------------------ |
| ① 画 Demo  | ✅/⏳/❌ | -        | `src/pages/{m}/List.tsx` |
| ② 接口合并 | ✅/⏳/❌ | -        | `src/api/{m}/types.ts`   |
| ③ 改造适配 | ✅/⏳/❌ | -        | -                        |
| ④ 接口对接 | ✅/⏳/❌ | -        | -                        |
| ⑤ 迭代修复 | ✅/⏳/❌ | -        | -                        |

## 待处理事项

- [ ] {事项描述}

## 用户声明

- {模块已完成声明 / 无}
```

**更新时机**:

1. 阶段开始时：创建/更新 lifecycle.md，标记当前阶段
2. 阶段完成时：更新完成记录，标记产出物路径
3. 用户声明"到此为止"时：记录声明，标记状态为已完成
4. 需求变更回溯时：更新当前阶段为回溯目标

**恢复逻辑**:

```
新会话开始
    │
    ├─ 检查 specs/{feature}/lifecycle.md 存在?
    │   └─ 是 → 读取并恢复上下文
    │           「检测到 {feature} 当前处于 {阶段}，继续执行?」
    │
    └─ 不存在 → 按用户请求正常判断阶段
```

### 3.8 弹性退出机制

**用户声明方式**:

```
用户说：
- "这个模块到此为止"
- "先这样，不用继续了"
- "简单页面不用改造了"
- "当前阶段结束"
─────────────────────────────────────────
AI 响应：
1. 生成当前模块完成摘要
2. 更新 lifecycle.md 标记为已完成
3. 列出已产出文件清单
4. 告知用户如需继续，可随时恢复
```

**完成摘要模板**:

```markdown
## {feature} 开发完成

**完成阶段**: ① → ② → ⑤（用户声明到此为止）

**产出文件**:

- `src/pages/{m}/List.tsx` — 列表页
- `src/api/{m}/types.ts` — 类型定义
- `src/api/{m}/index.ts` — API 方法

**待办事项**（如有）:

- 表单页尚未实现（如需要，从 ① 开始）

**恢复方式**: 提供 PRD/Swagger，我将自动检测并继续
```

## 四、阶段间数据流

```
① 画 Demo
   输入: PRD/需求文档（任意格式）
   输出:
     - src/pages/{m}/*.tsx（demo页面，含TODO标记）
     - src/api/{m}/types.ts（临时类型，TODO标注）
     - src/api/{m}/index.ts（占位API，TODO URL）
     - [可选] specs/{f}/spec.md + progress.md（多页面时）
     - specs/{f}/lifecycle.md（创建）
          ↓
② 接口合并（可多轮）
   输入: Swagger/接口文档 + ①的产出
   输出:
     - src/api/{m}/types.ts（真实类型，逐轮追加）
     - src/api/{m}/index.ts（真实方法签名，逐轮追加）
     - [条件] specs/{f}/feature-spec.md（有PRD时）
     - specs/{f}/lifecycle.md（更新）
          ↓
③ 改造适配
   输入: feature-spec.md + 已有demo页面 + 已有API类型
   输出:
     - 更新后的 src/pages/{m}/*.tsx（对齐feature-spec）
     - 更新后的 src/api/{m}/types.ts（如feature-spec有字段调整）
     - specs/{f}/lifecycle.md（更新）
          ↓
④ 接口对接
   输入: 真实接口地址 + 已有代码
   输出:
     - 更新后的 src/api/{m}/index.ts（真实URL）
     - 删除TODO注释
     - 按需更新页面联动
     - specs/{f}/lifecycle.md（更新）
          ↓
⑤ 迭代修复（贯穿生命周期）
   输入: 用户描述的具体修改
   输出: 最小范围修改
   流向: 根据变更性质，可能回到任意阶段
```

## 五、变更驱动的增量迭代机制

### 5.1 核心理念：变更 = diff，不是重做

任何变更（需求变更、接口变更）的处理本质是 diff 操作：

```
旧状态（已有代码 + 已有 feature-spec）
    ↕ diff
新状态（新 PRD / 新 Swagger / 用户描述的变更）
    ↓
变更集（delta）= 新状态 - 旧状态
    ↓
增量 Task 列表
    ↓
逐 Task 执行
```

### 5.2 两种变更源，统一处理流程

| 变更源   | 触发信号                 | diff 对象                             |
| -------- | ------------------------ | ------------------------------------- |
| 需求变更 | PRD有更新/提供新需求描述 | 新PRD vs 已有 feature-spec            |
| 接口变更 | 接口改了/提供新Swagger   | 新Swagger vs 已有 types.ts + index.ts |

**统一处理流程**:

```
变更到达
  │
  ├─ 1. 读取已有状态
  │     ├─ feature-spec.md（功能维度）
  │     ├─ types.ts + index.ts（接口维度）
  │     └─ 页面代码（实现维度）
  │
  ├─ 2. 生成变更 diff
  │     ├─ 需求变更 → 对比新旧 PRD / feature-spec
  │     └─ 接口变更 → 对比新旧 Swagger / types.ts
  │
  ├─ 3. 输出「变更影响报告」
  │     ├─ 变更项清单（增/改/删）
  │     ├─ 受影响文件清单
  │     ├─ 建议的增量 Task 列表
  │     └─ **等待用户确认**
  │
  ├─ 4. 用户确认后
  │     ├─ 更新 feature-spec.md（打变更标记）
  │     ├─ 生成增量 Task → 写入 spec.md / progress.md
  │     └─ 逐 Task 执行
  │
  └─ 5. 执行完毕
        └─ 更新 lifecycle.md
```

### 5.3 feature-spec 作为活文档的版本管理

feature-spec 不是一次性产物，是贯穿模块生命周期的单一事实来源。每次变更在上面做标记而非重写。

**字段级版本标记示例**:

```markdown
### User 实体
```

### 5.4 增量 Task 生成机制

从 diff 自动推导 Task 列表的决策树：

```
变更 diff
  │
  ├─ 类型/接口层面的变更?
  │   └─ Task: 更新 types.ts（新增/修改/删除字段）
  │   └─ Task: 更新 index.ts（新增/修改/删除方法）
  │
  ├─ 页面层面的变更?
  │   ├─ 新增页面? → Task: 走阶段①的页面生成流程
  │   ├─ 修改已有页面?
  │   │   ├─ 表格列变更? → Task: 更新 columns 定义
  │   │   ├─ 搜索条件变更? → Task: 更新 searchFields
  │   │   ├─ 表单字段变更? → Task: 更新 formFields
  │   │   └─ 交互逻辑变更? → Task: 更新事件处理
  │   └─ 删除页面? → Task: 删除文件 + 清理路由
  │
  └─ 业务规则变更?
      └─ Task: 更新对应的校验/联动逻辑
```

每个 Task 是最小可执行单元。

```markdown
| Task | 类型   | 状态 | 说明                     |
```

### 5.5 老功能迭代 vs 新功能 —— 统一入口

通过 lifecycle.md 存在性自动判断：

```
用户请求到达
  │
  ├─ 涉及的模块是否已有 lifecycle.md?
  │   │
  │   ├─ 有（老功能迭代）
  │   │   ├─ 读取 lifecycle.md 恢复上下文
  │   │   ├─ 读取 feature-spec.md 了解当前状态
  │   │   ├─ 对比变更 → 生成 diff
  │   │   ├─ 生成增量 Task
  │   │   └─ 执行（走变更流程）
  │   │
  │   └─ 无（新功能）
  │       ├─ 按阶段判断决策树正常路由
  │       ├─ 创建 lifecycle.md
  │       └─ 执行（走正常生命周期）
```

**关键点**：AI 不需要用户告知"这是老功能还是新功能"，通过检查 lifecycle.md 自动判断。

## 六、约束分级建议

### P0 致命级（违反则代码不可运行）

| 约束                           | 违反后果                 | 检测方式 |
| ------------------------------ | ------------------------ | -------- |
| 禁止 `import axios`            | 请求拦截器失效，认证丢失 | ESLint   |
| 禁止 `any` 类型                | 类型安全丧失             | tsc      |
| API 方法必须 `ByHTTP` 后缀     | 命名不一致，维护困难     | 人工检查 |
| 响应类型必须匹配 `PageData<T>` | 分页组件无法工作         | tsc      |
| 禁止 `../../` 相对路径         | 重构时路径失效           | ESLint   |

### P1 规范级（违反则不一致但可运行）

| 约束                       | 违反后果       | 优先级策略      |
| -------------------------- | -------------- | --------------- |
| 未使用参数加 `_` 前缀      | 代码风格不一致 | token充足时修复 |
| 使用 sdesign 组件替代 antd | UI 风格不一致  | token充足时修复 |
| 类型导入使用 `import type` | 编译效率降低   | 自动修复        |
| 路径使用 `@/` 别名         | 可读性降低     | 自动修复        |
| API 对象命名 `{module}Api` | 命名不一致     | 人工检查        |

### P2 建议级（最佳实践）

| 约束                                    | 说明     |
| --------------------------------------- | -------- |
| Modal 使用条件渲染 `{open && <Modal/>}` | 性能优化 |
| SForm 字段联动使用 `type: 'dependency'` | 代码简洁 |
| 复杂逻辑提取为 hooks                    | 可维护性 |
| 组件 props 使用 interface 定义          | 可读性   |

**Token 紧张时的处理策略**:

```
剩余 token < 20%
    │
    ├─ 优先保证 P0 约束全部满足
    ├─ P1 约束仅修复自动可修复的（eslint --fix）
    └─ P2 约束跳过，代码注释标记 `[P2-待优化]`

剩余 token < 10%
    │
    ├─ 仅保证 P0 约束
    └─ 向用户报告：「token 紧张，部分约束未完全检查」
```

## 十、风险与应对

| 风险                              | 影响                                       | 概率 | 应对措施                                                                            |
| --------------------------------- | ------------------------------------------ | ---- | ----------------------------------------------------------------------------------- |
| **阶段判断逻辑复杂，AI 误判**     | 进入错误阶段，用户困惑                     | 中   | 1. 决策树增加"无法判断→向用户确认"分支<br>2. 每个阶段入口增加状态检测，不匹配时提示 |
| **lifecycle.md 与实际代码不同步** | 恢复错误上下文                             | 中   | 1. 阶段完成时强制更新 lifecycle.md<br>2. 恢复时对比文件实际存在性，缺失时标记异常   |
| **条件分支检测错误**              | 有 PRD 却走分支B，或反之                   | 低   | 1. 明确检测逻辑：检查 specs/{f}/ 下是否有 PRD 相关文件<br>2. 分支选择前向用户确认   |
| **Task 拆解阈值设置不当**         | 文件数=2 时未拆解导致混乱，或=3 时过度拆解 | 低   | 1. 阈值设为 ≥3，留有缓冲<br>2. 用户可强制要求/禁止拆解                              |
| **需求变更影响范围计算错误**      | 遗漏受影响文件，或过度回溯                 | 中   | 1. feature-spec 中标注实体→页面依赖关系<br>2. 影响清单必须用户确认后才执行          |
