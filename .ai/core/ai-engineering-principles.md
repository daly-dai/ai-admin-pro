# AI 工程化指导思想

---

## 原则一：决策预编译

**不让 AI 在运行时做"关于如何工作"的决策。所有编排决策在设计时完成。**

5 个决策点 × 80% 正确率 = 32% 最终正确率。不是 AI 不行，是决策链太长。

| 应用                               | 示例                                         |
| ---------------------------------- | -------------------------------------------- |
| 能用路由表匹配的，不让 AI 推理     | 关键词 → 场景，不走"分析代码状态 → 判断阶段" |
| 能用步骤表固化的，不让 AI 自己编排 | Skill 固定步骤表，第 N 步做什么写得清清楚楚  |
| 能用模板填空的，不让 AI 自由生成   | 填空模板替代 JSON 生成器                     |

**判断标准**：新增一个能力时，问"AI 需要做几个决策才能完成？"——超过 2 个就值得预编译。

---

## 原则二：错题集驱动

**"不要做什么"比"要做什么"有效一个数量级。知识资产从翻车中提炼，不提前设计。**

21 行错题集 > 40,000 行全量知识库。正面知识无边界，负面知识有边界——AI 在这个项目里频繁犯的错就那十几类。

| 应用                                | 示例                                 |
| ----------------------------------- | ------------------------------------ |
| 新增约束的唯一时机：AI 实际翻车之后 | P001-P015 每条都有对应的真实翻车记录 |
| 超过 3 个月未触发的条目降级         | 从"硬约束"移入"参考"                 |
| 不预判 AI 可能会犯什么错            | "我觉得它可能会这样"→ 不加           |

**判断标准**：要加一条新规则时，问"这是 AI 实际犯过的错，还是我觉得它可能会犯的错？"——后者不加。

---

## 原则三：复杂度外移

**系统的复杂度守恒。你能选择的是放在 AI 侧还是工具侧。永远选工具侧。**

| 失败（复杂度在 AI 侧）                                  | 成功（复杂度在工具侧）                                |
| ------------------------------------------------------- | ----------------------------------------------------- |
| JSON 生成器：AI 理解 300 行类型定义 + 正确填写嵌套 JSON | 填空模板：人写模板多花 1 小时，AI 每次生成少犯 3 个错 |
| 让 AI 管理上下文                                        | 文件持久化 + 命令恢复                                 |
| 让 AI 判断自主权                                        | 用户显式声明模式                                      |
| 让 AI 判断处于哪个阶段                                  | 关键词路由表                                          |

**判断标准**：AI 在某个环节频繁翻车时，先问"能不能把这里的复杂度从我这侧解决掉（改模板/加验证/改约束）"。

---

## 原则四：约束优于自由

**AI 在小模块、紧约束、明确边界内工作最好。自由度越大，产出质量方差越大。**

| 应用                                | 示例                                                         |
| ----------------------------------- | ------------------------------------------------------------ |
| 能用"禁止 X"表达的，不用"推荐 Y"    | "禁止 import axios" > "推荐用 createRequest"                 |
| 能限制文件范围的，不给全项目权限    | 输出锁：仅允许写 `src/api/{module}/` + `src/pages/{module}/` |
| 能指定组件和 API 的，不让 AI 自己选 | 组件替换表：antd Table → SSearchTable                        |

**判断标准**：审查一条指令时，问"这条是缩小了 AI 的选择空间还是扩大了？"——缩小的是约束，扩大的是风险。

---

## 原则五：信息密度优先

**给 AI 的上下文，每段话必须有"不读就会犯错"的信息。正确废话是噪音。**

| 删掉                                        | 保留                                                |
| ------------------------------------------- | --------------------------------------------------- |
| AI 能从代码里自己读到的（技术栈、目录结构） | 这个项目特有的约束（"用 createRequest 不用 axios"） |
| 通用的最佳实践（"组件应该单一职责"）        | 这个项目的例外（"login 目录不受组件替换规则限制"）  |
| 正确废话（"系统采用前后端分离架构"）        | 翻车记录（"P001: 禁止手动管理 Modal open 状态"）    |

**判断标准**：每一段话，问"如果删掉这段，AI 会犯什么错？"——回答不上来就删。

---

## 原则六：能力分层适配

**不给弱模型选择权，不让强模型被约束拖慢。两条路线的差异不在知识内容，在决策自主权。**

|          | 强模型路线                | 弱模型路线                  |
| -------- | ------------------------- | --------------------------- |
| 核心约束 | 共享                      | 共享                        |
| 执行方式 | "先生成 API 层再生成页面" | "步骤 1: xxx → 步骤 2: xxx" |
| 模板     | 按需读取                  | bundle 内联，一次加载       |
| 决策模式 | 推理 → 决策 → 执行        | 预编译决策 → 机械执行       |

**判断标准**：新增一条约束时，问"这条对强模型和弱模型是一样的吗？"——一样放共享层，不一样按路线分层。

---

## 原则关系

```
  决策预编译 ←──→ 复杂度外移          （怎么设计）
  约束优于自由    信息密度优先
  错题集驱动 ←──→ 能力分层适配        （怎么迭代）
```

- **上面四条**回答"怎么设计"：预编译决策、外移复杂度、约束缩小选择空间、信息密度决定指令质量
- **下面两条**回答"怎么迭代"：等翻车再加规则、不同模型给不同自由度

---

## 为什么这些原则不变

模型会从 7B 升到 13B 到 70B，推理能力会变。但这六条不变，因为它们建立在一个更底层的事实上：

> **AI 和人类开发者之间永远存在信息不对称。AI 不知道你的项目历史、不知道哪些坑踩过、不知道当前版本装了什么包、不知道后端的约定。**

只要这个不对称存在，你就需要约束。约束的具体内容会变（组件库大版本 → 改替换规则），但"用约束而非自由"这个原则不变。
