# AI 工程体系改造方案

> 基于制造业百年沉淀（APQP / 精益 / 六西格玛 / TOC / FMEA）、嵌入式安全设计范式、ABS 控制哲学，与当前 ai-admin-pro 体系的相互印证，提出增量式改造方案。
>
> **原则：不推翻现有体系。每个改动点附带理论来源和验收标准。**

---

## 一、现状评估

### 1.1 已映射的制造思想

| 当前体系             | 制造理论来源                                         |
| -------------------- | ---------------------------------------------------- |
| ⑨ 人是最终责任者     | 丰田"尊重人" + Jidoka——机器检异常即停，人做判断      |
| ① 复杂度不灭         | TOC——复杂度不会消失，只会在系统内转移                |
| ③ 边界是宪法         | 单元化生产 + 控制计划——单元有明确 I/O 边界           |
| ② 为修改而设计       | SMED（快速换模）——让变化变便宜                       |
| ④ 假设一切会炸       | FMEA（失效模式与影响分析）——制造业投入最大的预防活动 |
| ⑦ 能观测才存在       | SPC + Andon——控制图监控，异常全线可见                |
| ⑧ 技术债是债务       | TPM（全员生产维护）——预防成本 < 故障成本             |
| ⑤ 先跑通再跑对再跑快 | APQP 五阶段——跳过阶段是制造第一大忌                  |
| ⑥ 代码是给人读的     | 可视化管理——任何人 5 秒看明白工作站状态              |
| 决策预编译           | SOP（标准作业）——专家判断固化为文件                  |
| 错题集驱动           | 8D + FMEA 反馈闭环——出问题才分析                     |
| 复杂度外移           | Poka-Yoke（防错）——用工具保证正确                    |
| 约束优于自由         | Poka-Yoke——物理阻止 > 标语提醒                       |
| 信息密度优先         | 一页纸标准作业——删掉这句人会犯什么错                 |
| 能力分层适配         | TOC 差异化管理 + APQP 阶段门控                       |

### 1.2 结构性缺口

| 缺口               | 制造参照                         | 具体表现                                                                     |
| ------------------ | -------------------------------- | ---------------------------------------------------------------------------- |
| 量化质量指标层缺失 | 六西格玛：CPK / SPC / 缺陷率目标 | 有 Poka-Yoke（防格式错），无 Eval Suite。无法回答"这个 prompt 的 CPK 是多少" |
| 风险分级思维缺失   | TOC：只在瓶颈处用力              | ①+④ 放一起 → "每个环节都防御"。缺少"哪些炸了可以接受"的清单                  |
| 阶段流程框架缺失   | APQP 五阶段 + PPAP 18 项提交     | 有执行流程（读→匹配→拆→执行），缺质量策划流程（概念→可行→设计→验证→放行）    |
| 输入质量无验证     | 嵌入式金丝雀 + "爸爸"实践        | 要求 AI 读文件，但无机制验证 AI 是否真的读了                                 |
| 故障响应无预设     | 嵌入式 Fail-Safe / Fail-Silent   | AI 不确定时行为未定义，可能乱说或沉默                                        |

## 二、指导思想补全：三条新增原则

### 新增原则⑩：Eval 是上线通行证

| 属性     | 内容                                                  |
| -------- | ----------------------------------------------------- |
| 原则     | 没有 Eval Suite 的 prompt，跟没有量具的生产线一样危险 |
| 判断标准 | 改 prompt 后，能拿出数据证明变好了吗                  |
| 优先级   | ⑨ > ① > ③ > ② > ④ > **⑩** > ⑦ > ⑧ > ⑤ > ⑥             |

**理论来源一 — SPC 统计过程控制（Shewhart, 1931）**

休哈特的发现：过程输出的波动分两种——正常波动（Common Cause）和异常波动（Special Cause）。区分二者的唯一方法是持续测量并建立统计基线。

```
SPC → AI 翻译：
  建立基线（μ ± 3σ）  →  对核心模块建 Eval Suite，跑出 baseline 分数
  持续监控            →  每次 prompt 变更后重跑 Eval Suite
  超控制限 = 异常     →  分数跌破红线 → 拒绝上线，强制回溯
```

**理论来源二 — CPK 过程能力指数**

| CPK 概念     | 制造含义       | AI 系统对应                     |
| ------------ | -------------- | ------------------------------- |
| 规格下限 LSL | 可接受的最小值 | 最低可接受 Eval 分数            |
| 过程均值 μ   | 输出平均值     | Eval Suite 平均得分             |
| 过程标准差 σ | 输出波动       | 同一 prompt 跑 N 次的分数标准差 |
| CPK ≥ 1.33   | 过程能力合格   | Eval 均分 > 红线 + 方差稳定     |
| CPK < 1.00   | 过程能力不足   | 分数波动大或接近红线 → 不可上线 |

**理论来源三 — POST 上电自检（嵌入式）**

嵌入式系统上电后先自检 CPU/RAM/ROM/IO——全过才进主循环。新 prompt 上线前必须跑全量 Eval Suite——不达标不运行。

**验收标准**：

- [ ] `.ai/evals/` 目录已创建，至少有一个核心模块的 Layer 1 Eval（断言检查脚本）
- [ ] 每次 prompt 变更后有 Eval 分数记录（`history.csv`）
- [ ] Eval 分数跌破预设红线时，prompt 变更被拒绝

---

### 新增原则⑪：在瓶颈处用力

| 属性     | 内容                                                       |
| -------- | ---------------------------------------------------------- |
| 原则     | 不是所有"假设会炸"的环节都要防御。只在用户最痛的那一点死磕 |
| 判断标准 | 当前整个 AI 管道，用户投诉或输出失败集中在哪一个环节       |
| 优先级   | ⑨ > ① > ③ > ② > ④ > ⑩ > **⑪** > ⑦ > ⑧ > ⑤ > ⑥              |

**理论来源 — TOC 约束理论（Goldratt, 1984）**

任何系统的产出由单一瓶颈决定。在非瓶颈环节优化，对全局产出零贡献。

```
TOC 五步法 → AI 翻译：
  识别瓶颈  →  用户投诉/Bad Case 集中在哪个环节？
  挖尽瓶颈  →  不先上复杂方案，先用简单 prompt 调优
  迁就瓶颈  →  瓶颈未解决前，用 Guardrail + Fallback 兜底
  打破瓶颈  →  前三步不够才用重武器（微调/换模型/RAG）
  找新瓶颈  →  永无止境
```

**防御分级表**：

| 等级      | 标准                   | 防御策略                  | 资源投入 |
| --------- | ---------------------- | ------------------------- | -------- |
| 🔴 阻断   | 用户无法完成任务       | 全自动闸门拦截            | 高       |
| 🟠 降级   | 能完成但体验差         | Guardrail + 告警 + 人兜底 | 中       |
| 🟡 瑕疵   | 不影响但不够好         | 攒够 N 例再修             | 低       |
| 🟢 可接受 | 理论上不完美，极少触发 | 不防。触发时追加错题集    | 零       |

**验收标准**：

- [ ] 防御分级表已在 AGENTS.md 或 task-gates.md 中体现
- [ ] 每个新闸门/新约束的引入，附带"是否值得"的判断（用户影响 × 触发频率）

---

### 新增原则⑫：反馈必须闭环

| 属性     | 内容                                                            |
| -------- | --------------------------------------------------------------- |
| 原则     | 每个 Bad Case 必须有归宿。分析→改进→验证→固化，四步缺一是假把式 |
| 判断标准 | 最近一个 Bad Case，从发现到规则更新，链路完整吗                 |
| 优先级   | ⑨ > ① > ③ > ② > ④ > ⑩ > ⑪ > ⑦ > **⑫** > ⑧ > ⑤ > ⑥               |

**理论来源一 — 8D 报告（福特, 1987）**

汽车行业强制客诉处理流程的八个步骤：

| D   | 名称     | AI 编程对应              | 现状      |
| --- | -------- | ------------------------ | --------- |
| D1  | 组建团队 | —                        | —         |
| D2  | 描述问题 | Bad Case 触发条件        | ✅ 错题集 |
| D3  | 围堵措施 | 紧急改 prompt 止血       | ❌ 缺失   |
| D4  | 根因分析 | prompt/RAG/模型/输入解析 | ❌ 缺失   |
| D5  | 永久对策 | 改进方案                 | ⚠️ 部分   |
| D6  | 验证有效 | 同 case + 变体确认不复现 | ❌ 缺失   |
| D7  | 预防再发 | 反灌错题集               | ✅ 错题集 |
| D8  | 团队总结 | Bad Case 复盘            | ❌ 缺失   |

**理论来源二 — PDCA 的 C→A 环节（Deming, 1950s）**

最常断裂的不是发现问题（Check），而是把发现的问题落实成改进（Act）。⑫ 的存在就是为了锁死 C→A 这条链。

**验收标准**：

- [ ] Bad Case 处理模板包含 D3-D6-D8 的步骤
- [ ] 每 10 个 Bad Case 抽 1 个做完整链路审计：从发现到规则更新，中间步骤齐全吗

## 三、AI 架构升级

---

### 改造项 1：生成后自动验证闭环（来自 ABS "检测早于灾难"）

**当前**：AI 生成 → 人手动跑 `pnpm verify` → 发现问题 → 改

**问题**：检测发生在人介入之后。人的注意力是最贵的传感器。

**改为**：

```
生成  →  [自动跑 pnpm verify]  →  报错自动喂给 AI
         ↓                            ↓
    秒级闭环                     AI 定向修正报错行
                                      ↓
                                 再自动跑 verify
                                      ↓
                                 全过 → 提交人终审
                                 修 2 轮仍不过 → 停（已有 R1）
```

**理论来源一 — ABS 闭环控制（Bosch, 1978）**

ABS 的传感器→ECU→液压调节器在毫秒级完成闭环，中间没有人类脚踩刹车的延迟。eslint/tsc 的执行速度是秒级——足够做自动闭环。

**理论来源二 — Jidoka 自働化（丰田）**

机器检测异常 → 自动停机 → 人只在停机后介入判断。而不是人盯着机器等异常。

**理论来源三 — 嵌入式看门狗（Watchdog）**

硬件定时器超时没喂 → 自动复位。对应：生成超时/verify 超 N 轮 → 自动终止，交人。

**验收标准**：

- [ ] 每个 Task 生成完成后，`pnpm verify` 自动执行（不需人手动触发）
- [ ] verify 报错自动定位到文件+行号，AI 只修报错行所在的逻辑块（不全盘重来）
- [ ] 自动修正最多 2 轮，超过则停止，交人判断（已有 R1 的扩展）
- [ ] 人始终有最终否决权——不设"自动放行"模式

### 改造项 2：闸门优先级分级（来自嵌入式"中断优先级"）

**当前**：G1-G15 闸门平权，所有闸门都是"生成后验证时查"

**问题**：有些问题（敏感信息、格式崩溃）一秒不能等。有些（命名风格）可以上线再改。

**改为**：G1-G15 标注 P0/P1/P2：

```
P0 — 生成即刻拦截（等不到 verify 阶段）
  - 输出格式 Schema 校验（每个文件生成完立刻跑）
  - 如果有安全/敏感信息检测，立刻跑

P1 — 生成完成后自动 verify 必查
  - G1 零 any / G2 零 axios / G3 import type / G5 API命名
  - G6 未用参数 / G11 pnpm verify / G13 圈复杂度
  - G14 嵌套深度 / G15 函数参数

P2 — 人工 review 时抽查
  - G4 跨模块导入用别名 / G7 无 SConfirm / G8 弹层工厂函数
  - G9 无 loading prop / G10 字典编码 / G12 更新状态+写复盘
```

**理论来源一 — 嵌入式中断优先级 + NMI 不可屏蔽中断**

CPU 不管在跑什么，NMI 来了立刻响应。P0 闸门就是 AI 系统的 NMI——不管生成到哪一步，立刻拦截。

**理论来源二 — Andon 安灯系统（丰田）**

操作工发现异常拉绳——所有人都停。P0 的语义就是"拉绳时刻"。

**验收标准**：

- [ ] task-gates.md 的 G1-G15 表中新增"优先级"列，每行标注 P0/P1/P2
- [ ] P0 闸门的检测已自动化（不需要人触发）
- [ ] P2 闸门有明确的抽查规则（频率、谁查）

### 改造项 3：降级策略预设（来自嵌入式 Fail-Safe / Fail-Silent）

**当前**：AI 不确定时，行为未定义——可能乱说（Fail-Arbitrary），可能沉默。

**改为**：在 AGENTS.md 新增 §五 "降级策略"，预设每种故障的默认行为。

| 情况                      | 默认行为                         | 嵌入式对应       |
| ------------------------- | -------------------------------- | ---------------- |
| 输出格式不符合 Schema     | 静默重试 1 次 → 仍失败则标注失败 | Fail-Silent      |
| 不确定答案（概率低/矛盾） | 明确说"不确定" + 给出已知信息    | Fail-Safe        |
| 主流程超时/模型不可用     | 切备选策略（弱模型/规则兜底）    | Fail-Operational |
| Token 即将耗尽            | 优先完成当前逻辑块，标记未完成   | Fail-Safe        |
| 检测到安全红线            | 立即停止生成，标记 + 上报        | NMI 不可屏蔽中断 |

禁止两件事：

- ❌ 不确定时编造（Fail-Arbitrary）
- ❌ 不确定时沉默（用户不知道哪里出了问题）

**理论来源 — 嵌入式四种故障模式**

| 模式             | 含义                  | AI 是否可用   |
| ---------------- | --------------------- | ------------- |
| Fail-Safe        | 出故障 → 切到安全状态 | ✅ 默认策略   |
| Fail-Silent      | 出故障 → 安静停止     | ✅ 格式错误时 |
| Fail-Operational | 出故障 → 降级继续     | ✅ 性能压力时 |
| Fail-Arbitrary   | 出故障 → 输出不可预测 | ❌ 禁止       |

**验收标准**：

- [ ] AGENTS.md 已新增 §五 降级策略
- [ ] 降级策略表每季度审视一次（和错题集同步）
- [ ] 任意 AI 输出路径都能回答"这一环出问题时，默认行为是什么"

### 改造项 4：金丝雀哨兵体系（来自嵌入式 Canary + "爸爸"实践）

**当前**：AGENTS.md §零 要求 AI 按顺序读文件。但无机制验证 AI 真的读了。

**问题**：如果 AI 跳过了 §零 直接生成，只能在验证代码质量时发现——成本太高。

**改为**：在关键指令文件里埋金丝雀哨兵，用自动化检测哨兵是否出现。

**哨兵 A — AGENTS.md 执行确认**

在 AGENTS.md §零 顶部加：

```markdown
> ⚠️ CANARY：读完本条后，本轮第一个生成的 .ts/.tsx 文件顶部注释必须写：
> `/* @ctx AGENTS.md checklist read */`
> 如果生成的代码里没有这行 → 本条未被读取，所有后续生成作废。
```

检测方式：`grep -r "@ctx AGENTS.md checklist read" src/`，没找到 → 整批作废，AI 重读重生成。

**哨兵 B — Task 生成检查点**

在 AGENTS.md §三 的"填空生成"步骤里加：

```markdown
> ⚠️ CANARY：本 Task 生成的所有 .ts/.tsx 文件顶部注释必须标注：
> `/* @task {task-id} checkpoint */`
```

**哨兵 C — 错题集确认**

在 `.ai/pitfalls/index.md` 顶部加：

```markdown
> ⚠️ CANARY：如果你参考了任一错题，在代码对应位置加注释：
> `/* @pitfall P00X avoided */`
```

**哨兵设计三原则**（从"爸爸"实践提炼）：

1. **哨兵必须是唯一字符串**——正常代码里永远不可能出现。`@ctx AGENTS.md checklist read` 不可能被 AI 随机写出。
2. **哨兵检测的是"AI 如果不执行指令就绝对做不出来的事"**——不读 AGENTS.md 就不可能在文件顶部写 `@ctx`。
3. **哨兵不过 = 内容不看**——不抱着"万一内容还行"的侥幸，直接触发重试。

**理论来源一 — 嵌入式金丝雀值（Stack Canary）**

```
[栈底] [局部变量] [金丝雀值 0xDEADBEEF] [返回地址] [栈顶]
                         │
                  缓冲区溢出先覆盖金丝雀
                  金丝雀值变了 → 栈被破坏 → 立即停机
```

金丝雀的唯一使命就是"死"——它不参与业务逻辑。哨兵 A/B/C 的唯一使命是"没出现就说明 AI 没读指令"。

**理论来源二 — "爸爸"实践（用户社区, 2025）**

```markdown
规约要求每次回复以"爸爸"开头。
只要看见不是以"爸爸"开头 → AI 开始胡编了。
```

这个实践揭示的洞见：**检测上下文完整性比检测输出质量更便宜、更早发现问题**。哨兵检测的是前导指标（AI 还记不记得规矩），而非滞后指标（输出好不好）。

**理论来源三 — 通止规（Go/No-Go Gauge）**

制造业最便宜的检验工具——不测尺寸读数值，只测"过/不过"。`grep` 一行就是 AI 系统的通止规。

**验收标准**：

- [ ] AGENTS.md §零 顶部已加入哨兵 A
- [ ] AGENTS.md §三 生成流程已加入哨兵 B
- [ ] `.ai/pitfalls/index.md` 顶部已加入哨兵 C
- [ ] pnpm verify 或 verify-wrapper.ts 已加入哨兵 A 的 grep 检查
- [ ] 哨兵注释在 verify 通过后由工具自动清除（不留技术债）
- [ ] 哨兵不过时，AI 自动触发重读文件 + 重新生成，不交给人判断

### 改造项 5：前置检查点——生成前先过三道门（来自通止规 + 嵌入式 POST）

**当前**：AI 开始生成之后才在过程中发现缺信息、输出锁不明确——边生成边补，验证成本高。

**改为**：在拆 Task → 逐 Task 执行之间，加一个显性前置检查。

```
拆 Task  →  [前置检查点]  →  逐 Task 执行
                │
                ├─ Q1: 模板/输入参数完整否？（缺字段 → 不生成，先补全）
                ├─ Q2: 输出锁明确否？（不明确 → 不生成，先锁定范围）
                └─ Q3: 金丝雀哨兵配齐否？（没配 → 不生成）
```

**理论来源 — 嵌入式 POST 上电自检**

系统启动前先检查 CPU/RAM/ROM/IO——全过才进主循环。前置检查 = AI 生成前先确认所有输入条件满足。

**理论来源 — 通止规使用时机：检夹具，不检零件**

通止规不用等零件加工完才去量——先量夹具和刀具对不对。夹具不对，后面加工全是废品。前置检查点 = 检夹具。

**验收标准**：

- [ ] AGENTS.md §三 生成流程中，"逐 Task 执行"之前有显性的"前置检查"步骤
- [ ] 前置检查的三个条件（模板完整、输出锁明确、哨兵配齐）写成硬规则（不是"建议"）
- [ ] 任一条件不满足 → AI 不进入生成，先补齐条件

### 改造项 6：非标 Lane → 三层半结构化工坊

**当前**：非标 Lane 匹配不上 → 写 PRD → 自由生成。PRD 用完就作废，验证成本全由人扛。

**改为**：非标 Lane 从"纯兜底"升级为三层结构：

```
Layer 1 — 所有非标强制约束
  ├─ 输出 Schema：至少定义文件结构 + 命名规范
  ├─ 核心闸门子集：G1-G6, G11, G13-G15（去掉 CRUD 专用 G7-G10）
  └─ 生成前扫一遍 pitfalls 索引

Layer 2 — 半标准模式库
  ├─ 同一模式在非标 Lane 出现 ≥3 次 → 升级为半标准 checklist
  ├─ 存放位置：.ai/nonstandard/{pattern-name}.md
  └─ 形式：~5 条 checklist，不建完整 Lane

Layer 3 — 错题集自动匹配（已有，非标 Lane 强制走）
```

**理论来源 — 单元工坊（Cellular Job-Shop）**

制造业处理"低频但不罕见"的产品——不建全自动线，但每个产品家族有一个半结构化的模子 + 核心 checkpoints。单元内自治，但不做全厂通用约束。

**验收标准**：

- [ ] `.ai/nonstandard/` 目录已创建，含 README
- [ ] `.ai/conventions/lanes.md` 非标 Lane 部分改为三层结构描述
- [ ] 历史非标 case 已扫描，≥3 次的模式已提炼为半标准 checklist（≥1 个）
- [ ] 非标 Lane 生成时强制走一遍 pitfalls 自动匹配

---

### 改造项 7：实验模式（原型车间）

**当前**：没有显性实验模式。所有生成都受 Lane 和闸门约束。快速验证时约束反而拖慢迭代。

**改为**：新增一条显性的"实验模式"路由：

```
用户意图含"原型/实验/试试/快速验证/PoC"时：
  1. AI 声明进入实验模式
  2. 豁免所有 Lane 约束、模板、Task 闸门（仅保留 G11 pnpm verify）
  3. 用户可自由迭代，不记入错题集
  4. 硬门：实验模式代码不能直接进主干
     → 结束弹"结果判定"：
       a. 可行 → 建议提炼为半标准/完整 Lane
       b. 不可行 → 废弃
       c. 不确定 → 保留草案
```

**理论来源 — Skunkworks（洛克希德原型团队, 1943）**

脱离量产产线全部约束的独立实验空间。出来的东西不能直接进产线——通了才建线，不通就废弃，损失可控。

**验收标准**：

- [ ] `.ai/conventions/experiment-mode.md` 已创建
- [ ] 路由策略中已加入实验模式分支（明确触发词）
- [ ] 实验模式代码有硬隔离（不能直接进主干）

---

### 改造项 8：Lane 生命周期管理

**当前**：Lane 是静态的，加 Lane 靠人工判断。

**改为**：Lane 有上有下，自动进化：

```
实验模式            →  验证可行 → 非标 Lane
非标 Lane           →  同一模式 ≥3 次 → 半标准 (.ai/nonstandard/)
半标准              →  同一模式 ≥10 次 → 完整 Lane
完整 Lane           →  6 个月无人匹配 → 降级为半标准
半标准              →  3 个月无人 → 归档
```

**决策矩阵**：

| 月均需求量 | 重复度 | 策略               | 存放位置         |
| ---------- | ------ | ------------------ | ---------------- |
| ≥10        | 高     | 完整 Lane          | lanes.md         |
| ≥3         | 高     | 半标准 checklist   | .ai/nonstandard/ |
| ≥3         | 低     | 非标，每例单独处理 | —                |
| <3         | 任意   | 不建基础设施       | —                |
| 首次       | 未知   | 实验模式           | —                |

**理论来源一 — APQP + Stage-Gate**

阶段推进不是永远的——有上就要有下。产品会退市，产线会退役。

**理论来源二 — Kaizen 持续改善**

改善不止是"加"，也包括"去掉不再需要的东西"。

**理论来源三 — 错题集降级机制（自身原则）**

"3 个月未触发→降级"在 pitfall 上已验证有效。扩展到 Lane。

**验收标准**：

- [ ] `.ai/conventions/lanes.md` 开头已加入 Lane 生命周期说明
- [ ] 决策矩阵已文档化
- [ ] 每条 Lane 有"最近命中时间"标记，供降级判断

## 四、流程补全：从执行流程到质量策划流程

### 4.2 建议加入的两个质量闸门

**闸门 A：可行性闸门（插入路由策略最前端）**

## 六、各改造项与现有体系的合规性

每条改动均对照宪法九条 + 执政纲领六条，全部合规：

| 改动项            | 宪法冲突 | 执政纲领冲突 | 规避方案                         |
| ----------------- | -------- | ------------ | -------------------------------- |
| 哨兵 A/B/C        | ②⚠️⑧⚠️   | 无           | 哨兵生成后自动清除，不留技术债   |
| 自动 verify 闭环  | ⑨⚠️      | 无           | 不设自动放行，人始终终审         |
| 闸门分级 P0/P1/P2 | 无       | 无           | —                                |
| 降级策略 §五      | ⑧⚠️      | 无           | 每季度审视                       |
| 前置检查点        | 无       | 无           | —                                |
| 非标 Lane 三层    | ⑤⚠️      | 无           | ≥3 次才升级，不预判              |
| 实验模式          | 无       | 无           | —                                |
| Lane 生命周期     | 无       | 无           | —                                |
| Eval Suite        | ①⚠️⑤⚠️   | 无           | 恪守"翻一个场景加一条"，不建全量 |

唯一需要重点警惕的是 Eval 体系——它同时踩了 ①（复杂度不灭）和 ⑤（先跑通再跑对）。规避方案：**Eval Suite 只建"翻过车的 3 个核心场景"，不做全量预见型覆盖。** 翻一个加一条，不翻不加——这就是"21 行 Eval Suite > 全量 Eval 基准"。
