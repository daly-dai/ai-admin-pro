# AI 工程体系改造方案

> 基于制造业百年沉淀（APQP / 精益生产 / 六西格玛 / TOC / FMEA）与当前 ai-admin-pro 体系的相互印证，提出增量式改造方案。
>
> **原则：不推翻现有体系，只做补充和升级。所有修改点均附带制造理论来源。**

---

## 一、现状评估：已映射的和缺位的

### 1.1 制造思想 → 当前体系的精确对应

当前体系的宪法九条 + 执政纲领六条已经事实构成了一套"精益 AI 框架"。下表逐条对应制造理论来源：

| 当前体系             | 制造理论来源                                                                           | 对应关系                            |
| -------------------- | -------------------------------------------------------------------------------------- | ----------------------------------- |
| ⑨ 人是最终责任者     | **丰田"尊重人"** + **Jidoka**（自働化）——机器检测异常即停，但判断和处理由人决定        | 精确                                |
| ① 复杂度不灭         | **TOC（约束理论）**——复杂度不会凭空消失，只会在系统内转移。打破瓶颈A，瓶颈在B处出现    | 精确                                |
| ③ 边界是宪法         | **单元化生产** + **控制计划**——每个制造单元有明确 I/O 边界，内部可自主优化             | 精确                                |
| ② 为修改而设计       | **SMED（快速换模）**——丰田将换模时间从几小时压缩到几分钟，核心是"让变化变便宜"         | 精确                                |
| ④ 假设一切会炸       | **FMEA（失效模式与影响分析）**——制造业投入最大的预防性活动。DFMEA 管设计，PFMEA 管过程 | 精确                                |
| ⑦ 能观测才存在       | **SPC（统计过程控制）** + **Andon 系统**——控制图监控工序稳定，异常拉绳全线可见         | 精确                                |
| ⑧ 技术债是债务       | **TPM（全员生产维护）**——日常保养是投资，事后抢修是负债。预防成本 < 故障成本           | 精确                                |
| ⑤ 先跑通再跑对再跑快 | **APQP 五阶段**——样机→试产→量产爬坡。跳过阶段是制造第一大忌                            | 精确                                |
| ⑥ 代码是给人读的     | **可视化管理**——任何人 5 秒内看明白工作站状态：做什么？正常吗？标准是什么？            | 精确                                |
| 决策预编译           | **SOP（标准作业）**——将专家判断固化为文件，操作工不每次重新决策                        | 精确                                |
| 错题集驱动           | **8D 报告** + **FMEA 反馈闭环**——出问题才深入分析，不出不瞎改                          | 精确，且"3 个月降级"超越了制造 FMEA |
| 复杂度外移           | **Jidoka** + **Poka-Yoke（防错）**——用工具/夹具保证正确，不给犯错空间                  | 精确                                |
| 约束优于自由         | **Poka-Yoke**——物理阻止 > 标语提醒。禁止比建议有效百倍                                 | 精确                                |
| 信息密度优先         | **一页纸标准作业**——好 SOP 一页 A4 讲清，"删掉这句人会犯什么错？"                      | 精确                                |
| 能力分层适配         | **TOC 瓶颈差异化管理** + **APQP 阶段门控**                                             | 精确                                |

### 1.2 已有机理 → 制造参照的交叉印证

当前体系中的一个重要创新——**错题集驱动（翻车了才加规则，不预判）**——恰好解决了传统 FMEA 的最大痛点：

| 传统 FMEA                            | 错题集驱动                      |
| ------------------------------------ | ------------------------------- |
| 事前穷举所有可能失效                 | 事后根据真实 case 追加规则      |
| 容易过度设计（防了很多不会发生的事） | 只防确实发生过的                |
| RPN 打分主观性强                     | 触发频率客观可测                |
| 维护成本高（大量条目长期未触发）     | 3 个月未触发→降级淘汰，自动瘦身 |

**这构成了一个"逆向 FMEA"闭环：不是"FMEA → 控制计划 → 永不更新"，而是"出错 → 分析 → 加规则 → 长期未触发 → 淘汰"。** 这是对传统 FMEA 方法的有意义改进。

### 1.3 缺位清单

与制造业完整体系对照，当前存在三个结构性缺口：

| 缺口                   | 制造参照                                | 缺失的具体表现                                                                                             |
| ---------------------- | --------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| **量化质量指标层缺失** | 六西格玛：CPK / SPC 控制图 / 缺陷率目标 | 有 Poka-Yoke（防格式错），但没有 Eval Suite 作为"过程能力指数"。无法回答"这个 prompt 的 CPK 是多少？"      |
| **风险分级思维缺失**   | TOC：只在瓶颈处用力，非瓶颈容忍不完美   | ①（复杂度不灭）+ ④（假设一切会炸）放在一起，容易推出"每个环节都加强防御"。缺少"哪些炸了可以接受"的清单     |
| **阶段流程框架缺失**   | APQP 五阶段 + PPAP 18 项提交            | 现有流程是**执行流程**（读取→匹配→拆 Task→逐 Task 执行），缺少**质量策划流程**（概念→可行→设计→验证→放行） |

### 新增原则⑩：Eval 是上线通行证

| 属性           | 内容                                                                        |
| -------------- | --------------------------------------------------------------------------- |
| **原则**       | 没有 Eval Suite 的 prompt，跟没有量具的生产线一样危险。                     |
| **判断标准**   | 改 prompt 后，能拿出数据证明变好了吗？                                      |
| **冲突优先级** | ⑩ 在 ④（假设一切会炸）之后、⑦（能观测才存在）之前。防了炸之后，必须有测量。 |

#### 制造理论来源：SPC（统计过程控制）+ CPK（过程能力指数）

**Shewhart（休哈特）1931 年提出控制图理论**：制造过程的输出有天然波动。区分"正常波动"和"异常波动"的唯一方法是持续测量并建立统计基线。SPC 的核心逻辑：

```
1. 建立基线：在过程稳定时连续采样，计算均值 μ ± 3σ 控制限
2. 持续监控：每次采样点在控制限内 → 过程受控。超出 → 怀疑异常，停线排查
3. 不靠感觉："看产品挺好"不能替代控制图
```

**翻译到 AI 编程**：

```
1. 建立基线：对核心模块建 Eval Suite（50-200 个标准用例），跑出 baseline 分数
2. 每次变更跑全量 Eval：改 prompt = 改工艺参数，必须重跑 Eval Suite
3. Eval 分数趋势可视化：每次跑的结果记录，画出趋势图
4. 红线机制：Eval 分数跌破阈值 → 拒绝上线，强制回溯
```

**CPK（过程能力指数）的 AI 翻译**：

| CPK 概念       | 制造含义       | AI 系统对应                      |
| -------------- | -------------- | -------------------------------- |
| 规格上限 (USL) | 可接受的最大值 | 最低可接受的 Eval 分数（如 85%） |
| 规格下限 (LSL) | 可接受的最小值 | 同上（单侧下限更常见）           |
| 过程均值 μ     | 输出平均值     | Eval Suite 平均得分              |
| 过程标准差 σ   | 输出波动程度   | 多次跑的分数标准差（一致性）     |
| CPK ≥ 1.33     | 过程能力合格   | Eval 均分 > 红线 + 标准差小      |
| CPK < 1.00     | 过程能力不足   | 分数波动大或接近红线 → 不可上线  |

#### 落地形式

```
每个核心模块在 .ai/evals/{module}/ 下维护：
  ├── eval-cases.md        ← 标准用例（含输入 + 期望输出约束）
  ├── baseline.txt         ← 当前 baseline 分数 + 跑分日期
  └── history.csv          ← 历史趋势：日期,分数,变更内容

prompt 变更门：
  改 prompt → 跑 Eval Suite → 分数不降才放行 → 记录趋势
```

> **核心洞见**：多数 AI 团队没有"在线 SPC"意识——上线了就上去了，输出质量漂移了都不知道。这正是制造业教的第一课：**能测量才存在**，而 ⑦ 的"能观测"必须有 Eval 作为实现手段。

### 新增原则⑪：在瓶颈处用力

| 属性           | 内容                                                         |
| -------------- | ------------------------------------------------------------ |
| **原则**       | 不是所有"假设会炸"的环节都要防御。只在用户最痛的那一点死磕。 |
| **判断标准**   | 当前整个 AI 管道，用户投诉或输出失败集中在哪一个环节？       |
| **冲突优先级** | ⑪ 在 ⑩ 之后、⑦ 之前。测量之后才能确定瓶颈在哪。              |

#### 制造理论来源：TOC（约束理论）—— Eliyahu Goldratt《目标》（1984）

Goldratt 的核心发现：**任何系统的产出由单一瓶颈决定。在非瓶颈环节优化，对全局产出零贡献。**

```
TOC 五步聚焦法：
1. 识别瓶颈 → 哪个环节产出最低/排队最长？
2. 挖尽瓶颈 → 不增加资源，先榨干瓶颈的现有能力
3. 迁就瓶颈 → 全系统按瓶颈的节奏运行，不在瓶颈上游堆积
4. 打破瓶颈 → 增加瓶颈资源（加设备、改工艺、外包）
5. 回到第 1 步 → 旧瓶颈消失，新瓶颈出现。永无止境。
```

**翻译到 AI 编程**：

```
1. 识别瓶颈 → 用户投诉/Bad Case 集中在：理解错误？输出格式？幻觉？工具调用失败？
2. 挖尽瓶颈 → 不急着上复杂方案。先优化 prompt 措辞、加 few-shot、调整温度
3. 迁就瓶颈 → 在瓶颈未解决前，用 Guardrail + Fallback 兜底，不让失败伤害用户
4. 打破瓶颈 → 微调模型、换模型、上 RAG——只有前三步不够才用重武器
5. 回到第 1 步 → 解决了幻觉，下一个瓶颈可能是延迟。永无止境。
```

#### 关键推论：不要平均用力

当前系统里 `① 复杂度不灭` + `④ 假设一切会炸` 放在一起，容易推出"每个环节都加强防御"。TOC 的补丁是：

```
对每个"可能炸"的环节，先问两个问题：
  Q1: 这个环节如果炸了，用户有多痛？（影响范围 × 严重程度）
  Q2: 不防它，省下的复杂度能不能用在更有价值的地方？

只有 Q1 >> Q2 时才投入防御资源。
```

**防御分级表**：

| 风险等级  | 标准                         | 防御策略                      | 资源投入 |
| --------- | ---------------------------- | ----------------------------- | -------- |
| 🔴 阻断级 | 用户任务完全无法完成         | 全自动闸门拦截                | 高       |
| 🟠 降级级 | 用户任务能完成但体验差       | Guardrail + 监控告警 + 人兜底 | 中       |
| 🟡 瑕疵级 | 不影响任务完成，但不够好     | 错题集记录，攒够 N 例再修     | 低       |
| 🟢 可接受 | 理论上不完美但实践中很少触发 | 不防。触发时再追加入错题集    | 零       |

### 新增原则⑫：反馈必须闭环

| 属性           | 内容                                                              |
| -------------- | ----------------------------------------------------------------- |
| **原则**       | 每个 Bad Case 必须有归宿。分析→改进→验证→固化，四步缺一是假把式。 |
| **判断标准**   | 最近一个 Bad Case，从发现到规则更新，链路完整吗？                 |
| **冲突优先级** | ⑫ 在 ⑪ 之后、⑧ 之前。找到瓶颈用力之后，用力必须有闭环。           |

#### 制造理论来源：8D 报告 + PDCA 的 C→A 环节

**8D 报告**（汽车行业强制客诉处理流程）的 8 个步骤：

| D   | 名称     | 含义                       |
| --- | -------- | -------------------------- |
| D1  | 组建团队 | 确定谁参与分析             |
| D2  | 描述问题 | 用 5W2H 定义问题           |
| D3  | 围堵措施 | 先止血（隔离受影响产品）   |
| D4  | 根因分析 | 5 Why / 鱼骨图，找根本原因 |
| D5  | 永久对策 | 针对根因的纠正措施         |
| D6  | 验证有效 | 实施后确认不再复发         |
| D7  | 预防再发 | 反灌 FMEA / 控制计划 / SOP |
| D8  | 团队总结 | 复盘并表彰                 |

**当前体系的对应与缺口**：

| 8D 步骤     | 当前体系中已有的                   | 缺口                                             |
| ----------- | ---------------------------------- | ------------------------------------------------ |
| D1 组建团队 | —                                  | AI 编程中不是问题，通常一人处理                  |
| D2 描述问题 | 错题集条目已包含触发条件和错误模式 | ✅                                               |
| D3 围堵措施 | —                                  | **缺**: 是否需要紧急修复 prompt？                |
| D4 根因分析 | —                                  | **缺**: 是 prompt 歧义？RAG 误召回？模型偏见？   |
| D5 永久对策 | 错题集里的"正确做法"               | ⚠️ 但有时只记录了症状和躲避方法                  |
| D6 验证有效 | —                                  | **缺**: 改完后有没有跑同样的 case 确认不再出错？ |
| D7 预防再发 | pitfall 条目                       | ✅ 错题集条目就是 D7                             |
| D8 团队总结 | Task 复盘                          | ⚠️ 面向 Task 而非面向 Bad Case                   |

**戴明 PDCA 循环**中，最常断裂的环节是 **C→A（Check→Act）**——检查到异常但没人落实改进。⑫ 的存在就是为了锁死 C→A 这条链。

#### 落地形式

```
Bad Case 处理模板（追加到错题集工作流）:

发现问题
  ↓
记录：现象 + 触发条件 + 用户影响 (D2)
  ↓
紧急止血：是否需要立刻改 prompt 兜底？(D3)
  ↓
根因分析：prompt / RAG / 模型 / 输入解析 / 其他 (D4)
  ↓
改进方案：具体改什么 (D5)
  ↓
验证：用同一个 case + 同类变体确认不再复现 (D6)
  ↓
固化：追加入错题集 / 更新模板 / 加闸门规则 (D7)
```

## 三、AI 架构升级：四个具体项目

### 3.1 升级一：非标 Lane → 三层半结构化工坊

#### 当前状态

```yaml
非标 Lane:
  触发: 匹配不上任何 Lane
  流程: 加载 PRD 模板 → 逐项提取 → 标 [待补充] → 等人工补齐 → 产出 PRD → 拆 Task → 执行
  问题:
    - PRD 写完就作废，不沉淀结构
    - 没有中间层约束，AI 自由度过大
    - "验证昂贵"在这里最严重
```

#### 制造参照：单元工坊（Cellular Job-Shop）

制造业处理"低频但不罕见"的产品，不建全自动产线，但也不毫无章法：

> **单元工坊 = 一个半结构化的模子 + 核心 checkpoints + 单元内自治**

每个产品家族一个单元，单元内高度自治，但不做全厂通用约束。对应 AI：每种"非标里的半标准"一个轻量 Lane。

#### 改造方案

非标 Lane 从"纯兜底"升级为三层结构：

```
Layer 1 — 所有非标强制约束（对标 G1-G15 的通用子集）
  ├─ 输出 Schema：至少定义文件结构 + 命名规范
  ├─ 核心闸门：G1-G6, G11, G13-G15（去掉 CRUD 专用的 G7-G10）
  └─ 生成前检查：走一遍 pitfalls 自动匹配

Layer 2 — 半标准模式库（高频非标的轻量 checklist）
  ├─ 识别规则：同一模式在非标 Lane 出现 ≥3 次 → 升级
  ├─ 形式：~5 条 checklist，不建完整 Lane
  ├─ 存放位置：.ai/nonstandard/{pattern-name}.md
  └─ 示例：
      - dashboard-nonstd.md: 数据源声明、刷新策略、空态处理、颜色主题
      - workflow-nonstd.md: 节点类型定义、状态转换规则、审批人解析
      - landing-nonstd.md: Grid 布局、响应式断点、动画约束

Layer 3 — 错题集自动匹配（已有，非标 Lane 强制走）
  └─ 非标生成时不再跳过 pitfalls 索引，而是把它作为生成前必读文件
```

#### 落地文件

- `.ai/nonstandard/` 目录（新建）
- 非标 Lane 流程描述更新（修改 `.ai/conventions/lanes.md` 非标部分）

#### 理论支撑

| 来源                                 | 提供了什么                                           |
| ------------------------------------ | ---------------------------------------------------- |
| 单元化生产（Cellular Manufacturing） | 不建全自动线，但每个单元有边界和 checklist           |
| SMED（快速换模）                     | 半标准模式就是"快速换模"——不建新 Lane 但快速切入场景 |
| 错题集驱动（自身原则）               | 出现 ≥3 次才升级，不改不升级，防止过度设计           |

### 3.2 升级二：新增实验模式（原型车间）

#### 当前状态

没有显性的"实验模式"。所有生成都受 Lane 和闸门约束。当用户只是想"试一下能不能跑通"时，约束反而拖慢迭代。

#### 制造参照：Skunkworks / Prototype Cell

制造业处理全新探索的方式：成立独立的原型车间，脱离量产产线的所有约束。核心规则：

> **原型车间出来的东西永远不能直接进量产产线。** 验证可行 → 提炼结构 → 建产线。失败了 → 废弃，损失可控。

#### 改造方案

新增一条显性的"实验模式"路由：

```
用户意图明确包含"原型/实验/试试/快速验证/PoC"时：

1. AI 声明进入实验模式
2. 过期所有 Lane 约束、模板、Task 闸门（保留 G11 pnpm verify）
3. 用户可自由迭代，不计入错题集
4. 硬门：实验模式代码不能直接进主干
   → 实验结束后自动弹"结果判定"：
     a. 可行 → 建议提炼为半标准 / 完整 Lane
     b. 不可行 → 废弃，不沉淀
     c. 不确定 → 保留为 PRD 草案，后续再判
```

#### 落地文件

- `.ai/conventions/experiment-mode.md`（新建）

#### 理论支撑

| 来源                           | 提供了什么                                  |
| ------------------------------ | ------------------------------------------- |
| Skunkworks（洛克希德原型团队） | 脱离主线约束的独立实验空间                  |
| APQP 阶段 1-2（策划+产品设计） | 原型属于"可行性验证"阶段，不应受量产约束    |
| TOC                            | 实验失败的成本 < 在约束条件下反复试错的成本 |

---

### 3.3 升级三：Eval Suite 基础设施（宪法级）

#### 当前状态

有 Poka-Yoke（Schema 校验、闸门、输出锁），但没有"输出质量测量"的基础设施。当前的验证依赖：

- 人的 Code Review（昂贵）
- pnpm verify（只测语法/类型/格式）
- 错题集（只防已知的错误模式）

**缺少的**：对"AI 输出好不好"的自动化定量测量。

#### 制造参照：SPC + CPK + MSA

| 制造概念                | AI 系统对应                                                    |
| ----------------------- | -------------------------------------------------------------- |
| **SPC 控制图**          | Eval 分数趋势图——每次变更后自动跑 Eval Suite 并记录            |
| **CPK（过程能力指数）** | Eval Suite 得分及一致性——同一 prompt 跑 N 次的分数均值和标准差 |
| **MSA（量具系统分析）** | Eval 方法本身的可信度——评分 prompt/模型的准确性和一致性        |
| **控制限（UCL/LCL）**   | Eval 分数红线和黄线——低于红线拒绝上线                          |
| **首件检验**            | 每次改 prompt 后第一个生成的结果人工确认                       |

#### 改造方案

```
Eval Suite 三层结构（按投入产出比排序）：

Layer 1 — 轻量断言 Eval（每个核心模块必备）
  ├─ 形式：对 AI 输出做自动化断言检查
  ├─ 覆盖：Schema 合规、关键字段存在、禁止词不在、格式正确
  ├─ 实现：脚本 + JSON Schema validation + 正则
  └─ 成本：低。建一次，永久生效

Layer 2 — 用例集 Eval（高价值模块）
  ├─ 形式：50-200 个标准输入 → 跑 AI → 自动比对期望输出
  ├─ 覆盖：典型场景 + 边界 case + 已知 bad case
  ├─ 存放：.ai/evals/{module}/eval-cases.md
  └─ 成本：中。需要人工编写和持续维护用例

Layer 3 — LLM-as-Judge Eval（最不可替代的场景）
  ├─ 形式：用另一个 LLM 或同一 LLM 的评分 prompt 评估输出质量
  ├─ 覆盖：不能机械判断的维度（语气、完整性、合理性）
  ├─ 注意：必须先做 MSA 验证评分 prompt 本身的可信度
  └─ 成本：高。需要反复调评分 prompt + 人工校准

运行时机：
  - 每次 prompt 变更 → 跑 Layer 1 + Layer 2
  - 每周 → 跑 Layer 2 全量
  - 上线前 → 跑全部三层 + 人工抽样 Layer 3 的 10%
```

#### 落地文件

- `.ai/evals/` 目录结构（新建）
- `.ai/evals/README.md` —— Eval 使用规范

#### 理论支撑

| 来源                | 提供了什么                                                                              |
| ------------------- | --------------------------------------------------------------------------------------- |
| Shewhart SPC 控制图 | 建立基线 → 持续监控 → 异常即停                                                          |
| 六西格玛 CPK        | 量化过程能力。CPK < 1.33 不上线                                                         |
| MSA（量具分析）     | Eval 方法本身必须经过验证，否则测量的是噪声                                             |
| Deming 第 3 条原则  | "停止依赖大规模检验来保证质量。改为在过程中内建质量"——Eval 不是替代闸门，是闸门的量化层 |

---

### 3.4 升级四：Lane 生命周期管理

#### 当前状态

Lane 是静态的——CRUD、大屏、多 Tab、非标——加 Lane 靠人工判断，没有量化标准。

#### 制造参照：需求量 × 重复度矩阵

制造业决定"该建专用产线还是一般工坊"的标准：**如果年需求量 × 重复度超过阈值，建专用线摊销成本。**

#### 改造方案：Lane 生命周期

```
        实验模式（原型车间）
              │
              │ 验证可行
              ▼
        非标 Lane（纯兜底）
              │
              │ 同一模式出现 ≥3 次
              ▼
        半标准（轻量 checklist）
         .ai/nonstandard/{pattern}.md
              │
              │ 同一模式出现 ≥10 次
              ▼
        完整 Lane（模板 + 闸门 + 配方）
         .ai/conventions/lanes.md 新增条目
              │
              │ 6 个月无人匹配
              ▼
        降级为半标准 → 再过 3 个月 → 归档
```

**决策矩阵**：

| 月均需求量 | 重复度 | 策略                   | 存放位置                   |
| ---------- | ------ | ---------------------- | -------------------------- |
| ≥ 10       | 高     | 建完整 Lane            | `.ai/conventions/lanes.md` |
| ≥ 3        | 高     | 建半标准 checklist     | `.ai/nonstandard/`         |
| ≥ 3        | 低     | 保持非标，每例单独处理 | —                          |
| < 3        | 任意   | 非标兜底，不建基础设施 | —                          |
| 首次       | 未知   | 实验模式验证           | —                          |

#### 落地文件

- 修改 `.ai/conventions/lanes.md`，在文件开头加入 Lane 生命周期说明

#### 理论支撑

| 来源                   | 提供了什么                                                |
| ---------------------- | --------------------------------------------------------- |
| APQP + Stage-Gate      | 阶段推进不是永远的——有上有下                              |
| 丰田 Kaizen            | 持续改进包括"去掉不再需要的东西"                          |
| 错题集驱动（自身原则） | "3 个月未触发→降级"已经在 pitfall 上验证有效，扩展到 Lane |

## 四、流程补全：从执行流程到质量策划流程

### 4.1 当前流程（执行流程，已完备）

```
需求 → Lane匹配 → 拆Task → 逐Task执行 → pnpm verify → 完成
```

这是一个**执行流程**，对应制造里的"产线运行"。它管的是"怎么做"。

### 4.2 缺失的流程（质量策划流程）

制造里 APQP 管的是"做之前想清楚"：

| APQP 阶段        | 制造活动                  | 当前体系的状态                                  | 建议补充                                  |
| ---------------- | ------------------------- | ----------------------------------------------- | ----------------------------------------- |
| **1. 策划**      | 客户需求→设计目标、可行性 | ❌ 缺失：没有"这个需求该不该用 AI"的判断步骤    | 在路由策略中加一个**可行性闸门**          |
| **2. 产品设计**  | DFMEA、原型、设计验证     | ⚠️ 部分：PRD 模板（非标）有，但标准 Lane 跳过了 | CRUD/大屏 Lane 前置"模式确认"而非直接生成 |
| **3. 过程设计**  | PFMEA、控制计划           | ✅ 完备：模板 + Task 拆解 + 闸门 + 输出锁       | —                                         |
| **4. 确认**      | 试产、PPAP、CPK           | ❌ 缺失：没有"上线前质量报告"的概念             | Eval Suite 跑完后产出**上线检查清单**     |
| **5. 量产+改进** | SPC、纠正预防             | ⚠️ 部分：错题集覆盖了事后纠正，但没有在线监控   | Eval 在线趋势 + Bad Case 自动采集         |

### 4.3 建议补充的两个闸门

#### 闸门 A：可行性闸门（插入路由策略最前端）

```
在 Lane 匹配之前，先过三个问题：

Q1: 这个需求适合用 AI 生成代码吗？
    Yes → 继续。No → 建议人工编写。

Q2: 当前 Lane/半标准/配方能覆盖核心路径吗？
    Yes → 走对应路径。部分 → 走半标准+人工补位。No → 走实验模式先验证。

Q3: 产出物验收标准明确吗？
    Yes → 进入生成。No → 先定义验收标准（至少输出 Schema + 核心闸门）。
```

> 制造参照：APQP 阶段 1 的"可行性承诺"——在投入设计资源之前，先确认这事能做且值得做。

#### 闸门 B：上线检查清单（追加到 Task 完成之后）

```
全 Task 完成后，追加一个"上线前确认"步骤：

□ Eval Suite（如有）已跑通，分数不降
□ pnpm verify 0 error
□ 核心用户路径手工走通一遍
□ Bad Case 已知列表已更新
□ 本需求涉及的关键决策已记录（为什么选这个方案）
```

> 制造参照：PPAP 18 项提交——量产放行前必须提交的文件包。对应 AI："上线前必须提交的检查清单"。

## 五、实施路线：三期增量叠加

### 原则

- 每一期**不破坏**现有体系
- 每一期产出**独立可用**的增量
- 期与期之间可以暂停，不影响已有功能

---

### 第一期：思想补全 + 非标升级（1-2 个需求周期）

**目标**：把三条新原则加入宪法，升级非标 Lane。

| 步骤 | 做什么                                                              | 产出物                                   |
| ---- | ------------------------------------------------------------------- | ---------------------------------------- |
| 1.1  | 将 ⑩⑪⑫ 三条原则加入 AGENTS.md 和/或 ai-engineering-principles.md    | 更新的原则清单                           |
| 1.2  | 创建 `.ai/nonstandard/` 目录和 README                               | 半标准模式库基础设施                     |
| 1.3  | 将现有非标 Lane 流程改为三层结构                                    | 更新 `.ai/conventions/lanes.md` 非标部分 |
| 1.4  | 从历史非标 case 中识别 ≥3 次出现的模式，提炼 1-2 个半标准 checklist | `.ai/nonstandard/{pattern}.md`           |

**这一期改完，非标场景就从"每次都重新来"变成"第三次开始有章法"。**

---

### 第二期：Eval 基础设施 + 实验模式（2-4 个需求周期）

**目标**：建立 Eval Suite 基础设施，开通实验模式。

| 步骤 | 做什么                                                | 产出物                     |
| ---- | ----------------------------------------------------- | -------------------------- |
| 2.1  | 创建 `.ai/evals/` 目录 + README 使用规范              | Eval 基础设施              |
| 2.2  | 为核心模块创建 Layer 1 Eval（断言检查脚本）           | `.ai/evals/{module}/`      |
| 2.3  | 将 Eval 跑分集成到 prompt 变更后的验证流程            | 流程更新                   |
| 2.4  | 创建实验模式规范 `.ai/conventions/experiment-mode.md` | 实验模式开通               |
| 2.5  | 在路由策略中新增实验模式分支                          | 更新 `routing-strategy.md` |

**这一期改完，就有了"改 prompt 之前和之后，数据说话"的能力。**

---

### 第三期：流程闭环 + Lane 生命周期（持续）

**目标**：补齐流程闸门，让 Lane 体系自我进化。

| 步骤 | 做什么                                                        | 产出物                      |
| ---- | ------------------------------------------------------------- | --------------------------- |
| 3.1  | 在路由策略前端加入可行性闸门                                  | 更新 `routing-strategy.md`  |
| 3.2  | 在 Task 完成流程中加入上线检查清单                            | 更新 `task-gates.md` 或新建 |
| 3.3  | 建立 Lane 生命周期文档和决策矩阵                              | 更新 `lanes.md`             |
| 3.4  | 按决策矩阵，将符合条件的半标准升级为完整 Lane（或反过来降级） | Lane 体系自我进化           |

**这一期改完，体系就从"被设计"变成"会进化"。**

## 六、风险与不做的清单

### 本方案刻意不做的

| 不做的事                                 | 理由                                             |
| ---------------------------------------- | ------------------------------------------------ |
| 不建完整的钩子系统（像 webpack Tapable） | 配方 < 10，不需要多阶段协调。YAGNI               |
| 不把 Lane 全部重构为配方                 | CRUD Lane 经过实战验证，推倒重来是无意义的破坏   |
| 不在第一期就建 LLM-as-Judge              | MSA 验证成本高，先做断言 Eval                    |
| 不预设"哪些场景会成半标准"               | 只在出现 ≥3 次后才提炼——这是错题集驱动的逻辑延伸 |
| 不对现有 G1-G15 闸门做任何修改           | 现有闸门经过大量实战，不做没有实践依据的改动     |

### 风险提示

| 风险                               | 缓解                                                 |
| ---------------------------------- | ---------------------------------------------------- |
| 新增原则让体系膨胀，AI prompt 太长 | 新原则精简到一行，类似现有宪法格式；详细论证放本文档 |
| Eval Suite 维护成本被低估          | 第一期限 Layer 1（断言），零维护。第二期才进 Layer 2 |
| 半标准模式太多变成另一种臃肿       | 生命周期管理包含降级和归档机制                       |

## 七、总结：改造前 vs 改造后

| 维度       | 改造前                               | 改造后                                                    |
| ---------- | ------------------------------------ | --------------------------------------------------------- |
| 指导思想   | 9 宪法 + 6 纲领（覆盖精益/TOC/防错） | +3 条（补上六西格玛的量化、TOC 的风险分级、PDCA 的闭环）  |
| 非标场景   | 纯兜底，每次从头来                   | 三层半结构化工坊，第 3 次开始有章法                       |
| 实验探索   | 被 Lane 约束绑住                     | 显性实验模式，约束自愿松绑，但结果不进主干                |
| 质量测量   | 定性闸门（Poka-Yoke）                | + 定量 Eval（SPC），能回答"改 prompt 之后好了还是坏了"    |
| 流程完整性 | 有执行流程                           | + 质量策划流程（可行性闸门 + 上线检查清单）               |
| 体系进化   | Lane 静态，靠人工加                  | Lane 生命周期自进化：实验→非标→半标准→完整 Lane→降级→归档 |

---

> **一句话**：当前体系已经是制造业思想在 AI 编程领域的完整映射，宪法九条 + 纲领六条构成了自洽的道与法。本轮改造不做任何推翻，只在三个方向做增量——**补上六西格玛的量化层（Eval）、补上 TOC 的风险分级（瓶颈用力）、补上 PDCA 的闭环（反馈归宿）**——让一个已经正确的框架变得更完整。

_理论参考源：Shewhart SPC 控制图理论（1931）、Deming 质量管理十四点（1982）、Goldratt 约束理论（1984）、大野耐一丰田生产系统（1978）、AIAG APQP & PPAP 手册、Poppendieck 精益软件开发（2003）。_
