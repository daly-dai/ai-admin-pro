# 插件化架构 — 指导性路线

> 本文档记录架构演进的设计原理、决策推导和取舍理由。
> 不是操作手册——操作规则在 `.ai/core/routing-strategy.md`。
>
> **核心思想**：借鉴 webpack/Vite 的插件哲学——核心流程不变，扩展可插可拔。

---

## 一、设计起点

### 现状

AGENTS.md 定义的管道系统对 CRUD 场景极其高效——
Lane 匹配 → 模板填空 → pnpm verify，一套机械流程，AI 推理负担为零。

但问题出在管道边界上：

```
CRUD 场景（90%）         非 CRUD 横切场景（10%）
  ✅ 模板覆盖               ❌ 无模板
  ✅ 输出锁明确             ❌ 触发越界确认
  ✅ 闸门完整               ❌ 闸门不适用
```

10% 的横切场景（权限管控、存储层、菜单过滤、路由守卫）没有机制承接，
要么走非标 Lane 的 PRD 兜底（太重），要么频繁越界确认（体验差）。

### 核心洞察

**管道越窄，质量越高。拓宽管道去迁就特殊场景，会牺牲 90% 场景的质量保证。**

正确的做法不是拓宽管道，而是**给特殊场景建同等质量的副管道**——
webpack 的 loader 机制、Vite 的插件接口、ESLint 的 rule 体系，都是这个思路。

## 二、借鉴的架构原则

| 原则                         | 来源                                           | 我们的映射                                              |
| ---------------------------- | ---------------------------------------------- | ------------------------------------------------------- |
| **核心不知道插件存在**       | webpack Compiler 不感知具体 plugin             | Lane 不知道配方存在，配方不修改 Lane                    |
| **插件共享核心底层能力**     | webpack plugin 通过 compilation 访问模块图     | 配方继承公共底盘（pitfalls + G1-G12 + api-conventions） |
| **按需加载**                 | VSCode activationEvents                        | 匹配表索引 → 命中才读取配方文件                         |
| **极简接口**                 | Vite 复用 Rollup 插件格式                      | 配方格式：钩子 + 触发条件 + 做什么                      |
| **核心功能本身也用插件实现** | Vite 的 JSX/CSS/HMR 都是插件                   | 未来 Lane 本身也可以重构为配方（不现在做）              |
| **先有实践再有抽象**         | webpack 的 loader 是 2014 年加的，不是设计之初 | 遇到场景 → 直接处理 → 成功后判断 → 沉淀为配方           |

### 刻意不借鉴的

| webpack/Vite 的做法                      | 不借鉴的原因                                           |
| ---------------------------------------- | ------------------------------------------------------ |
| Tapable 钩子链（优先级排序、异步流水线） | 我们的配方数量 < 10，不需要复杂的排序和异步协调        |
| 社区生态模型（第三方发布插件）           | 配方是项目内资产，不跨项目共享                         |
| Rollup 的 transform/load/resolve 多钩子  | 当前只有一个钩子点（after-page），够用。未来需要再扩展 |

## 四、关键设计决策

### 决策 1：后置任务表，不是钩子链

**选择了什么**：Lane 执行完毕后，扫一张匹配表。关键词命中 → 执行对应配方。

**为什么不是钩子链**：

- 配方数量 < 10，不需要多阶段协调（webpack 有 30+ 内置钩子是因为 100+ 第三方插件生态）
- 当前所有配方都是"CRUD 生成完之后的补充操作"，一个钩子点（after-page）够用
- 钩子链要求声明执行顺序和冲突检测，复杂度远超当前需求

**什么时候升级**：配方超过 10 个，或出现需要在生成前/生成中/验证前不同阶段执行的配方。届时增加 pre-page / pre-verify 钩子点。

### 决策 2：匹配表索引，不是全量预载

**选择了什么**：routing-strategy.md 里只存「关键词 → 配方文件名」的映射（每行 ~30 tokens）。

**为什么不全量预载**：

- 配方数量增长后 prompt 线性膨胀
- 不命中的配方不需要付 token 成本
- 和现有 Lane 匹配表完全同构（关键词 → 查表 → 命中就走），AI 推理负担为零

### 决策 3：粒度闸门，不是路径全禁止

**选择了什么**：router/ layouts/ stores/ 从"全禁止"改为"看改动量"。
≤5 行纯增量直接改，5-30 行告知后改，>30 行或改签名确认。

**为什么**：

- 路径全禁止导致加一个菜单项也要越界确认（假阳性太高）
- AI 会绕过闸门（在自然语言里让你手动改，而不是编辑文件）
- plugins/ 保持全禁止——基础设施变更风险高

### 决策 4：配方不自带闸门

**选择了什么**：配方只声明触发条件和做什么，闸门来自公共底盘。

**为什么**：

- 避免每个配方重复定义质量规则
- 公共底盘是经过大量实践迭代的（P001-P017），配方不需要重新发明
- pnpm verify 对所有路径生效

### 决策 5：先实践后沉淀

**选择了什么**：首次遇到横切场景 → AI 直接处理（规约兜底）→ 成功后三问自检 → 全 Yes 才生成配方。

**为什么**：

- 空想的配方不可靠——不知道步骤是否完整、文件路径是否准确
- 只有跑通过的流程才值得保存
- 三问自检过滤一次性需求（"只遇到这一次"的不占坑）

### 决策 6：Lane 命中边界明确化

**选择了什么**：需求中任一关键词匹配 Lane 触发词表 → 即算命中 Lane（即使需求也包含其他内容）。完全不包含任何 Lane 触发词 → 不命中 Lane。

**为什么**：

- "部分命中"就是命中——不需要完全匹配。一个需求可以同时命中 Lane 和配方
- 避免 AI 在边界上犹豫——犹豫本身就是推理负担
- 和现有 AGENTS.md §一 的匹配逻辑完全一致（"包含关键词即命中"）

### 决策 7：配方执行失败回滚

**选择了什么**：配方指令执行失败时，撤销配方改动，回退到 Lane 执行完毕时的状态。告知用户失败原因，不自动重试。

**为什么**：

- 配方操作的是磁盘文件，不是内存数据结构——改动是实打实的
- 失败后留一个半改状态比完全回退更危险
- 符合核心原则 ⑨（人是最终责任者）——失败后人工判断原因

### 决策 8：沉淀确认机制（AI 提议 + 人工确认）

**选择了什么**：初期配方沉淀采用"AI 生成草案 → 暂停 → 向用户展示提议 → 用户确认后写入"的混合模式。

**为什么**：

- 符合核心原则 ⑨（人是最终责任者）
- AI 判断可能出错（漏沉淀或错误沉淀），人工确认是安全网
- 日常配方执行不暂停（匹配表命中直接执行），只有沉淀提议时才暂停
- 配方成熟后可以考虑自动化（但初期必须人工确认）

## 五、配方沉淀三问

执行成功后 AI 自问：

```
Q1: 这次操作涉及的文件组合，下次还会遇到吗？
    Yes 例：改 stores/user.ts + RequireAuth + MainLayout → RBAC 标准组合
    No  例：改一个特殊业务页面的某个 div 样式 → 仅此一次

Q2: 操作步骤能写成 3-7 条明确指令吗？
    Yes → 可以沉淀
    No  → 太依赖上下文判断，强行写成配方反而绑死手脚

Q3: 这个场景在未来项目中出现的概率 >50%？
    Yes → 沉淀
    No  → 不沉淀，占表空间
```

## 六、演进路线

### 当前（初版）

```
- 匹配表框架就绪
- 配方文件位为空
- AI 遇到横切场景 → 直接处理 → 手工判断是否沉淀
```

### 短期（实践 2-3 个需求后）

```
- 2-3 个配方已沉淀（预期：RBAC 权限管控、localStorage 存储层）
- 新需求命中匹配表 → 自动执行
- 配方创建流程跑通 1-2 次
```

### 中期（实践 5-10 个需求后）

```
- 匹配表扩展到 5-8 行
- 如果需要 → 按 Lane 分组
- 如果单个配方超过 7 条指令 → 考虑拆成 pre + post 两个钩子
```

### 长期（配方 > 10）

```
- 此时才需要升级到正式钩子系统
- 增加执行顺序声明和冲突检测
- 但这是 YAGNI——现在不应该做
```

## 七、名字约定

| 英文             | 中文     | 说明                                     |
| ---------------- | -------- | ---------------------------------------- |
| Lane             | 管道     | CRUD/大屏/多Tab详情 的代码生成流程       |
| Recipe           | 配方     | 针对特定横切场景的生成后补充指令         |
| Recipe Table     | 匹配表   | 关键词 → 配方文件名的索引表              |
| Chassis          | 公共底盘 | pitfalls + G1-G12 + 约定（所有路径共享） |
| Granularity Gate | 粒度闸门 | 按改动行数和风险放行，替代路径全禁止     |
