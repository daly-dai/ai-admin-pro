```markdown
## 一、架构模型
```

```
配方是 Lane 和非标 Lane 之间的**缓冲层**——命中了就不走非标 Lane 的重流程。
非标 Lane 保留作为**安全网**——全新场景兜底，同时也是新配方的**输入源**。

## 二、完整路由流程

### 2.1 Lane 命中判定

Lane 匹配逻辑不变（AGENTS.md §一）。**"命中"的定义**：

- 需求中任一关键词匹配 Lane 触发词表 → 命中（即使需求也包含其他内容）
- 例："新建用户管理 + 权限控制" → 命中 CRUD Lane（"用户管理"命中），同时也扫配方表
- 需求完全不包含任何 Lane 触发词 → 不命中 Lane

### 2.2 路由顺序

```

1. Lane 匹配（AGENTS.md §一）
   ├── 命中 → 执行 Lane（模板填空）
   │ ↓
   │ Lane 执行完毕 → 扫配方匹配表（§三）
   │ ├── 命中 → 读取配方文件 → 按指令执行 → pnpm verify
   │ └── 不命中 → pnpm verify
   │
   └── 不命中 Lane → 扫配方匹配表（§三）
   ├── 命中配方 → 读取配方文件 → 按指令执行 → pnpm verify
   └── 不命中配方 → 非标 Lane → PRD → 确认 → 执行 → verify
   │
   └── 成功后三问自检 → 可能沉淀为新配方

```

### 2.3 配方执行失败处理

配方指令执行失败（文件冲突、语法错误等）→ 撤销配方改动，回退到 Lane 执行完毕时的状态 → 告知用户失败原因 → 配方不标记为完成，人工判断是否修复配方文件。

### 2.4 示例

例 1：
```

→ Lane 匹配: "用户管理" + "CRUD" → CRUD Lane
→ CRUD Lane 执行完毕

→ 执行 rbac-scaffold → pnpm verify


→ 扫配方表: "权限控制" → 命中 rbac-scaffold

→ 扫配方表: 不命中

```
## 三、配方匹配表

Lane 执行完毕后，或 Lane 不命中时，**两个时机都扫此表**。
需求文本命中了「关键词」列的任一关键词，则读取并执行对应配方文件。
多行命中则按表顺序依次执行。配方文件自我完整，不依赖 Lane 执行上下文。

### 3.1 匹配规则

- 需求包含关键词列中任一关键词 → 命中
- 同一关键词命中多行 → 全部执行，按表顺序
- 不同配方含重叠关键词 → 允许。表顺序决定执行顺序

### 3.2 关键词冲突处理

当两个配方共享同一关键词时（如"权限"同时出现在配方 A 和配方 B 中），以表顺序为准：先执行的配方不阻塞后执行的配方。如需互斥（只走一个），在触发条件中使用更精确的关键词区分。

### 3.3 匹配表

| 关键词 | 配方文件 | 说明 |
|--------|---------|------|
| 权限控制、路由守卫、菜单过滤、按钮权限、hasPermission | [rbac-scaffold](.ai/recipes/rbac-scaffold.md) | RBAC 权限管控注入 |
| 前端存储、localStorage mock、无后端先行 | [storage-layer](.ai/recipes/storage-layer.md) | 前端 localStorage 存储层 |
| 左树右表、非标布局、自定义页面布局 | [non-crud-page](.ai/recipes/non-crud-page.md) | 非标准页面布局 |

未命中任何关键词 → 跳过，直接进入 pnpm verify。

## 四、粒度闸门

对原全局禁止列表中的目录（`src/router/`, `src/layouts/`, `src/stores/`），
配方执行时采用粒度判断，替代全有或全无禁止：

| 改动量 | 处理 |
|--------|------|
| ≤5 行，纯增量（加一项配置、加一行导入、加一个数组元素） | 直接改，告知用户 |
| 5-30 行，改逻辑但不动函数签名 | 告知 + 改 |
| >30 行，或改函数签名、删除已有逻辑、重构 | 暂停确认 |

`src/plugins/` 保持全禁止——基础设施变更需用户确认。
```

## 六、AI 推理负担分析

| 场景                     | 改造前                                  | 改造后                                    | 负担变化                               |
| ------------------------ | --------------------------------------- | ----------------------------------------- | -------------------------------------- |
| 命中 Lane + 命中配方     | 模板填空 → verify                       | 模板填空 → 扫表 → 读配方 → 执行 → verify  | 轻微增加（+扫表+读文件，均为机械操作） |
| 命中 Lane + 不命中配方   | 模板填空 → verify                       | 模板填空 → 扫表 → verify                  | 几乎不变（+扫表，和 Lane 匹配同构）    |
| 不命中 Lane + 命中配方   | 非标 Lane → PRD → 拆解 → 执行（重推理） | 扫表 → 读配方 → 执行（机械操作）          | **显著减轻**                           |
| 不命中 Lane + 不命中配方 | 非标 Lane → PRD → 拆解 → 执行           | 扫表 → 非标 Lane → PRD → 执行（同改造前） | 几乎不变                               |

> 最大的负担不是匹配表扫描——匹配表扫描是机械的（和 Lane 匹配表同构）。
> 最大的负担是横切场景每次重新推理一遍 PRD。配方消除的正是这个重复推理。

### 上下文迷失风险

配方在 Lane **完全结束后**才执行，两者串行不交叉。AI 不需要同时记住管道状态和配方进度。

前提（已写入 recipe-conventions.md S3）：配方内容必须自我完整——文件路径明确，不依赖"刚才 Lane 生成了哪些文件"这类隐式上下文。
