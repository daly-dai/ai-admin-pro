# 制造业 × AI 编程交叉参考手册

> 本文档提供制造业思想到 AI 编程的完整术语映射、原则对应关系和深度阅读指南。
> 可作为查表工具独立使用，也可配合 `ai-engineering-renovation-plan.md` 阅读。

## 一、核心术语映射表

### 质量管理体系

| 制造术语         | 英文         | 制造定义                                          | AI 编程对应                                                  |
| ---------------- | ------------ | ------------------------------------------------- | ------------------------------------------------------------ |
| 先期产品质量策划 | APQP         | 从概念到量产的5阶段框架，每阶段有强制交付物       | AI 系统从需求到上线的阶段流程，每阶段有交付标准              |
| 生产件批准程序   | PPAP         | 量产前必须提交的18项文件包，证明"有把握稳定生产"  | 上线前必须通过的检查清单：Eval 报告 + Prompt 版本 + 回滚方案 |
| 标准作业         | SOP          | 每道工序的标准化操作步骤，不依赖个人判断          | Prompt 模板 / Lane 模板，不依赖 AI 自由发挥                  |
| 检验标准         | SIP          | 定义什么叫"合格"，不是凭感觉                      | Eval 断言 + Schema 校验 + 输出格式规范                       |
| 控制计划         | Control Plan | 对每道关键工序规定控制方法、检验频率、反应计划    | Guardrail 规则 + Gate checklist + 异常处理流程               |
| 统计过程控制     | SPC          | 用控制图监控工序波动，区分正常波动和异常波动      | Eval 分数趋势图，每次 prompt 变更后重跑 eval suite           |
| 过程能力指数     | CPK          | 量化工序能否稳定产出合格品（CPK < 1.33 = 靠运气） | 量化 AI 输出稳定性：同一 prompt 跑100次的准确率方差          |
| 测量系统分析     | MSA          | 量具本身准不准？量不准 = 瞎判                     | Eval 本身是否可靠？评分 prompt/模型的准确率一致性            |

### 失效预防

| 制造术语               | 英文       | 制造定义                                               | AI 编程对应                                                              |
| ---------------------- | ---------- | ------------------------------------------------------ | ------------------------------------------------------------------------ |
| 失效模式与影响分析     | FMEA       | 在设计/工艺阶段系统化识别"怎么失败→后果多严重→怎么防"  | 预判 AI Pipeline 每一步的失败模式：输入歧义→RAG 召回错误→幻觉→输出格式错 |
| 防错（日语：ポカヨケ） | Poka-Yoke  | 物理装置让错误不可能发生（如夹具让零件只能一个方向装） | Schema 强制校验、输出格式约束、枚举限制——让错的东西根本上不去            |
| 自働化（日语：自働化） | Jidoka     | 机器检测到异常自动停机，不让缺陷流到下道工序           | Guardrail 自动拦截不合格输出 + 触发 fallback                             |
| 安灯                   | Andon      | 产线工人发现异常拉绳停线，所有人可见                   | Eval 分数跌破红线 → 自动告警 / 自动降级                                  |
| 根本原因分析           | Root Cause | 不是修表面症状，是找到根本原因并消除                   | Bad Case 分析：不是改这一个输出，是改 prompt/RAG 消除这一类错误          |

### 效率与流程

| 制造术语     | 英文         | 制造定义                                           | AI 编程对应                                                |
| ------------ | ------------ | -------------------------------------------------- | ---------------------------------------------------------- |
| 七大浪费     | TIMWOOD      | 搬运/库存/动作/等待/过量生产/过度加工/缺陷         | 无效 token 消耗 / 冗余 context / 复杂架构 / 重复生成       |
| 价值流图     | VSM          | 从原料到交付的全流程可视化，标出哪里在等、哪里在堆 | AI 应用全链路：用户输入→RAG→推理→后处理→输出→反馈，标瓶颈  |
| 快速换模     | SMED         | 把模具切换从几小时压到几分钟                       | Prompt 修改的快速验证机制：改一行 → 跑 eval → 即刻知道效果 |
| 看板拉动     | Kanban       | 下道工序消耗了，上道工序才生产                     | 按需加载：命中了才读文件，不预载全量知识库                 |
| 单元化生产   | Cellular Mfg | 每种产品家族一个独立单元，单元内自治               | 每种场景一个 Lane，Lane 内有自己的模板+约束+闸门           |
| 约束理论     | TOC          | 全链条产出 = 瓶颈产出，非瓶颈优化 = 自欺欺人       | 不看最快的一环，看最卡用户价值的那个瓶颈环节               |
| 改善         | Kaizen       | 小步持续改进，永不停止                             | Prompt 的持续迭代：每周回顾最差10个输出 → 改一点 → 测一下  |
| 全员生产维护 | TPM          | 操作工是设备的第一保养人，不等坏了再修             | 技术债主动管理，不等系统炸了再重构                         |

### 流程框架

| 制造术语  | 英文       | 制造定义                                                           | AI 编程对应                                                   |
| --------- | ---------- | ------------------------------------------------------------------ | ------------------------------------------------------------- |
| 阶段-关口 | Stage-Gate | 每个阶段花完钱前必须过评审门：合格继续，不合格杀掉                 | Lane 匹配→Task 拆解→逐 Task 验证→每个 Task 过闸门才能进下一个 |
| 戴明环    | PDCA       | Plan→Do→Check→Act，所有ISO管理体系的骨架                           | Bad Case：分析(P)→改Prompt(D)→跑Eval(C)→固化进错题集(A)       |
| DMAIC     | DMAIC      | 六西格玛的结构化改进流程（Define-Measure-Analyze-Improve-Control） | 定义问题→测量基线→分析根因→改进prompt/RAG→在线监控不复现      |
| 8D报告    | 8D         | 客诉后的八步闭环：紧急措施→根因→纠正→预防→验证→固化                | Bad Case 处理流程：紧急拦截→分析→改 prompt→验证→沉淀错题集    |

## 二、三大制造哲学 → AI 编程原则映射

### 精益生产 → 精益 AI

| #   | 精益原则 | 在 ai-admin-pro 的实现                     | 状态              |
| --- | -------- | ------------------------------------------ | ----------------- |
| 1   | 识别价值 | AI 生成的内容用户愿意用吗？                | ⚠️ 需要 Eval 量化 |
| 2   | 价值流   | 全链路可视化                               | ❌ 缺失           |
| 3   | 流动     | "逐 Task 执行" = 丰田的"小批量+工序内检查" | ✅                |
| 4   | 拉动     | 按需加载文件（AGENTS.md §零 按需读取）     | ✅                |
| 5   | 尽善尽美 | Kaizen → Bad Case 周回顾                   | ❌ 缺失           |
| 6   | 消除浪费 | "信息密度优先" = 消除信息浪费              | ✅                |
| 7   | 尊重人   | "人是最终责任者"⑨ = Toyota 尊重人          | ✅                |

### 六西格玛 → AI 质量工程

| DMAIC 阶段 | 制造方法                | AI 编程方法                         | 状态           |
| ---------- | ----------------------- | ----------------------------------- | -------------- |
| Define     | 定义CTQ（关键质量特性） | 定义准确性/一致性/幻觉率/格式正确率 | ❌ 缺失        |
| Measure    | 建立测量系统（MSA）     | 验证 Eval 本身是否可靠              | ❌ 缺失        |
| Analyze    | 用数据找关键因子        | 分析 Bad Case 找最大贡献因子        | 部分（错题集） |
| Improve    | DOE 找最优参数          | Prompt A/B 测试 + 系统化参数搜索    | ❌ 缺失        |
| Control    | SPC + 控制计划          | Eval 在线监控 + Guardrail 自动拦截  | 部分（闸门）   |

### TOC 约束理论 → AI 瓶颈管理

| TOC 五步法  | 制造操作                       | AI 编程操作                            | 状态               |
| ----------- | ------------------------------ | -------------------------------------- | ------------------ |
| 1. 找瓶颈   | 测每台机器的实际节拍           | 测 AI Pipeline 每步的用户价值损失      | ❌ 缺失            |
| 2. 挖尽瓶颈 | 瓶颈机器不停机、不做非增值任务 | 瓶颈环节优先保障准确率，其他环节可放松 | ❌ 缺失            |
| 3. 迁就瓶颈 | 全厂节奏按瓶颈节拍走           | 其他环节不抢资源、不提前优化           | ❌ 缺失            |
| 4. 打破瓶颈 | 扩产能/技改                    | 升级瓶颈环节（换模型/加RAG/加约束）    | 部分（缺系统方法） |
| 5. 找新瓶颈 | 回头从步骤1开始                | 持续识别新瓶颈                         | ❌ 缺失            |

## 三、软件设计宪法 → 制造源头对照

| #   | 宪法原则           | 制造源头                                                 | 引用来源                           |
| --- | ------------------ | -------------------------------------------------------- | ---------------------------------- |
| ⑨   | 人是最终责任者     | 丰田"尊重人" + Jidoka——机器检异常即停，但人决定怎么处理  | 大野耐一《丰田生产方式》           |
| ①   | 复杂度不灭         | TOC——复杂度不会消失，只会转移。扩了瓶颈A，别处出瓶颈B    | Goldratt《目标》第13章             |
| ②   | 为修改而设计       | SMED——丰田把500吨冲压机换模从几小时压到3分钟             | 大野耐一《丰田生产方式》第4章      |
| ③   | 边界是宪法         | 单元化生产——每个单元有明确的物料入口和产品出口           | Womack & Jones《精益思想》         |
| ④   | 假设一切会炸       | FMEA——制造业花时间最多的事：DFMEA + PFMEA                | AIAG FMEA 手册第4版                |
| ⑤   | 先跑通再跑对再跑快 | APQP——跳过阶段是制造业第一大忌：试产没过就爬坡 = 灾难    | AIAG APQP 手册第2版                |
| ⑥   | 代码是给人读的     | 可视化管理——任何人5秒内看到工作站状态                    | 大野耐一《丰田生产方式》           |
| ⑦   | 能观测才存在       | SPC + Andon——控制图+产线拉绳，操作工发现问题所有人都知道 | Shewhart《产品制造质量的经济控制》 |
| ⑧   | 技术债是债务       | TPM——不在坏了才修，日常保养是投资，不保养是负债          | 中岛清一《TPM入门》                |

## 四、AI 执政纲领 → 制造源头对照

| #   | 执政纲领     | 制造对应                                                | 制造来源                                        |
| --- | ------------ | ------------------------------------------------------- | ----------------------------------------------- |
| 1   | 决策预编译   | SOP（标准作业）——专家判断固化为文件，操作工不每次决策   | 标准化作业三要素：节拍时间+工序顺序+标准在制品  |
| 2   | 错题集驱动   | 8D + FMEA 反馈闭环——出问题了才深入分析，不出不瞎改      | 福特8D方法论（1987）+ FMEA 迭代更新             |
| 3   | 复杂度外移   | Poka-Yoke——夹具保证零件只能一个方向装，操作工不可能装反 | 新乡重夫《零缺陷质量控制：源头检验与防错系统》  |
| 4   | 约束优于自由 | Poka-Yoke——物理阻止 > 标语提醒。禁止比建议有效一百倍    | 新乡重夫防错等级：Level 1物理阻止 > Level 3标语 |
| 5   | 信息密度优先 | 一页纸标准作业——好的SOP一页A4讲清楚                     | 丰田标准作业票格式                              |
| 6   | 能力分层适配 | APQP阶段门控——不同阶段不同控制力度                      | APQP手册：第3阶段验证要求 ≠ 第5阶段监控要求     |

| ------------------------- | --------------------------------------------------------- | --------------------------------------- | -------------------------------------------------- |

| Mary & Tom Poppendieck    | 精益软件开发——把丰田七原则映射到软件                      | 《精益软件开发》（2003）                | 精益AI的直接前身                                   |
| Gene Kim                  | DevOps = 精益+TOC 在 IT 的应用                            | 《凤凰项目》（2013）                    | 用小说讲TOC在IT的实践，可直接套用到AI Pipeline     |

## 七、推荐阅读顺序

```
第1阶段（建立直觉）
  └─ Goldratt《目标》 → 2天读完，小说体，TOC 入门
  └─ Gene Kim《凤凰项目》 → 1周，TOC + 精益 在IT的完整故事

第2阶段（建立体系）
  └─ 大野耐一《丰田生产方式》 → 精益的原始思想
  └─ Poppendieck《精益软件开发》 → 制造业→软件的完整映射

第3阶段（深化质量）
  └─ Deming《转危为安》 → 质量管理的哲学根基
  └─ Crosby《质量免费》 → 零缺陷思维（与你的"约束>自由"高度共鸣）

第4阶段（AI专项）
  └─ Google SRE Book → 运维视角的SPC+防错（最接近AI Pipeline的运维模式）
  └─ Potemkin et al. 关于 LLM Eval 的最新论文 → 搜索 "LLM evaluation SPC statistical process control"
```
