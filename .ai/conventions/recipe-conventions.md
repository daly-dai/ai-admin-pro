# 配方约定

## 闸门

| #   | 规则                                        | 说明                                       |
| --- | ------------------------------------------- | ------------------------------------------ |
| S1  | 一个配方只覆盖一个横切场景                  | 禁止万能配方。场景不重合就拆成两个         |
| S2  | 触发条件用关键词，`/` 分隔                  | 需求文本包含任一关键词即触发               |
| S3  | 「做什么」每步指定具体文件路径              | 禁止「修改相关文件」「调整布局」等模糊描述 |
| S4  | 输出锁在文件路径中隐式声明                  | 粒度闸门自动生效，无需额外声明             |
| S5  | 配方不自带闸门和规约                        | 闸门来自公共底盘 G1-G12 + pnpm verify      |
| S6  | 新配方需在 routing-strategy.md 匹配表中注册 | 否则 AI 找不到                             |

## 格式

按 `.ai/templates/recipe.md` 骨架填写。配方文件放在 `.ai/recipes/` 目录。

## 沉淀机制

**初期模式：AI 提议 + 人工确认**（符合核心原则 ⑨ 人是最终责任者）。

AI 在处理未命中配方的横切场景后，回答三问自检。全 Yes → **生成配方草案**，**暂停执行**，向用户展示：

```
📦 配方沉淀提议

场景: {一句话描述}
涉及文件: {文件列表}
步骤:
  1. {操作描述}
  2. {操作描述}

以上将保存为 .ai/recipes/{name}.md 并注册到匹配表。是否确认？
```

用户确认 → 写入配方文件 + 注册匹配表。用户拒绝 → 不沉淀。

> 注意：暂停只发生在沉淀提议时。日常配方执行（匹配表命中）不暂停，直接执行。

## 沉淀前自检

AI 在处理未命中配方的横切场景后，回答三问。

全 Yes → 生成配方。任一 No → 不生成。

1. 这次操作涉及的文件组合，下次还会遇到吗？
2. 操作步骤能写成 3-7 条明确指令吗？
3. 这个场景在未来项目中出现的概率 >50% 吗？
