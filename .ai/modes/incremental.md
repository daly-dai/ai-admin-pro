# 阶段⑤：迭代修复

> 最小范围修改。需求变更、bug 修复、加字段、改交互等小修改统一走此流程。

## 前置判断

| 需求类型                           | 走向                                                                       |
| ---------------------------------- | -------------------------------------------------------------------------- |
| 新增表单/详情/列表等标准 CRUD 组件 | 读取 `.ai/tools/scaffold/usage.md`，用 scaffold 生成骨架后再回到本流程定制 |
| 修改已有代码 / 加字段 / 改交互     | 直接进入下方步骤                                                           |

## 步骤

1. **读目标文件**：根据用户描述定位并读取
2. **如涉及类型变更**：同时读取 `src/api/{module}/types.ts`
3. **读错题集**：Read `.ai/pitfalls/index.md` 匹配修改涉及的页面类型
4. **读 sdesign 文档**：如涉及 sdesign 组件修改，读取 对应组件文档
5. **最小范围修改**：只改用户提到的内容，不重构不相关代码
6. **验证**：`pnpm verify`，最多 3 轮

## 约束

- 最小修改原则，不动用户没提到的代码
- 不需要读模板（已有代码就是参考）
- 不需要 spec.md/progress.md

## 输出锁

- **修改已有代码**：🔒 仅允许修改用户指定的目标文件及其直接关联的类型文件，禁止创建新文件
- **scaffold 生成**：🔒 由 scaffold 工具创建的文件 + 后续定制修改范围
