# 复杂组合页分析手册

> 本文件是步骤 3 中分析**复杂组合页**文件时的完整执行指令。
> 复杂组合页是指包含多个功能区域的页面（Tab 切换 / 左右分栏 / Steps 步骤 / 上下分区等），
> 每个区域可能是列表、表单、详情或自定义内容。
>
> 包含：采集维度（WHAT）→ 提取指令（HOW）→ 产出模板（OUTPUT）→ 完整性检查（VERIFY）

---

## 一、采集维度清单（WHAT）

### 1. 页面结构

| 维度         | 说明                                                               |
| ------------ | ------------------------------------------------------------------ |
| 整体布局类型 | Tab 切换 / 左右分栏 / 上下分区 / Steps 步骤 / 混合                 |
| 容器组件     | 使用的容器（Tabs / Steps / Row+Col / 自定义布局）                  |
| 区块数量     | 总共几个独立功能区块                                               |
| 区块列表     | 每个区块的标题/Tab 名称 + 子场景类型（列表/表单/详情/图表/自定义） |
| 默认激活     | 默认显示哪个 Tab / 步骤                                            |
| 区块切换方式 | 点击 Tab / 步骤前进后退 / 始终全部可见                             |

### 2. 每个子区块

> 根据子区块的场景类型，递归应用对应场景的采集维度：
>
> - 子区块是列表 → 按 `list-page.md` 的维度采集
> - 子区块是表单 → 按 `form-page.md` 的维度采集
> - 子区块是详情 → 按 `detail-page.md` 的维度采集
> - 子区块是图表/自定义 → 按下面的通用维度采集

#### 图表/自定义区块通用维度

| 维度       | 说明                             |
| ---------- | -------------------------------- |
| 区块标题   | 显示标题                         |
| 功能描述   | 该区块做什么                     |
| 数据来源   | API 接口 / 父区块数据 / 共享状态 |
| 使用的组件 | 图表库组件 / 自定义组件名称      |
| 交互行为   | 点击/筛选/联动等                 |

### 3. 区块间关联

| 维度     | 说明                                                      |
| -------- | --------------------------------------------------------- |
| 数据流向 | 区块 A 的选择/操作是否影响区块 B 的数据                   |
| 共享状态 | 多个区块是否共享某个状态变量（如选中的 ID、筛选条件）     |
| 操作联动 | 在 A 区块操作后是否需要刷新/更新 B 区块                   |
| 通信方式 | props/emit / provide/inject / eventBus / store / 直接引用 |

### 4. 页面级逻辑

| 维度           | 说明                                                 |
| -------------- | ---------------------------------------------------- |
| 初始化加载顺序 | 页面挂载时各区块的数据加载先后顺序                   |
| 路由参数       | 路由参数与页面状态的关系（如 ?tab=xxx 控制默认 Tab） |
| 权限控制       | 整个 Tab / 区块的显隐是否受权限控制                  |
| 页面标题       | 是否动态标题（根据数据变化）                         |
| 全局操作       | 页面级别的操作按钮（如整体保存、整体提交）           |

## 二、提取指令（HOW）

### 2.1 页面结构识别

```
1. 在 <template> 中找到顶层布局容器：
   - Tabs 组件 → Tab 切换型
   - Steps 组件 → 步骤型
   - Row + Col / Flex 布局 → 分栏型
   - 多个并列的 Card → 分区型

2. 列出每个区块：
   - Tab 型：每个 tab-pane 就是一个区块
   - Steps 型：每个步骤对应的内容区域
   - 分栏/分区型：每个独立功能区域

3. 对每个区块进行场景识别：
   - 区块内有 Table 组件 → 列表型子区块
   - 区块内有 Form + 提交按钮 → 表单型子区块
   - 区块内有 Descriptions → 详情型子区块
   - 区块内有图表组件 → 图表型子区块
   - 其他 → 自定义型子区块
```

### 2.2 子区块递归分析

```
对每个子区块：

1. 确定子场景类型
2. 读取对应场景手册的提取指令
3. 在该子区块的 template 范围内执行提取
4. 注意子区块可能使用独立的子组件（通过 import）
   → 追踪到子组件文件，在子组件中继续分析
```

### 2.3 区块间关联分析

```
分析区块之间的关系：

1. 查看 <script setup> 中的状态变量：
   - 哪些变量被多个区块引用？→ 共享状态
   - 是否有 watch 某个区块的状态变化 → 触发另一个区块更新？

2. 查看 Tab 切换事件（@change / @tab-click）：
   - 切换 Tab 时是否触发数据加载？
   - 是否有懒加载（首次切换到某 Tab 才加载数据）？

3. 查看子组件间的通信：
   - emit 事件 → 父组件 → 另一个子组件的 props
   - provide/inject
   - 直接操作 store
```

### 2.4 页面级逻辑分析

```
1. 分析 onMounted 中的初始化逻辑：
   - 加载顺序（哪些数据先加载）
   - 是否有数据依赖（B 的加载依赖 A 的结果）

2. 分析路由参数的使用：
   - route.query / route.params 控制初始状态
   - Tab 切换是否同步到 URL

3. 分析全局操作按钮：
   - 整体保存/提交 → 需要收集多个区块的数据
   - 提交前的跨区块校验逻辑
```
