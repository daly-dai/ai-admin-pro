## 阶段 C：弱模型适配与轻量化改造

**时间**：2026-03-27 ~ 2026-04-02

### C1. 弱模型轻量化改造

- **背景**: 公司内部部署的模型推理能力有限、上下文窗口小，原架构为强模型设计
- **核心策略**: 用规则驱动替代推理驱动
- **废弃的特性**:
  - 工具优先级链（AI 需要推理用哪个工具）
  - 上下文生命周期管理（9 段式摘要 → 简化为 progress.md）
  - 渐进式自主权控制（AI 根据历史表现自动调整 → 用户显式声明三档模式）
- **新增**: 静态变量表、量化约束（响应≤200 词）、阶段文件添加前置约束章节

### C2. 文档精简系列

- **api-conventions.md**: 324 行→133 行（压缩率 59%），保留所有硬约束
- **feature-spec-workflow.md**: 414 行→60 行（压缩率 85%）
- **form-designer.md**: 123 行→59 行，重复描述彻底清除，独有规则保留
- **核心手段**: 代码块→简表+单示例、反模式→一句话、移除低频配置

### C3. 文档泛化

- **规范**: 所有 AI 文档使用抽象占位符替代具体业务实体
  - `User` → `{Entity}`, `user` → `{module}`, `/api/users` → `/api/{module}`
- **影响文件**: coding-standards.md, architecture.md, incremental-development.md, api-module.md 等
- **目的**: 提升 AI 对多种业务场景的泛化适应能力

### C4. llms.txt 精简优化

- **内容**: 精简 4 个核心模板文件，总计减少约 1165+ 行冗余代码示例
- **原则确立**: llms.txt 为 AI 获取组件属性的唯一权威源，文档源统一

### C5. 请求层重构

- **commit**: `2f9a0f6` (2026-04-02)
- **内容**: 重构请求插件以支持多实例与多后端配置

### C6. 可编辑表格选型沉淀 (P002)

- **commit**: `b2033be` (2026-04-02)
- **内容**: P002 纠错沉淀——SForm type='table' 和 STable 不支持行内编辑，可编辑表格场景使用 EditableProTable
- **根因**: AI 多次尝试用 SForm 的 table 类型处理可编辑表格，实际不支持

### C7. 工作模式与规格书工作流

- **commit**: `eddb5c4` (2026-04-02), `f2e463f` (2026-04-01)
- **内容**: 新增功能规格书生成工作流、工作模式说明
- **产出**: spec-gen 模式、page-first 模式、api-gen 模式等

### D6. 强模型适配升级

- **AGENTS.md 升级为强模型专用入口**:
  - 引用深度取消硬性 2 层上限，改为动态控制（按需读取、自动去重、循环引用终止、15 文件软上限暂停确认）
  - 取消 200 词响应限制，文件操作上限 ≤5 → ≤10
  - 新增上下文生命周期管理、渐进式自主权控制
  - 预读策略从禁止改为「优先当前阶段，按需预读相邻阶段」

### D7. 弱模型单一入口决策

- **决策**: 弱模型环境仅保留 AGENTS-LITE 作为唯一入口
- **设计哲学**: AGENTS-LITE 不是简化系统，而是同一套 .ai/ 基础设施的「另一个入口」——同路不同驾（多停、多读、少猜）
- **职责**: 页面生成类需求→路由到 Skill；修改类需求→T1-T8 快速模板；非标场景→查 .ai/templates/ 仿造

- **输出锁**: 仅允许写入指定路径范围内的文件
- **自包含 bundle**: 模板内联了组件 Props 摘要、错题集提醒、约束规则
- **弱模型友好**: 零决策点，纯执行。把所有「元工作」在设计时预编译完成

## 阶段 G：宏内核→微内核 演进判断

**时间**：2026-04-24

### G1. 触发背景 —— 与同事架构的对比

公司内部多位同事已用弱模型完成了项目的落地，他们的架构特征与本项目形成鲜明对照：

| 维度       | 本项目                                          | 同事方案                                                              |
| :--------- | :---------------------------------------------- | :-------------------------------------------------------------------- |
| Skill 结构 | 1 个 sdesign-gen-page Skill 承载所有能力        | N 个独立 Skill（代码规范、样式规范、功能文档、知识复利...）           |
| 需求澄清   | AGENTS.md 阶段判断关键词匹配                    | 功能文档 Skill：控制 AI 只问 4-5 个问题，每个问题为下一个锁定确定项   |
| 知识增长   | 蒸馏管道（AGENTS.md → AGENTS-LITE.md 单向同步） | 知识复利 Skill：读取提交记录+代码 → 识别可沉淀模式 → 自动生成新 Skill |
| 扩展方式   | 修改 SKILL.md（增量追加内容）                   | 新增独立 Skill 文件（按协议组装）                                     |
| 维护模式   | 个人设计 + 个人维护                             | Skill 协议 + 知识复利（Skill 自己生长）                               |

核心差异被类比为 **安卓（宏内核）vs 鸿蒙（微内核）**：

- 当前 sdesign-gen-page Skill = 宏内核：所有服务（硬约束、API 约定、错题集、验证、页面生成）在一个 Skill 内
- 同事方案 = 微内核：每个规范/约定/验证/生成都是独立 Skill，通过引用协议组装

### G2. 宏内核诊断

**当前 sdesign-gen-page Skill 的内部结构**（SKILL.md 240 行）:

```
§1 路由表（关键词→页面类型→文件清单）
§2 固定步骤表（CRUD 10步 / 详情 8步 / 表单 10步）
§3 硬约束（组件使用、导入规则、类型安全）
§4 API 约定（命名规范、HTTP 方法后缀）
§5 错题集（P001-P006）
§6 验证清单
§7 示例
```

**宏内核的优势**（为什么过去有效）:

- 自包含：Skill 加载后 AI 拥有所有信息，不需要跨文件跳转
- 一致性：所有规则在同一文件中，不存在版本不同步
- 简单：一个人维护，不需要协议协商

**宏内核的边界信号**（为什么现在要变）:

- 240 行且持续增长（每新增 1 个 P 编号、1 条约束、1 个模板都在膨胀）
- 约束规则（§3）与页面生成（§2）耦合——改约束要改 Skill，改 Skill 要测所有页面类型
- 单一维护者：知识增长完全依赖个人沉积，无自动复利
- 不可拆解：sdesign-gen-page Skill 只在 sdesign 中后台项目有效，无法复用到其他项目类型

### G3. 同事「功能文档 Skill」和「知识复利 Skill」的设计提取

**功能文档 Skill**：

```
用户一句话描述 → Skill 控制提问（最多 4-5 轮）
→ 每轮回答锁定一个维度（功能/需求/用户事件/用户故事/...）
→ 输出标准模板文档
```

核心设计：**约束 AI 的提问行为**而非约束 AI 的输出行为。通过限制对话轮次和问题结构，用弱模型也能产出结构化需求文档。

**知识复利 Skill**：

```
读取 git log + 代码变更 → 识别重复模式（组件使用/错误修复/代码结构）
→ 判断哪些模式可沉淀为 Skill → 生成 Skill 草稿 → 人工审批
```

核心设计：**让代码库的增长自动驱动 Skill 的增长**，形成「写代码 → 产 Skill → 更好的代码 → 更多的 Skill」的正向循环。

### G4. 演进判断结论

**不推翻重来，而是拆解重组**。演进的关键不是重写，而是：

1. 把宏内核里的服务拆出来（约束→独立 Skill，约定→独立 Skill，验证→独立 Skill，错题集→独立 Skill）
2. 定义它们之间的通信协议（@require 引用语法、优先级覆盖规则、输出锁隔离）
3. 让知识复利 Skill 成为调度器（自动发现 → 自动蒸馏 → 人工审批）

**不会改变的核心设计**（在微内核中继续保留）:

| 设计         | 在微内核中的保留方式                                            |
| :----------- | :-------------------------------------------------------------- |
| 输出锁       | 提升到调度协议层，跨 Skill 合并校验                             |
| 蒸馏管道     | 从 AGENTS→LITE 扩展到 Skill→Skill 多向蒸馏                      |
| 纠错飞轮     | apply 范围从单一 Skill 扩展到 Skill 集群                        |
| 三级验证     | 各 Skill 独立执行 Level 1+2，调度层合并结果                     |
| 阶段生命周期 | 不依赖于 Skill 结构，继续作为 AGENTS.md 顶层路由                |
| 双入口架构   | AGENTS.md / AGENTS-LITE.md 指向 Skill 组合而非单一 Skill        |
| 三原则       | 弱模型优先 / 漏斗法则 / 范例驱动，决定每个 Skill 的信息编码策略 |

### G5. 演化路线图

```
Phase 1（立刻）: 真实项目验证
  找真实项目从头到尾用这套架构跑一遍
  目标: ≥100 次提交，暴露架构的真实裂痕
  产出: 断裂点记录

Phase 2（1-2 周）: 拆 sdesign-gen-page → Skill 集群
  §3 硬约束 → coding-standards Skill
  §4 API 约定 → api-conventions Skill
  §5+§6 错题集+验证 → verification Skill
  §1+§2+references → sdesign-gen-page Skill（声明 @require 上方 Skill）
  产出: 5 个独立 Skill + 引用协议 v1

Phase 3（1 个月）: 实现知识复利 Skill
  代码提交 → 识别重复模式 → 生成 Skill 草稿 → 人工审批

  AGENTS→LITE 单向蒸馏 → Skill A→Skill B 多向蒸馏
  知识复利 Skill 与蒸馏管道融合
  多项目间 Skill 共享
```
