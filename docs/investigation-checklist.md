# 同事架构调研清单

> 明天去公司逐项确认，回来对照着分析。

---

## 1. 代码地图 Skill

### 核心信息

- [ ] SKILL.md 完整内容（截图或文件）
- [ ] 触发方式：提交后自动？手动命令？
- [ ] 产出文件：叫什么名、放在哪个目录、什么格式（mermaid / ASCII / JSON）？
- [ ] 地图内容：是目录树还是依赖图还是模块关系？

### 判断点

- [ ] 给 **AI 读** 还是给 **人读**？
- [ ] 文件大小（行数），弱模型能不能一次看完？
- [ ] 更新机制：每次提交自动刷新 or 手动执行？
- [ ] 如果自动刷新，hook 是什么？（pre-commit / post-commit / postinstall？）

---

## 2. write-prd 知识沉淀实际产出

### 核心信息

- [ ] 跑过几次后的 `docs/knowledge/` 目录结构
- [ ] 实际生成的文档样例（如 `订单管理.md`），看格式和粒度
- [ ] 沉淀了什么：痛点？流程？业务要素？业务规则？都有还是只有部分？

### 判断点

- [ ] 知识文档是给人看还是给 AI 看的？
- [ ] 同类型功能第二次写 PRD 时，AI 推断准确率有没有明显提升？
- [ ] 知识库有没有膨胀问题？（跑 10 个 PRD 后文档量多大）

---

## 3. Skill 引用协议（@require）

### 核心信息

- [ ] 协议定义在哪里？（独立文档 or 写在某个 SKILL.md 里？）
- [ ] `@require` 具体语法是什么？
- [ ] 优先级覆盖规则：引用的 Skill 冲突时谁优先？
- [ ] 循环引用怎么检测？

### 判断点

- [ ] 这协议是 **真正在用的** 还是 **设计阶段**？
- [ ] 协议有没有给 AI 做版本感知？（A Skill 依赖 B Skill v1.2，B 升级后 A 要不要改？）

---

## 4. 知识复利 Skill

### 核心信息

- [ ] SKILL.md 完整内容
- [ ] 触发方式：定时扫描？手动触发？
- [ ] 识别规则：什么算"可沉淀的重复模式"？（≥3 次出现？固定代码结构？）
- [ ] 审批流程：人工审批在哪一步？

### 判断点

- [ ] 目前 **实际产出过几个 Skill**？还是设计文档？
- [ ] 生成的 Skill 质量怎么样？直接可用还是需要大量手动改？
- [ ] 和 write-prd 的 Step 2.5 有没有打通？（PRD 知识 → 知识库 → 知识复利？）

---

## 5. 全局问题

### 必须问的

- [ ] **这些 Skill 的目录结构是怎样的？** 都在一个目录下还是分散放？
- [ ] **Skill 怎么注册到 AI 编辑器？** 手动配置 settings.json or 放在约定目录自动发现？
- [ ] **整体架构有没有文档？** 有没有一份"Skill 体系总览"能拍回来？

### 最后顺手问的

- [ ] 弱模型下，同事有遇到过"AI 循环修复不存在的问题"吗？怎么解决的？
- [ ] 觉得当前架构最大的坑是什么？（可以省你以后很多时间）
