# 弱模型漏斗法则 (Funnel Principle)

> 给弱模型的信息，不是越多越好，而是信噪比越高越好。

## 核心命题

**弱模型的有效注意力是稀缺资源。所有注入模型的知识，都应该经过"漏斗"过滤——只保留信噪比最高的形式。**

## 问题的本质

大多数 AI 辅助开发架构在知识管理上犯同一个错误：**把"信息量"等同于"帮助量"。** 给模型的文档越多、规则越细、API 参考越全，效果应该越好——对吗？

不对。至少对弱模型来说不对。

Stanford 2023 年的 "Lost in the Middle" 研究揭示了一个关键发现：**语言模型对上下文头部和尾部保持较好的注意力，但中间段的信息会被系统性忽略。** 这意味着 128K 的上下文窗口是理论值，实际有效注意力可能只有 8K-16K。

arXiv 2024 年底的 Cursor Rules 实证研究进一步发现：**过度的上下文供给（规则过载）反而降低生成准确率。** 研究中 ~29% 的规则行存在重复，这种冗余直接浪费上下文窗口并降低模型注意力聚焦能力。

**给弱模型塞 745 行 API 参考文档，不如给它一个 100 行的完整可运行范例。** 不是因为范例"信息量更大"，而是因为范例的"信噪比更高"。

## 漏斗模型

```
                    信息量（从多到少）
                         ↓
    ┌─────────────────────────────────┐
    │  全量知识（745行 API 参考文档）   │  ← 强模型可以消化
    │  ┌───────────────────────────┐  │
    │  │ 规则集（300行规范文档）     │  │  ← 中等模型勉强可以
    │  │  ┌─────────────────────┐  │  │
    │  │  │ 黄金范例（100行代码） │  │  │  ← 弱模型最优区间
    │  │  │  ┌───────────────┐  │  │  │
    │  │  │  │ 错题速查（20行）│  │  │  │  ← 辅助防护网
    │  │  │  └───────────────┘  │  │  │
    │  │  └─────────────────────┘  │  │
    │  └───────────────────────────┘  │
    └─────────────────────────────────┘

    信噪比：错题速查 > 黄金范例 > 规则集 > 全量知识
    ↑ 弱模型应从漏斗底部开始加载
```

### 四层知识编码

| 层级 | 形式     | 信噪比 | 适用模型 | 典型大小    |
| ---- | -------- | ------ | -------- | ----------- |
| L1   | 黄金范例 | 最高   | 所有模型 | ~100-150 行 |
| L2   | 错题速查 | 高     | 所有模型 | ~20-30 行   |
| L3   | 规则集   | 中     | 中等以上 | ~200-300 行 |
| L4   | 全量文档 | 低     | 仅强模型 | 500+ 行     |

**弱模型只加载 L1 + L2，中等模型加到 L3，强模型可以全量加载。**

## 为什么"黄金范例"比"规则文档"更有效

这背后有认知科学和模型架构两个层面的解释。

### 认知层面：推理 vs 模式匹配

规则文档要求模型"理解规则 → 应用到具体场景"，这是**推理密集型任务**。

```
规则："API 方法名必须带 HTTP 后缀"
→ 模型需要：理解"HTTP 后缀"的含义 → 枚举 ByGet/ByPost/ByPut/ByDelete
  → 在生成代码时记住这个约束 → 正确应用
```

黄金范例只要求模型"看到模式 → 复制模式"，这是**模式匹配任务**。

```
范例中：export const getListByGet = ...
→ 模型看到这个命名模式 → 新方法自然命名为 createByPost
```

弱模型的推理能力有限，但模式匹配能力是足够的。**选择与模型能力匹配的知识编码方式，而不是强制模型做它做不好的事。**

### 模型层面：注意力分配

Transformer 的注意力机制是有限资源。300 行规则文档中，每条规则都在争夺注意力权重。模型需要在"记住 API 命名规则"和"记住组件替换规则"和"记住类型导入规则"之间分配注意力——这些规则之间没有结构化关系，模型只能平均分配。

而一个完整范例是一个**有内聚结构的整体**——import 声明、类型定义、组件使用、API 调用……它们在同一个代码文件中自然关联，模型的注意力可以沿着代码结构流动，而不是在离散的规则条目间跳转。

## 黄金范例的设计原则

| 原则                | 说明                                 | 反例                               |
| ------------------- | ------------------------------------ | ---------------------------------- |
| **完整可运行**      | 必须是可直接复制到项目中运行的代码   | 代码片段、伪代码                   |
| **覆盖核心场景**    | 一个范例覆盖一个页面类型的最常见用法 | 试图用一个范例覆盖所有 edge case   |
| **内嵌最佳实践**    | 规则编码在代码里，不需要额外解释     | 代码旁边附一堆注释解释为什么这样写 |
| **控制在 150 行内** | 超过此阈值对弱模型的注意力收益递减   | 300 行的"完整示例"                 |

### 示例：745 行 → 100 行的压缩

```
EditableProTable 全量 API 文档（745 行）：
  - 12 个章节
  - 120+ 个属性说明
  - 多种使用模式（受控/非受控/与 Form 配合/树形...）

→ 压缩为黄金范例（~100 行）：
  1. 一个受控模式完整可编辑表格（~50 行）
     覆盖: value/onChange/columns/editable/recordCreatorProps
  2. 一个与 Form 配合的嵌入示例（~30 行）
  3. 五条做/不做速查（~20 行）
```

90% 的实际使用场景只用到 10% 的 API。黄金范例覆盖的就是这 10%。剩下的 90%——保留给人类查阅和强模型使用。

## 漏斗法则的应用场景

| 场景       | 应用方式                                                  |
| ---------- | --------------------------------------------------------- |
| Skill 设计 | 弱模型路径只加载 L1+L2，强模型路径可加载 L3+L4            |
| 组件库文档 | 为每个核心组件维护一个黄金范例，全量 API 文档作为参考手册 |
| 非标场景   | 每次完成新场景后提取一个黄金范例，扩大 L1 覆盖面          |
| 错题集     | 控制在单表格形式，每条一行，直接可执行                    |

## 一句话总结

**不要问"我给了模型多少信息"，要问"模型真正用到了多少信息"。** 漏斗越窄，弱模型越准。
