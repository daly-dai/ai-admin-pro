# 弱模型优先原则 (Weak Model First)

> 所有 AI 工作流的架构设计，必须先在弱模型上验证可行，再在强模型上优化效率。

## 核心命题

**弱模型是构建安全和健壮 AI 辅助开发架构的基石，不是累赘。强模型能力强，但更能粉饰太平。**

这个结论违反直觉。大多数团队把弱模型当作"降级方案"或"过渡手段"——等模型能力上来就不需要了。但实践中我们发现，弱模型在架构设计中扮演的角色，和 Netflix 的 Chaos Monkey 在分布式系统中扮演的角色完全一样。

## Netflix Chaos Monkey 的启示

2010 年 Netflix 迁移到 AWS 后做了一个当时被认为疯狂的决定：在生产环境中随机杀掉服务器实例。这就是 Chaos Monkey。

他们的工程哲学是：

> "If you don't intentionally build resilience, you'll accidentally build fragility."
> （如果你不有意地构建韧性，你就会无意地构建脆弱性。）

Chaos Monkey 强迫每个服务都必须能容忍节点故障。**弱基础设施迫使架构具备真正的韧性。** 在完美条件下通过的测试，面对真实故障时一击即溃。

这个原理完全适用于 AI 辅助开发：

| Netflix Chaos Engineering    | AI 辅助开发架构           |
| ---------------------------- | ------------------------- |
| 故意杀掉服务器               | 弱模型的"随机丢失指令"    |
| 强迫服务无状态 + 自愈        | 强迫知识编码为可复制范例  |
| 弱基础设施上验证的架构才可靠 | 弱模型跑通的 Skill 才可靠 |
| 强服务器只是锦上添花         | 强模型只是效率倍增器      |

## 强模型的"粉饰"问题

强模型（GPT-4、Claude 3.5+）在代码生成场景中有一个隐蔽的危险：**它靠推理能力掩盖架构缺陷。**

具体表现：

**1. 猜对隐式规则**
文档中没写清楚的约定（比如"Modal 必须用 createModal 工厂函数"），强模型可以从上下文中推断出来。你以为文档够用了，其实是模型在兜底。换一个模型，或者同一个模型注意力没对齐时，这些隐式规则就会被忽略。

**2. 上下文过载时仍然大概率正确**
给强模型塞 300 行规范文档，它可能还是能生成正确代码。你以为知识组织没问题，其实模型正在用大量算力去"消化"本可以更高效传递的信息。

**3. 自行修复类型错误**
强模型遇到类型不匹配时会自行调整，你以为 pitfalls 不需要沉淀。但同样的类型陷阱在弱模型上会反复出现，因为弱模型没有能力自行推断正确的类型处理方式。

**这些都是模型能力在兜底，不是架构在兜底。** 一旦你换模型、模型降级、或者场景复杂度超过能力边界，所有"粉饰"瞬间消失。

## 弱模型暴露了什么

弱模型因为"兜不住底"，会把架构的每一个漏洞暴露出来：

| 弱模型症状                | 暴露的架构问题                            | 正确的修复方向               |
| ------------------------- | ----------------------------------------- | ---------------------------- |
| 文档 300 行就丢失内容     | 知识密度过高，缺乏优先级                  | 漏斗法则，收敛知识编码       |
| 组件库文档已提供但仍出错  | API 文档是"人类参考手册"而非"AI 执行指令" | 范例驱动替代规则驱动         |
| scaffold 产出 AI 反而不懂 | 代码生成链路中有"认知断层"                | 消除 AI 不参与生成的中间环节 |
| 同一错误反复出现          | 错误沉淀机制依赖人工，覆盖率低            | 半自动飞轮                   |
| 跨模块误改文件            | 输出锁是文档约束而非工程约束              | 工程化的 scope 检查          |

**这些问题在强模型上大概率不会暴露，但它们真实存在。** 弱模型是你的X光机。

## 原则表述

**弱模型优先原则（Weak Model First）：**

1. 所有 AI 工作流设计，先在弱模型上验证可行，再在强模型上优化效率
2. 弱模型跑通的架构一定兼容强模型，反之不成立
3. 在弱模型上反复出错的环节，说明架构设计有缺陷，而不是模型能力不够
4. 强模型的价值是"效率倍增器"而非"架构兜底器"

## 类比：前端的 Mobile First

这个原则和前端领域的 "Mobile First" 设计完全同构：

- **Mobile First**：先为屏幕最小、性能最差的设备设计，再往大屏增强。在手机上好用的布局一定在桌面上好用，反之不成立。
- **Weak Model First**：先为上下文最小、推理最弱的模型设计，再往强模型增强。在弱模型上跑通的 Skill 一定在强模型上跑通，反之不成立。

两者的共同逻辑：**在约束最大的环境下验证的设计，自然具备向上兼容性。**

## 实践启示

1. **不要把弱模型当作"等模型进步后就能解决的临时问题"。** 模型能力会提升，但总会有"相对弱"的模型（成本更低、延迟更低、可私有化部署）。为弱模型设计的架构是长期资产。

2. **定期用弱模型"体检"你的 AI 工作流。** 即使日常主要用强模型，也应该定期拿弱模型跑一遍，看看它暴露了什么新问题。

3. **弱模型上的挫败感是信号，不是噪音。** 每一次"弱模型又出错了"，都是在告诉你架构的某个环节可以更健壮。
