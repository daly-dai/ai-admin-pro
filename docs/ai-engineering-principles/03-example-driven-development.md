# 范例驱动开发 (Example-Driven Development)

> 弱模型不看规则，看例子。让模型"抄"而不是"理解"。

## 核心命题

**在 AI 辅助开发中，知识的编码方式决定了生成质量的上限。规则驱动适合强推理模型，范例驱动适合所有模型。**

## 一个发现引发的转变

在实践中我们观察到一个现象：

- 给弱模型提供 300 行规范文档，它经常丢失规则、生成不合规代码，修复多轮
- 但让同一个弱模型**按照已有代码进行仿造**，效果明显好得多

这不是偶然。这指向了一个根本性的差异：**规则驱动是推理密集型的，范例驱动是模式匹配型的。** 弱模型的推理能力有限，但模式匹配能力是足够的。

## 规则驱动 vs 范例驱动

### 规则驱动的工作方式

```
输入：一组规则（"API 命名必须带 HTTP 后缀"、"Modal 必须用 createModal"...）
模型行为：理解每条规则 → 在生成时记住所有规则 → 逐条应用
失败模式：规则数量超过注意力容量 → 遗漏 → 生成不合规代码 → 多轮修复
```

### 范例驱动的工作方式

```
输入：一个完整的黄金范例（可运行代码）
模型行为：识别代码中的模式 → 在新代码中复制这些模式
失败模式：范例未覆盖的场景 → 需要额外范例或退化到规则
```

### 对比

| 维度         | 规则驱动                 | 范例驱动               |
| ------------ | ------------------------ | ---------------------- |
| 上下文消耗   | 高（300 行规则）         | 低（100-150 行范例）   |
| 推理要求     | 高（理解 → 应用）        | 低（模式匹配 → 复制）  |
| 弱模型出错率 | 高（规则遗漏）           | 低（所见即所得）       |
| 灵活性       | 好（规则可覆盖任意场景） | 较差（新场景需新范例） |
| 可维护性     | 一般（规则之间可能冲突） | 好（改范例即改行为）   |
| 强模型表现   | 好                       | 同样好                 |

**范例驱动的劣势是灵活性较差，但这个劣势可以通过"持续积累范例库"来弥补。**

## 为什么行业在向范例驱动收敛

### Vercel v0 和 shadcn/ui

Vercel 的 v0 能够精确生成基于 shadcn/ui 的代码，核心原因不是 shadcn 的文档写得好，而是 **shadcn/ui 的 API 设计本身就是"所见即所得"的组合式结构**。

```tsx
// shadcn 风格（组合式）—— 代码即文档
<Table>
  <TableHeader>
    <TableRow>
      <TableHead>Name</TableHead>
    </TableRow>
  </TableHeader>
  <TableBody>
    {data.map((row) => (
      <TableRow>
        <TableCell>{row.name}</TableCell>
      </TableRow>
    ))}
  </TableBody>
</Table>
```

每一行代码都自解释。模型不需要查阅任何文档就知道 `TableHead` 是表头、`TableCell` 是单元格。

对比配置式组件库（如 antd ProTable、SSearchTable 等）：

```tsx
// 配置式 —— 概念密集，需要文档辅助
<SSearchTable
  columns={[{ title: 'Name', dataIndex: 'name', searchType: 'input' }]}
  requestFn={getListByGet}
  actionColumns={[{ actionType: 'edit', onClick: handleEdit }]}
/>
```

代码更短，但模型需要同时理解 `columns` 的完整 schema、`searchType` 的可选值、`requestFn` 的签名约定、`actionColumns` 的 `actionType` 枚举……

**配置式 API 不是不好——它对人类开发者很好。但它的"概念密度"远高于组合式 API，对弱模型不友好。**

### Cursor Rules 的实证数据

arXiv 2024 年底的 Cursor Rules 实证研究发现：

1. **过度规则供给反而降低准确率**——规则越多，模型越难聚焦
2. **具体示例对生成质量的正向影响最显著**——远超抽象规则的效果
3. **成熟项目倾向于精简规则到核心要点**——经验积累后，规则数量减少，示例质量提高

## 范例驱动的实施方法

### 1. 黄金范例（Golden Example）

每种页面类型维护一个"黄金范例"——一个完整、可运行、包含最佳实践的代码文件。

```
.ai/examples/
  crud-list.tsx          # CRUD 列表页的黄金范例
  crud-form-modal.tsx    # 表单弹窗的黄金范例
  crud-detail.tsx        # 详情页的黄金范例
  editable-table.tsx     # 可编辑表格的黄金范例
```

**黄金范例的要求：**

- 完整可运行（不是代码片段）
- 内嵌所有最佳实践（规则编码在代码里）
- 控制在 150 行以内
- 覆盖该页面类型 90% 的使用场景

### 2. 错题速查（Anti-Pattern Quick Ref）

范例告诉模型"怎么做对"，错题速查告诉模型"什么是错的"。两者配合使用。

```markdown
| 编号 | 场景 | 正确做法 | 错误做法 |
| P001 | 弹层管理 | createModal 工厂函数 | 手动管理 open 状态 |
| P003 | 未使用参数 | (\_, record) => ... | (text, record) => ... |
```

每条一行，直接可执行，不需要额外解释。

### 3. 范例积累飞轮

范例库不是一次性建成的，而是持续积累的：

```
第一次做某类页面 → 手动/半手动完成
                 → 提取最终实现为黄金范例
                 → 下次遇到同类页面，AI 直接仿造
                 → 仿造过程中发现范例不足之处
                 → 更新范例
                 → 范例越来越精准
```

**每个非标场景做完后，把最终代码沉淀为该场景类型的黄金范例。第一次是探路成本高，但第二次遇到类似场景就有范例可抄了。这就是飞轮——做的越多，覆盖越广。**

## 如何处理配置式组件库

如果你的项目已经使用了配置式组件库（不太可能重写为组合式），正确的做法不是放弃组件库，而是**把组件库的使用方式编码进范例**：

| 路径                               | 做法                         | 适用场景                   |
| ---------------------------------- | ---------------------------- | -------------------------- |
| 保留配置式组件库 + 范例驱动        | 为每个核心组件写一个黄金范例 | 已有项目，组件库已深度使用 |
| 弱模型用原生组件，强模型用业务组件 | 按模型能力分流               | 代码风格一致性要求不高     |
| 重新设计组件 API 向组合式靠拢      | 大规模改造                   | 新项目或有充足改造周期     |

**对大多数团队来说，第一条路径 ROI 最高。**

## 一句话总结

**规则是"让模型理解后应用"，范例是"让模型看到后复制"。弱模型更擅长后者。**
