## 二、双入口分支体系

### 2.1 入口选择逻辑

```
用户请求到达
    │
    ▼
模型能力判定（由 IDE / 用户选择 / 入口文件隐式决定）
    │
    ├── 强模型 / 标准模式
    │       │
    │       ▼
    │   AGENTS.md（~200 行，推理驱动）
    │       │
    │       ▼
    │   阶段判断 → 匹配 5 阶段之一
    │       │
    │       ▼
    │   读取对应 modes/{mode}.md → 按步骤执行
    │
    └── 弱模型 / Lite 模式
            │
            ▼
        AGENTS-LITE.md（~120 行，模式匹配驱动）
            │
            ▼
        工作模式选择 → sdesign-gen-page Skill
            │
            ▼
        SKILL.md 的 10 步执行契约 → @FILL 填空模板
```

### 2.2 两套入口的核心差异

| 维度           | AGENTS.md                               | AGENTS-LITE.md                                        |
| :------------- | :-------------------------------------- | :---------------------------------------------------- |
| **规模**       | ~200 行                                 | ~120 行                                               |
| **决策机制**   | 阶段判断（5 阶段生命周期推理）          | 模式匹配（路由表关键词 → 固定分支）                   |
| **知识编码**   | 规则集为主（阶段表、约束表、决策表）    | 范例 + 速查为主（T1-T8、8 条禁止、8 条报错修复）      |
| **流程控制**   | 渐进式自主权（动态调节介入频率）        | 硬停规则（固定阈值熔断：>20 行 / >3 文件 / 配置修改） |
| **错误处理**   | 三级验证 → 最多 3 轮自修复              | 常见报错速查 → 修一轮 → 失败即停                      |
| **执行引擎**   | `modes/` 下 5 个模式文件                | `sdesign-gen-page` Skill                              |
| **代码模板**   | `.ai/templates/` （规范参考型）         | Skill `references/` （填空骨架型 @FILL）              |
| **上下文管理** | 引用深度 / 生命周期管理 / 跨会话 Prompt | 无（硬停规则代替）                                    |
| **适用场景**   | 完整功能开发、多页面任务、复杂决策      | 新建模块、标准 CRUD、小修改                           |

### 2.3 AGENTS-LITE.md 的工作模式分配

```
用户需求关键词
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 新建模块 / 新建页面 / 新增功能      → sdesign-gen-page Skill │
│ 给已有模块加表单 / 详情 / 列表     → sdesign-gen-page Skill │
│ 只需类型定义 / 只需 API 层         → sdesign-gen-page Skill │
│ 改字段 / 加列 / 改文案 / 修 bug    → 修改路径 (T1-T8)      │
│ 非标场景（无 Skill 覆盖）          → 查模板仿造 / 问用户     │
└─────────────────────────────────────────────────────┘
```

## 三、生命周期阶段流转

### 3.1 五阶段线性流程

```
PRD 到达
    │
    ▼
  ① 画 Demo
    │  规范化 PRD → Task 拆解 → 生成 demo 页面 + 占位 API
    │  输入: PRD + "画页面/出骨架/demo"
    │  产出: 页面 + 占位 API + spec.md + progress.md
    │
    ▼
  ② 接口合并（可多轮）
    │  分支A: 有 PRD → 完整合并 + feature-spec
    │  分支B: 无 PRD → 仅生成 API 层
    │  输入: Swagger/接口文档
    │  产出: feature-spec.md (分支A 阻断性) 或 API 代码
    │
    ▼
  ③ 改造适配
    │  根据 feature-spec 改造已有 demo 页面
    │  输入: feature-spec 就绪 + "改造/对齐规格"
    │  产出: 变更清单 → 确认 → 执行修改
    │
    ▼
  ④ 接口对接
    │  占位 URL → 真实 URL，删除 TODO 注释
    │  输入: 真实接口就绪 + "对接/联调/替换mock"
    │  产出: diff 对比 → 确认 → 替换
    │
    ▼
  ⑤ 迭代修复
       最小范围修改
       输入: "改一下/加字段/修复/调整"
       产出: 最小范围改动
```

### 3.2 需求变更回溯

```
PRD 有更新
    │
    ▼
┌──────────────────────────────────────────────────────┐
│ 变更类型                  │ 回到阶段  │ 处理范围       │
├───────────────────────────┼───────────┼───────────────┤
│ 新增页面/模块级功能        │ ① 画 Demo │ 仅新增部分     │
│ 修改数据模型/字段/业务规则  │ ② 接口合并 │ 更新 spec → ③ │
│ 修改页面交互/表单字段/列   │ ③ 改造适配 │ 仅变更的页面   │
│ 删减功能或字段             │ ③ + ⑤     │ 删元素 + 清API │
│ 文案调整/样式小改          │ ⑤ 迭代修复 │ 仅目标文件     │
└──────────────────────────────────────────────────────┘
混合变更取最早阶段，仅对变更部分执行，已完成且未受影响的部分不重做。
```

### 3.3 允许的非线性跳转

```
① ──────────────────────────→ ②  接口一步到位，跳过 demo
① ──────────────────────────→ ④  简单页面不需要改造
① ──────────────────────────→ ⑤  简单页面不需要改造和对接
④ ──────────────────────────→ ②  对接时发现接口有变
```

## 四、Skill 执行引擎

### 4.1 Skill 内部结构

```
sdesign-gen-page Skill
│
├── SKILL.md (~240 行)          ← 执行契约
│   ├── §1 路由表               ← 关键词 → 页面类型 → 模板 + 文档 + 输出锁
│   ├── §2 固定步骤表           ← CRUD(10步) / 详情(8步) / 表单(10步)
│   ├── §3 硬约束速查           ← 组件替换 + 导入规则 + 全局类型
│   ├── §4 API 约定摘要         ← 命名规则 + useRequest + 字段类型映射
│   ├── §5 错题集              ← P001-P005
│   ├── §6 验证清单            ← Level 1 + Level 2
│   └── §7 示例                 ← 输入→产出示范
│
└── references/                 ← 代码模板（基于 Skill 目录）
    ├── api-template.md         ← types.ts 填空模板 + api/index.ts 填空模板
    ├── crud-template.md        ← 列表页模板 + 弹层封装原则 + @FILL 填空版
    ├── detail-template.md      ← 详情页模板（独立页+Drawer 双模式）
    └── form-template.md        ← 表单页模板（独立页+Modal+联动）
```

### 4.2 路由表（确定性分支）

```
用户关键词                      页面类型        读取的模板               读取的组件文档
───────────────────────────────────────────────────────────────────────────────
列表/CRUD/管理/搜索表格/增删改查  → CRUD 列表页 → crud-template + api-template → SSearchTable+SForm+SButton+SDetail
详情/查看/detail                  → 详情页      → detail-template + api-template → SDetail+SButton
表单/新增/编辑/form               → 表单页      → form-template + api-template   → SForm+SButton
```

### 4.3 固定步骤（以 CRUD 为例，10 步不可跳）

```
步骤 1  收集需求            —           需求清单
步骤 2  读取组件 API        4个组件文档   组件知识
步骤 3  读取代码模板        2个模板文件   模板知识
步骤 4  生成 types.ts       填空模板      Entity+Query+FormData
步骤 5  生成 api/index.ts   填空模板      createRequest+CRUD方法
步骤 6  生成 index.tsx       模板+API     列表页
步骤 7  生成弹层组件         弹层原则      FormModal/DetailDrawer
步骤 8  自检                §5+§6        修复问题
步骤 9  pnpm verify          —            验证通过
步骤 10 报告                 —            文件清单
```

### 4.4 填空模板（@FILL）——弱模型兜底

```
原理: 将「理解规范→组装代码」降级为「识别 @FILL → 文本替换」

模板结构:
  ┌─────────────────────────────────┐
  │ // ===== 固定部分：严禁修改 ===== │  ← AI 不可动
  │ import { SSearchTable } from ... │
  │                                  │
  │ // @FILL: 导入 API 函数          │  ← AI 只替换这些行
  │ // @FILL: 导入类型               │
  │ // @FILL: 定义弹窗 ref           │
  │                                  │
  │ const columns = [                │
  │   // @FILL: 表格列配置          │  ← 或填充这些块
  │ ];                               │
  │ // ===== 固定部分结束 =====      │
  └─────────────────────────────────┘

使用场景: 多次生成偏离预期、弱模型不稳定、希望快速产出可运行骨架
非首选: 先尝试标准 step-by-step 生成（步骤 1-7），失败再降级到填空
```

### 4.5 两步文档解析机制

```
Skill 文件中的路径引用有两种解析基准:

references/xxx.md          → 从 Skill 安装目录解析（跨项目可复用的框架模板）
.ai/sdesign/components/xxx  → 从项目根目录解析（项目特定的组件文档）

设计意图: Skill 骨架跨项目复用，项目血肉从当前项目动态加载
```


### 5.2 蒸馏追踪表结构

```
distill-tracker.md 是一个 19 行 × 7 列的矩阵:

| 组号 | 源文件                   | 主题            | 优先级 | LITE状态  | LITE位置       | 确认日期   |
| G01  | AGENTS.md#S1             | 生命周期与路由    | -      | - 不蒸馏   | —              | 2026-04-23 |
| G02  | AGENTS.md#S2-组件         | 组件替换+阻断读取 | P0     | 已覆盖     | S2 组件替换     | 2026-04-23 |
| G03  | AGENTS.md#S2-导入         | 导入规则+全局类型 | P0     | 已覆盖     | S2 禁止模式     | 2026-04-23 |
| G04  | AGENTS.md#S2-流程         | 上下文/自主权     | -      | - 不蒸馏   | —              | 2026-04-23 |
| ...  | ...                      | ...             | ...    | ...       | ...            | ...        |
| G19  | .ai/pitfalls/*.md         | 错题集 P001-P006 | P0     | 已覆盖     | S1 第4步读取    | 2026-04-23 |

优先级:
  P0 = 必须蒸馏（弱模型依赖）
  P1 = 部分蒸馏（有精简版）
  -  = 不蒸馏（强模型专属 或 Skill 已覆盖）

LITE 状态:
  已覆盖   = 完整蒸馏
  部分覆盖 = 有精简版本
  - 不蒸馏 = 有意不蒸馏
```

### 5.3 蒸馏决策模型

```
源文件变更
    │
    ▼
distill:check 检测到漂移
    │
    ▼
这个知识点对弱模型是否必需？
    │
    ├── 是 → 弱模型缺少它无法正确工作
    │        │
    │        ├── 可以精简表达 → P1 部分蒸馏
    │        └── 必须完整表达 → P0 完全蒸馏
    │
    └── 否 → 仅强模型需要 / Skill 已覆盖 / 弱模型无法有效使用
             │
             └── 标记为 - 不蒸馏
```

```
                          │          │    line, rule, message,   │
                          │          │    tool }                 │
                          │          └────────────┬────────────┘
                          │                       │
                          │                       ▼
                          │          ┌─────────────────────────┐
                          │          │  第二层：模式聚合（周频）  │
                          │          │                          │
                          │          │  pnpm pitfall:scan       │
                          │          │  统计 raw.jsonl 高频错误  │
                          │          │  同模式 >= 3 次           │
                          │          │  → 自动生成 pitfall 草稿  │
                          │          │  → error-log/pending/    │
                          │          └────────────┬────────────┘
                          │                       │
                          │                       ▼
                          │          ┌─────────────────────────┐
                          │          │  第三层：人工审批（保留权） │
                          │          │                          │
                          │          │  审查 pending/ 草稿:       │
                          │          │  · 确认 → pitfalls/       │
                          │          │  · 误报 → 删除            │
                          │          │  · 细化 → 编辑后移入       │
                          │          └────────────┬────────────┘
                          │                       │
                          │                       ▼
                          │          ┌─────────────────────────┐
                          │          │  规则生效                  │
                          │          │                          │
                          │          │  AGENTS-LITE.md 自动消费   │
                          │          │  Skill 自动读取            │
                          │          │  下次生成自动规避           │
                          │          └─────────────────────────┘
                          │
                          └──────────────────────────────────────
```

### 6.2 飞轮的关键属性

| 属性           | 说明                                                                      |
| :------------- | :------------------------------------------------------------------------ |
| **对 AI 透明** | verify-wrapper 是 shell 管道，AI 照常跑 `pnpm verify`，不知道后台在记日志 |
| **零额外成本** | 追加 JSONL 是 O(1) 操作，不影响验证流程                                   |
| **软告警**     | pitfall:scan / distill:check 永不阻断（exit 0）                           |
| **自增强**     | 每一圈飞轮转动，pitfalls 增长一点，下次生成多避开一个坑                   |
| **人工审批**   | 不是所有高频错误都值得沉淀为规则，保留人的判断力                          |

### 6.3 飞轮与蒸馏的区别

```
           错误飞轮                              蒸馏管道
           ────────                              ────────
触发:      pnpm verify 失败                      源文件 git 变更
输入:      tsc/eslint/prettier 编译错误           知识文档（AGENTS.md / .ai/ 文件）
产出:      pitfalls/index.md 新增条目             AGENTS-LITE.md 同步更新
方向:     错误 → 规则（后向修复）                 源 → 目标（前向预防）
频率:     每次 verify 自动捕获 + 周频聚合          按需（源文件变更后手动）
检测:     pitfall:scan 聚合 >= 3 次               distill:check 比较 git 日期
```

## 九、三原则与工程映射

### 9.1 原则链

```
弱模型优先原则
  │  「弱模型是架构的 X 光机，强模型是粉饰匠」
  │  「在约束最大的环境下验证的设计，自然具有向上兼容性」
  │
  ▼
漏斗法则
  │  「给模型的信息不是越多越好，是信噪比越高越好」
  │  「黄金范例 > 错题速查 > 规则集 > 全量文档」
  │
  ▼
范例驱动开发
  │  「让模型『抄』而不是『理解』」
  │  「规则是推理密集型的，范例是模式匹配型的」
```

### 9.2 原则在架构中的工程投影

| 原则                          | 在架构中的具体投影                                              |
| :---------------------------- | :-------------------------------------------------------------- |
| **弱模型优先**                | AGENTS-LITE.md 作为独立入口；Skill 填空模板；硬停规则           |
| **漏斗法则**                  | LITE 只加载范例+速查（L1+L2）；distill-tracker 的「不蒸馏」标记 |
| **范例驱动**                  | `@FILL` 填空模板；T1-T8 快速模板；Skill 路由表的模式匹配        |
| **Netflix Chaos Monkey 类比** | 弱模型暴露架构漏洞 → 驱动架构加固 → 强模型自然受益              |
| **Mobile First 类比**         | 最小约束环境验证 → 向上兼容 → 强模型是效率倍增器而非架构兜底器  |

```
## 十四、架构演进方向：宏内核 → 微内核

### 14.1 当前架构的工程类比

当前架构的核心特征：**所有能力集中在一个 Skill 内部**。`sdesign-gen-page` Skill 同时承载了代码规范、组件约束、API 约定、错题集、验证清单、页面生成逻辑。

```

┌─────────────────────────────────────────┐
│ sdesign-gen-page Skill │ ← 宏内核
│ ┌──────┐ ┌──────┐ ┌──────┐ ┌───────┐ │
│ │代码规范│ │组件规范│ │API规范│ │错题集 │ │
│ │ §3 │ │ §3 │ │ §4 │ │ §5 │ │
│ └──────┘ └──────┘ └──────┘ └───────┘ │
│ ┌──────┐ ┌────────────────────────┐ │
│ │验证清单│ │ 页面生成逻辑 §1+§2 │ │
│ │ §6 │ │ + 4 references │ │
│ └──────┘ └────────────────────────┘ │
│ 所有服务在同一上下文，耦合运行 │
└─────────────────────────────────────────┘

```

这与 Android 的 Linux **宏内核**同构：
- 所有模块在内核态运行，通信零开销 → 一次加载 Skill，全部就绪
- 修改一处可能影响全局 → 改组件替换表，§3/§5/§6 全部直接生效
- 扩展需要修改内核 → 加新能力要改 SKILL.md
- 维护依赖个人 → 一个人管所有约束

**适用场景：** 单一项目、单一场景（sdesign 中后台管理系统）。此时宏内核是最佳选择——零通信开销，一次加载全部就绪。

### 14.2 探测到的边界信号

| 信号 | 说明 |
|:--|:--|
| 同事体系的启发 | 多个独立 Skill 自由组装，知识复利 Skill 自动生成新 Skill |
| 两套模板维护 | `.ai/templates/` 和 `references/` 同时维护，熵在增长 |
| 蒸馏管道的单向性 | 目前只覆盖 AGENTS→LITE，未来需要 Skill A→Skill B 的知识同步 |
| 新场景需求 | 如果未来不是只有 sdesign 中后台一种项目类型，需要不同的 Skill 组合 |

这些信号指向同一个方向：**从单一 Skill 内聚所有能力，演进为独立 Skill 通过协议组装。**

### 14.3 目标架构：微内核 Skill 生态

```

          ┌──────────────────────┐
          │    Skill 调度协议     │  ← 微内核：只做通信 + 发现 + 加载
          │  (引用 + 组装 + 隔离)  │

                 │

    │            │            │              │
    ▼            ▼            ▼              ▼

┌───────┐ ┌───────┐ ┌───────────┐ ┌──────────────┐
│ 代码 │ │ 组件 │ │ 功能文档 │ │ 知识复利 │ ← 用户态 Skill
│ 规范 │ │ 规范 │ │ Skill │ │ Skill │
│ Skill │ │ Skill │ │ │ │ (自动生成Skill)│
└───────┘ └───────┘ └───────────┘ └──────────────┘
│ │ │ │
└────────────┼────────────┴──────────────┘
│

```
这与鸿蒙的**微内核**同构：
- 内核只做 IPC + 调度 → 调度协议只做引用 + 发现 + 加载
- 服务在用户态独立运行 → 每个 Skill 独立，崩溃不拖死全局
- 热插拔 → 知识复利 Skill 自动生成新 Skill 并挂载
- 按需剪裁 → 不同项目类型加载不同 Skill 组合

### 14.4 宏内核 vs 微内核 对照

| 维度 | 当前（宏内核） | 目标（微内核） |
|:--|:--|:--|
| Skill 结构 | 1 个 Skill 内嵌所有能力 | N 个独立 Skill，按协议组装 |
| 扩展方式 | 修改 SKILL.md | 新增独立 Skill 文件 |
| 维护者 | 个人 | Skill 协议 + 知识复利 |
| 代码规范 | Skill 内嵌 §3 | 独立 `coding-standards` Skill |
| 组件规范 | Skill 内嵌 §3 | 独立 `sdesign-conventions` Skill |
| API 约定 | Skill 内嵌 §4 | 独立 `api-conventions` Skill |
| 错题集 | Skill 内嵌 §5 | 独立 `pitfalls` Skill（已有 .ai/pitfalls/） |
| 验证 | Skill 内嵌 §6 | 独立 `verification` Skill |
| 页面生成 | Skill 内嵌 §1+§2+references | 独立 CRUD/表单/详情 Skill，引用其他 Skill |
| 知识增长 | 蒸馏管道（单向同步） | 知识复利 Skill（自动从代码库提取） |
| 适用项目 | sdesign 中后台 | 任何项目类型（按需组装 Skill 组合） |

### 14.5 Skill 调度协议设计（微内核的 IPC 层）

拆 Skill 之前，先定义协议。这是整个微内核架构的成败关键。

```

1. Skill 引用语法
   Skill A 声明依赖：
   @require: coding-standards
   @require: api-conventions
2. 加载链
   AI 加载 Skill A → 自动解析 @require → 递归加载依赖 Skill → 去重 + 循环检测

3. 优先级 / 覆盖规则
   项目本地 Skill > 团队 Skill > 通用 Skill
   Skill A 的规则与 Skill B 冲突 → 声明 @override 的一方获胜

4. Skill 注册 / 发现
   约定目录: .qoder/skills/ 或 project-skills/
   索引文件: skills-manifest.json 或自动扫描

5. 输出锁隔离
   每个 Skill 声明自己的输出锁范围
   调度层合并所有已加载 Skill 的输出锁 → 超范围写入被拦截

6. 故障隔离
   一个 Skill 的验证失败 ≠ 全局阻断
   调度层记录失败 Skill，其余 Skill 继续执行

```

### 14.6 演进路线图

```

Phase 1（立刻）：找真实项目跑一遍
目标: 积累 100+ 次提交，暴露架构的真实裂痕
产出: 断裂点记录

Phase 2（1-2 周）：拆 sdesign-gen-page → Skill 集群
行动:
§3 硬约束 → coding-standards Skill
§4 API 约定 → api-conventions Skill
§5+§6 错题集+验证 → verification Skill
§1+§2+references → sdesign-gen-page Skill（保留，声明 @require 上方 Skill）
产出: 5 个独立 Skill + 引用协议 v1

Phase 3（1 个月）：实现知识复利 Skill
基于 Phase 1 的真实代码积累
复用 pitfall-scan 的模式：代码提交 → 识别重复模式 → 生成 Skill 草稿 → 人工审批
产出: 知识复利 Skill

Phase 4（长期）：蒸馏管道升级
从 AGENTS→LITE 单向蒸馏 → Skill A→Skill B 多向蒸馏
知识复利 Skill 和蒸馏管道融合：自动发现 → 自动蒸馏 → 人工审批
多项目间 Skill 共享

```

### 14.7 不会改变的东西

演进不意味着推翻重来。以下核心设计在微内核架构中**继续保留**：

| 设计 | 保留方式 |
|:--|:--|
| **输出锁** | 提升到调度协议层，跨 Skill 合并校验 |
| **蒸馏管道** | 从 AGENTS→LITE 扩展到 Skill→Skill |
| **纠错飞轮** | apply 范围从单一 Skill 扩展到 Skill 集群 |
| **三级验证** | 各 Skill 独立执行 Level 1+2，调度层合并结果 |
| **阶段生命周期** | 不依赖于 Skill 结构，继续作为顶层路由 |
| **双入口架构** | AGENTS.md / AGENTS-LITE.md 仍然存在，指向 Skill 组合而非单一 Skill |
| **三原则** | 弱模型优先 / 漏斗法则 / 范例驱动，决定每个 Skill 的信息编码策略 |

### 14.8 一句话总结

**从「一个人设计所有 Skill 的内容」→「设计让 Skill 自己生长的规则」。**

当前架构是一个设计精密的宏内核——可靠但依赖你。目标架构是一个按协议组装的微内核 Skill 生态——粗糙但能自我复利。演进的关键不是重写，而是把内核里的服务拆出来，定义它们之间的通信协议，然后让知识复利 Skill 成为调度器。
```
