## 二、核心业务流程：一个需求从生到死的完整旅程

### 2.1 主流程：指标口径变更

```
发起阶段          澄清阶段         开发阶段          评审阶段         上线阶段
────────        ────────        ────────        ────────        ────────
业务方以项目      开发人员与       开发人员在        领导审核         审核通过
形式发起口径  →   业务方进行   →   外部平台进行  →   技术口径    →   项目推进
```

## 六、你的成长切入点

基于以上梳理，识别出对你个人成长最有价值的深挖方向：

### 6.1 可迁移的业务知识

这个平台涉及的以下概念在整个数据行业通用，值得深入理解：

- [ ] **数据治理（Data Governance）**：为什么大公司都在做？解决的本质问题是什么？
- [ ] **元数据管理（Metadata Management）**：指标的口径定义本身就是元数据
- [ ] **数据血缘（Data Lineage）**：一个指标从原始数据到最终报表经过了哪些环节
- [ ] **数据质量（Data Quality）**：怎么定义"数据是对的"？你们平台有没有相关能力？

### 6.2 可迁移的系统设计模式

从这个平台中可以提炼的通用系统设计模式：

- [ ] **审批工作流引擎**：几乎所有企业系统都有审批流，本质结构是一样的
- [ ] **版本化管理**：口径变更本质上是"配置的版本管理"，和代码版本管理同构
- [ ] **状态机设计**：项目从发起到上线的状态流转，是标准的有限状态机
- [ ] **多系统协作**：你的平台不做数据开发，但要管理开发流程——这是典型的"编排层"设计

### 6.3 后端学习的自然切入点

结合这个平台学 Java 后端，优先理解这些：

- [ ] 你们平台后端的分层架构是什么？Controller / Service / DAO 分别在哪？
- [ ] 项目状态流转在后端是怎么实现的？硬编码 if-else 还是状态机框架？
- [ ] 审批流用的是什么方案？自研还是 Activiti/Flowable 这类引擎？
- [ ] 口径的版本管理在数据库层面是怎么设计的？（追加新行？还是用 history 表？）
