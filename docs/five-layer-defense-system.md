# 五层防御体系：从想法到落地

> 一份帮你理清思路的文档。讲清楚最初想做什么、实际做了什么、为什么会变成现在这样。

---

## 最初的想法

让 AI 按我的规则生成代码。AI 会犯错，那我就把错误记下来，下次别犯。

很简单，对吧？大概就是：

```
AI 出错 → 我纠正 → AI 记住 → 下次不犯了
```

---

## 落地之后

这个"很简单"的想法，实际搞出了五层。每一层都不是一开始就设计好的——是踩坑踩出来的。

```
pnpm verify 报错
    ↓
L0：查速查表匹配签名 → 命中就按表里的方法修
    ↓
L1：ESLint 已经在 AST 层面拦住了（any、axios、antd 原生组件...）
    ↓ 上面两层都没拦住
L2：AI 生成代码时自己对照硬约束 + 自检清单逐条检查
    ↓
L3：特定场景（弹窗/表格/表单联动）加载对应的错题文件
    ↓
L4：错题集总索引，每次生成页面代码前读一遍
```

---

## 每一层到底是什么

### L1：ESLint 机械拦截（最可靠的一层）

**文件**：`eslint.config.mjs`
**成本**：0 token，不经过 AI
**可靠性**：100%

这是五层里唯一**不需要 AI 参与**的一层。就是在 ESLint 配置文件里写死规则，AI 生成代码后 `pnpm verify` 一跑，违反的直接报错。

拦截的东西：

- `any` 类型 → 报错
- `import axios` → 报错，引导用 `src/plugins/request`
- antd 的 `Table`/`Form`/`Button`/`Descriptions` → 报错，引导用 sdesign 替代
- 纯类型没有 `import type` → 报错
- 函数式组件没写 `const Xxx = () => {}` → 报错
- 未使用的参数没加 `_` 前缀 → 报错

**为什么这是最关键的一层**：因为它不消耗 AI 的上下文。AI 不需要记住"不能用 antd Table"，ESLint 会帮它记住。你越往这一层推，AI 要记的规则就越少。

### L0：verify 错误速查（飞轮的第一圈）

**文件**：`.ai/pitfalls/verify-errors.md`
**触发**：`pnpm verify` 报错后
**成本**：~150 token（读一次速查表）

这是 `pnpm verify` 报错后的第一反应——先别分析，先查表。

表里有 20+ 条错误签名 → 修复方法的映射：

| 错误签名                                                         | 修复方法                     |
| ---------------------------------------------------------------- | ---------------------------- |
| `TS2307: Cannot find module 'src/api/xxx'`                       | 检查 api 文件是否存在        |
| `TS2322: Type 'any' is not assignable`                           | 改 `Record<string, unknown>` |
| `TS2339: Property 'loading' does not exist on type 'SFormProps'` | 用 `<Spin>` 包裹             |
| `no-restricted-imports: 'antd' import 'Table'`                   | 换 SSearchTable              |
| ...                                                              | ...                          |

匹配到了 → 直接按表里的方法修。匹配不到 → 表底部有"未匹配时的组件文档路由表"，按报错关键词定位要读的组件文档。

**这里有个关键机制**：`pnpm verify` 不是原生的 tsc/eslint/prettier。它被 `.ai/tools/verify-wrapper.ts` 包装过。每次 verify 失败，所有错误自动追加到 `.ai/error-log/raw.jsonl`，格式是结构化的 JSON：

```json
{
  "timestamp": "2026-04-30",
  "file": "src/pages/user/index.tsx",
  "line": 42,
  "col": 5,
  "rule": "TS2322",
  "message": "Type 'string' is not assignable",
  "tool": "tsc"
}
```

这就是飞轮的**动力源**——错误自动采集，不需要人手动记录。

### L2：硬约束 + 自检清单

**文件**：`AGENTS.md` §2 + `AGENTS-LITE.md` §2 + `.ai/conventions/verification.md` Level 2
**触发**：AI 会话加载 AGENTS.md 时 + 生成代码后自检
**成本**：~200-500 token（常驻上下文）

硬约束是 AI 在会话中一直"记得"的规则——组件替换表、导入规则、API 命名规范、全局类型。这些写在了 AGENTS.md 和 AGENTS-LITE.md 里，AI 每次会话都会读到。

自检清单是 AI 生成完代码后逐条 check 的 9 项：

1. 业务页面用了 sdesign 组件而非 antd 原生组件
2. 没有 any 类型，没有直接 import axios
3. API 方法名带 HTTP 后缀（`getListByGet` 而非 `getList`）
4. 字段联动用了 `SForm.useWatch` 而非 `type: 'dependency'`
5. 确认弹窗用了 `Modal.confirm` 而非 `SConfirm`
6. Modal/Drawer 用了 `createModal`/`createDrawer` 工厂函数
7. API 调用走 `useRequest` 包装
8. 写操作 `useRequest` 配了 `onSuccess`
9. `types.ts` 类型完整（Entity + Query + FormData）

**L2 的特点**：拦的是 ESLint 拦不住的**语义错误**。比如"你用了 Form，但字段联动的写法不对"——这在语法上没问题，ESLint 不会报错，但业务上有问题。这种只能靠 AI 自己对照清单查。

### L3：场景文件模式触发

**文件**：`.ai/pitfalls/` 下的详细文件（`modal-drawer-encapsulation.md`、`editable-table.md`、`unused-params-in-render.md`）
**触发**：AI 进入特定场景时才加载
**成本**：~200 token/次（按需加载，不常驻）

当 AI 要生成弹窗代码时 → 触发加载弹窗封装错题文件。要生成可编辑表格时 → 触发加载可编辑表格错题文件。

这些文件很小（150-200 token），正反例对照——"错误写法长这样，正确写法长那样，为什么？一句话。"

**和 L2 的区别**：L2 是常驻的通用规则，L3 是场景化的专项规则。弹窗的坑只有生成弹窗时才需要关心，平时不占上下文。

### L4：错题集速查

**文件**：`.ai/pitfalls/index.md`（P001-P015）
**触发**：每次生成页面代码前
**成本**：~200 token（读一次）

15 条核心规则，按"适用场景"列标注了每条规则适用于什么页面类型。AI 只需要关注与当前页面类型匹配的行。

```
P001: 列表页 + Modal → 禁止手动管理 open，用 createModal
P004: 表单页 + 联动 → 禁止 type:'dependency'，用 useWatch
P008: 枚举列/下拉 → 禁止硬编码 options，用 dictKey
...
```

## 一个完整的错误处理链路

假设 AI 生成了一个 CRUD 页面，然后在 `pnpm verify` 时报了错。整个链路是这样跑的：

```
1. pnpm verify 执行
   └→ verify-wrapper.ts 自动把错误写入 .ai/error-log/raw.jsonl  ← 零人力

2. AI 看报错信息
   └→ 先查 L0 速查表（verify-errors.md）
      ├→ 签名匹配 → 按表里的修复方法执行 → 重新 verify
      └→ 签名不匹配 → 查底部的组件文档路由表 → 读对应组件文档 → 修复

3. 修复后重新 pnpm verify
   └→ 最多 3 轮
      ├→ 通过 → 到 L2 自检
      └→ 3 轮还不过 → 停止，报告用户

4. 用户纠正了 AI 的一个新错误模式
   └→ 触发纠错沉淀流程（correction-workflow.md）
      ├→ 能加 ESLint 规则？ → 加 L1（最优）
      ├→ 高频出错？ → 写进 L2 自检清单或硬约束
      ├→ 特定模式触发？ → 创建 L3 场景文件
      └→ 低频但需记录？ → 追加 L4 错题集索引
```

## 五层的核心逻辑

不是五层平级叠加，是**从高成本向低成本迁移**。

| 层  | 机制                        | 成本        | 目标                     |
| --- | --------------------------- | ----------- | ------------------------ |
| L4  | 总索引，AI 读               | AI token    | 记录所有已知错误模式     |
| L3  | 场景触发，按需读            | AI token    | 专项错误不占常驻上下文   |
| L2  | 常驻约束 + 自检             | AI token    | 高频规则，AI 一直记得    |
| L0  | 速查表匹配                  | AI token    | verify 报错的快速修复    |
| →   | ← 越往上，越想让规则脱离 AI | →           | →                        |
| L1  | ESLint AST 拦截             | **0 token** | 确定性机械拦截，终极目标 |

**每次沉淀一个新错误，都问同一个问题：「这个规则能改成 ESLint 规则吗？」** 能 → 推到 L1，AI 永远不用再记这条。不能 → 放到它该在的层，设置上限，超限淘汰。

这就是为什么你的体系不是"规约越来越多"——它有淘汰机制，有容量上限，有晋升路径。它是一个**活的系统**，不是一本越写越厚的规则手册。

## 和最初想法的对比

| 最初想的                   | 实际做的                                                                |
| -------------------------- | ----------------------------------------------------------------------- |
| AI 犯错 → 我纠正 → AI 记住 | AI 犯错 → 五层分别拦截 → 按优先级分配处理方式                           |
| 规则越积越多               | 每层有容量上限 + 淘汰策略 + 晋升路径                                    |
| 靠 AI 自己记住             | 能转 ESLint 的转 ESLint，能查表解决的查表解决                           |
| 一层搞定                   | 五层分工：L1 机械拦截、L0 速查修复、L2 常驻约束、L3 场景触发、L4 总索引 |
| 人驱动                     | verify-wrapper.ts 自动采集 + raw.jsonl 自动沉淀                         |

**复杂度的增加不是因为设计过度——是因为弱模型的现实迫使你只能在确定性系统（L1）和 AI 提示（L0/L2/L3/L4）之间寻找最优分工。** 每一层都是被具体问题逼出来的，不是凭空设计的。

## 和整体架构的关系

五层防御体系是你的 AI 工程化体系里**质量保障**那一环。它在整个流程里的位置：

```
PRD 到达
  ↓
⓪ PRD→Spec（前端范围提取）
  ↓
① 画 Demo / ② 接口合并（代码生成）
  ↓
← 五层防御体系在这里起作用 →
  L4 错题集 → L3 场景文件 → L2 硬约束 → L0 速查 → L1 ESLint
  ↓
pnpm verify 通过
  ↓
③ 改造适配 → ④ 接口对接 → ⑤ 迭代修复
```

它不替代代码生成，也不替代需求理解。它只做一件事：**AI 生成代码之后、交付给人之前，把能拦的错误都拦住。**

---

## 怎么讲给面试官

不用讲五层。讲三层就够了：

1. **确定性拦截层**（L1）：把规则写成 ESLint 配置，不依赖 AI，100% 可靠，0 token 消耗。所有能用 AST 判断的规则——禁止 any、禁止直接 import axios、强制组件替换——都在这一层。

2. **速查修复层**（L0）：把常见的类型错误和修复方案做成签名 → 修复方法的映射表。AI 看到报错先查表，大部分情况不需要分析就能直接修复。

3. **知识沉淀层**（L2-L4）：高频规则常驻 AI 上下文，场景化规则按需加载，低频规则索引用完即走。每层有容量上限和淘汰策略，确保规则总量不膨胀。

核心思想一句话：**把对 AI 的约束逐步迁移到确定性系统里，让 AI 需要记住的规则越来越少。**
