# LITE & AGENTS 工业级演进方案

> 核心目标：**相同场景产出一致代码，不同模型量力而行，整体质量可度量可收敛。**
>
> 工业级 = 一致性（同场景同产出）+ 最优性（不满足于模板下限）+ 可观测（质量有数字）+ 可进化（越用越好）

---

## 〇、两条路线的分工

```
         LITE 路线                      AGENTS 路线
         ────────                       ────────
适用      公司内部 AI（弱推理）            强推理模型探索
目标      一致性 + 速度                   最优性 + 可解释
角色      AI = 熟练工人                   AI = 初级架构师
约束密度   高（锁死选择权）                低（门禁后置兜底）
前置阅读   零（Skill 内建）               2 文件（brief + 金标准）
生成方式   Skill 模板填空                  范例驱动 + 自评审
验证      表驱动修复                       门禁后置
```

**共享资产**：pitfall 数据库、verify 管道、非标 Lane 兜底、配方沉淀框架

---

## 一、LITE 路线演进

### 阶段 L.1 — 跑通（当前 → 1 周内）

**目标**：验证 Skill + 模板 + verify 全链路能产出可运行的代码。

| #     | 行动                                  | 验收标准                     |
| ----- | ------------------------------------- | ---------------------------- |
| L.1.1 | 用一个简单 CRUD（商品管理）走完整管道 | 生成代码 pnpm verify 0 error |
| L.1.2 | 用一个查询列表页走完整管道            | 同上                         |
| L.1.3 | 用一个表单页走完整管道                | 同上                         |
| L.1.4 | 记录每步消耗的轮次和遇到的错误        | 产出第一份管道执行日志       |

**产物**：管道执行日志（Lane 命中率 / verify 轮次 / 错误类型分布）

---

### 阶段 L.2 — 一致（1-2 周）

**目标**：相同场景两次生成，产出代码结构、命名、模式完全一致。

| #     | 行动                                              | 验收标准                                   |
| ----- | ------------------------------------------------- | ------------------------------------------ |
| L.2.1 | 从 L.1 日志中提取"模型写出了不同代码"的差异点     | 差异清单                                   |
| L.2.2 | 对每个差异点加锁——在 Skill 模板或禁止模式表中固化 | 差异点归零                                 |
| L.2.3 | 两次生成同一个 CRUD，diff 两套代码                | 仅字段名和业务逻辑不同，结构和模式完全一致 |
| L.2.4 | 把 L.1-L.2 发现的新错误签名加入 verify-errors.md  | 错误匹配表覆盖率 > 80%                     |

**关键做法**：用 `diff` 而不是人工审查来验证一致性。两次生成的代码 diff 应该只有业务字段名不同。

---

### 阶段 L.3 — 覆盖（2-4 周）

**目标**：扩展 Skill 覆盖范围，减少非标 Lane 触发次数。

| #     | 行动                                                  | 验收标准             |
| ----- | ----------------------------------------------------- | -------------------- |
| L.3.1 | 统计实际项目中非标 Lane 触发频率和场景类型            | 非标场景分类         |
| L.3.2 | 最高频非标场景（出现 ≥3 次）→ 升级为 Skill 新页面类型 | 非标触发率降至 < 20% |
| L.3.3 | 次高频非标场景 → 沉淀为配方                           | 配方表开始有内容     |
| L.3.4 | 低频非标场景保持 prd-fallback 兜底                    | 不强行覆盖一次性的   |

**原则**：出现 3 次才沉淀，不预判。保持"错题集驱动"。

---

### 阶段 L.4 — 自治（持续）

**目标**：AI 在生成时主动发现并修复问题，减少人工介入。

| #     | 行动                                                         | 验收标准           |
| ----- | ------------------------------------------------------------ | ------------------ |
| L.4.1 | verify 报错 → AI 自主匹配 §3 表修复 → 3 轮内通过率达到 > 90% | 人工介入次数 < 10% |
| L.4.2 | 新增错误类型自动建议加入 verify-errors.md                    | 错误表持续生长     |
| L.4.3 | 超过 3 个月未触发的错误条目自动降级/归档                     | 错误表不膨胀       |

## 二、AGENTS 路线演进

### 阶段 A.1 — 减负（当前最优先）

**目标**：砍掉强模型不需要的前置负担，从"防犯错"切换到"信模型 + 验结果"。

| #     | 行动                                                                        | 验收标准           |
| ----- | --------------------------------------------------------------------------- | ------------------ |
| A.1.1 | 预读从 8 文件压到 2 文件：`project-brief.md`（≤200 行）+ 当前场景金标准范例 | 生成前消耗 ≤ 2 轮  |
| A.1.2 | 其余文件标注触发条件，AI 按需主动读取而非强制预读                           | 按需读取机制就绪   |
| A.1.3 | 重写 AGENTS.md §零 预读表——从"必须读"改为"遇到以下情况时读"                 | AGENTS.md 更新完成 |

**改前 vs 改后**：

```
改前：project-brief → routing-strategy → task-splitting → lanes → template → sdesign → pitfalls → recipes
      7 轮预读，代码还没写

改后：project-brief + 金标准范例
      1-2 轮预读 → 开始生成 → 门禁卡住再按需查规范
```

---

### 阶段 A.2 — 升级闸门（1-2 周）

**目标**：从语法级闸门升级到架构级闸门。语法正确性交给 verify，架构完整性交给闸门。

| #     | 行动                                                                                 | 验收标准                             |
| ----- | ------------------------------------------------------------------------------------ | ------------------------------------ |
| A.2.1 | 删掉 G1-G15 中对强模型无意义的语法级条目（G1 零 any / G3 import type / G6 参数前缀） | 闸门从 15 条压缩到 ~8 条             |
| A.2.2 | 新增架构级闸门                                                                       | 覆盖关注点分离、副作用隔离、跨层调用 |
| A.2.3 | 语法级规则全部下沉到 eslint（机器检查），不要放在 AI 闸门（人工推理）                | AI 闸门只保留 eslint 查不到的东西    |

**新增架构级闸门**（强模型才需要被检查的）：

| #   | 规则                                                                       | 检查方式 |
| --- | -------------------------------------------------------------------------- | -------- |
| G16 | API 层不含 UI 逻辑（不在 api/index.ts 里调 message.error / Modal.confirm） | review   |
| G17 | 页面不直接操作 localStorage（走 store 或 api 层）                          | review   |
| G18 | 跨页面状态不通过 props 传递（走 store 或 URL params）                      | review   |
| G19 | 一个 useEffect 做一件事                                                    | review   |
| G20 | 删除操作有二次确认（Modal.confirm，不用 window.confirm）                   | review   |
| G21 | 新增/编辑提交后刷新列表（不在列表组件内手动 setState）                     | review   |

---

### 阶段 A.3 — 金标准驱动（2-4 周）

**目标**：从"模板填空"切换到"范例驱动 + 自评审"。强模型从范例推理模式，而不是背诵规则。

| #     | 行动                                                                    | 验收标准           |
| ----- | ----------------------------------------------------------------------- | ------------------ |
| A.3.1 | 为每种 Lane 选定 1 个金标准范例页面（真实项目中质量最高的那个）         | 4 个金标准范例就位 |
| A.3.2 | 生成流程改为：读范例 → 推理模式 → 生成 → 自评（对比范例，好在哪差在哪） | 自评审步骤落地     |
| A.3.3 | 自评不通过 → 修正后重新生成，不靠闸门逐条卡                             | 生成质量不低于范例 |
| A.3.4 | 如果新生成代码显著优于金标准 → 提升为新金标准                           | 金标准持续进化     |

**金标准范例 vs 模板的区别**：

```
模板：填空骨架，告诉你"要写什么"
金标准：完整真实代码，让模型自己推理"为什么这样写"+"怎样可以更好"

模板 = 下限保护
金标准 = 上限牵引
```

---

### 阶段 A.4 — 可观测（持续）

**目标**：架构自己能回答"质量是变好了还是变差了"。

| #     | 行动                                                       | 验收标准         |
| ----- | ---------------------------------------------------------- | ---------------- |
| A.4.1 | 记录每次生成的 Lane 匹配、轮次、闸门触发、verify 结果      | 日志结构定义完成 |
| A.4.2 | 建立 3 个核心指标                                          | 指标面板可用     |
| A.4.3 | 指标恶化 → 自动定位根因（哪个 Lane / 哪个闸门 / 哪个模板） | 根因定位链路跑通 |

**3 个核心指标**：

| 指标       | 定义                                             | 健康值   |
| ---------- | ------------------------------------------------ | -------- |
| 一次通过率 | 生成 → verify 1 轮通过的比例                     | > 70%    |
| 一致性分数 | 同需求两次生成，代码 diff 只有业务字段不同的比例 | > 90%    |
| 闸门触发率 | 每个闸门被触发的频率（高触发 = 模板没防住）      | 持续下降 |

## 三、两条路线的交汇点

```
                LITE                          AGENTS
                 │                              │
  L.1 跑通 ──────┤                              │
                 │                              │
  L.2 一致 ──────┼──────── 共享 pitfall ────────┤── A.1 减负
                 │         共享 verify           │
  L.3 覆盖 ──────┤         共享 recipe 框架      │── A.2 升级闸门
                 │                              │
  L.4 自治 ──────┼──────── 共享可观测 ──────────┤── A.3 金标准驱动
                 │         共享金标准            │
                 │                              │── A.4 可观测
```

- LITE 的错题 → 丰富 verify-errors.md → AGENTS 也受益
- AGENTS 的金标准 → 提升模式质量 → LITE 的 Skill 模板可以升级
- 两条路线共享 `pnpm verify` / pitfall 数据库 / 配方框架
- 各自独立演进，不互相阻塞

## 四、工业级标准的 5 条硬指标

到演进完成时，应该达到：

| #   | 指标                                       | 当前估计       | 目标                           |
| --- | ------------------------------------------ | -------------- | ------------------------------ |
| 1   | 生成 → verify 1 轮通过率                   | 未知（还没测） | > 80%                          |
| 2   | 同场景两次生成一致性                       | 未验证         | > 95%（diff 只有业务字段不同） |
| 3   | verify 3 轮内通过率                        | 未知           | > 95%                          |
| 4   | 非标 Lane 触发率                           | —              | < 15%                          |
| 5   | 人工介入率（需要用户回答/确认/修复的比例） | —              | < 10%                          |

## 五、立即行动的优先级

```
本周：
  1. L.1 跑通 — 商品管理 CRUD 全链路
  2. 记录第一份管道执行日志

下周：
  3. L.2 一致性 — diff 两次生成结果 + 锁差异点
  4. A.1 减负 — 重写 AGENTS.md 预读表

之后：
  5. L.3 覆盖 — 统计非标触发 + 高频升级
  6. A.2 升级闸门 — 语法 → 架构
```

## 六、不做什么

- **不预判** — 没遇到过的问题不加规则（错题集驱动）
- **不重复** — 一条规则只在一处定义（LITE 收敛到 Skill，AGENTS 收敛到 AGENTS.md）
- **不平铺** — 配方出现 3 次才沉淀，一次性的不占坑
- **不膨胀** — 每个月检查文件数量，新增的同时归档死文件
